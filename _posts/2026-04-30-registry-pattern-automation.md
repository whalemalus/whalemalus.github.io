# 注册表模式：让自动化任务不再硬编码\n\n> **摘要**：本文介绍一种让自动化任务动态发现工作目标的「注册表模式」。通过维护一个配置文件作为唯一数据源，cron 任务不再硬编码路径和目标，而是从注册表中动态读取。结合我们在服务器自动化运维中的实践，展示如何用两个简单的文本文件管理所有 GitHub 仓库同步和内容发布任务。\n\n> **关键词**：`注册表模式` `自动化` `cron` `DevOps` `配置驱动`\n\n---\n\n## 楔子\n\n我们服务器上跑着十几个定时任务：GitHub 仓库同步、博客文章发布、飞轮迭代、备份……最初每个任务的 prompt 里都硬编码了要检查哪些目录、推送到哪个仓库。\n\n直到有一天我发现一个问题：我在对话中创建了一个新的 GitHub 项目，但定时任务根本不知道它的存在 — 因为它的路径没有写在任何 cron 任务的 prompt 里。\n\n这让我意识到：**硬编码是自动化的天敌。**\n\n---\n\n## 1. 问题：硬编码的自动化\n\n### 1.1 典型的硬编码 cron 任务\n\n```yaml\n# ❌ 硬编码方式\nprompt: |\n  检查以下目录是否有变更：\n  - /home/user/project-a/\n  - /home/user/project-b/\n  - /opt/some-other-project/\n  如有变更则 git push\n```\n\n**问题**：\n- 新增项目时必须手动修改 cron 任务\n- 项目路径分散在多个 cron 任务中，难以管理\n- 删除一个项目后，cron 任务还在检查一个不存在的路径\n- 项目搬了位置，所有引用它的 cron 任务都要改\n\n### 1.2 更隐蔽的硬编码\n\n```yaml\n# ❌ 看似灵活，实际也是硬编码\nprompt: |\n  扫描 /home/user/ 目录下所有 git 仓库\n  如有变更则 git push\n```\n\n**问题**：\n- 项目不一定都在 `/home/user/` 下\n- 有些目录是临时的，不该同步\n- 有些仓库没有 remote，不该尝试 push\n- 无法区分哪些仓库需要同步、哪些不需要\n\n---\n\n## 2. 解决方案：注册表模式\n\n### 2.1 核心思想\n\n**一个配置文件作为唯一数据源（Single Source of Truth）。**\n\n```\n注册表文件（配置）\n    │\n    ▼\n定时任务读取注册表\n    │\n    ▼\n逐个执行：检查 → 操作 → 报告\n```\n\n### 2.2 注册表格式\n\n使用简单的文本格式，人可读、机器可解析：\n\n```\n# 以 # 开头的行是注释\n<字段1> | <字段2> | <字段3> | ...\n```\n\n**为什么不用 YAML/JSON？**\n- 文本文件更简单，不需要解析库\n- 人可以直接用 `vim` 编辑，不需要格式知识\n- `grep`、`awk` 等 Unix 工具可以直接处理\n- 不容易出现格式错误（YAML 的缩进、JSON 的引号）\n\n### 2.3 两种注册表\n\n我们实践了两种注册表：\n\n**注册表 1：GitHub 仓库同步**\n\n```bash\n# 文件: /home/user/.agent/github-repos.txt\n# 格式: <本地路径> | <GitHub 仓库> | <描述>\n\n/home/user/wiki | user/llm-wiki | LLM Wiki 知识库\n/home/user/project | user/project | 项目代码\n/root/.config | user/config-backup | 配置备份\n```\n\n**注册表 2：内容来源监控**\n\n```bash\n# 文件: /home/user/.agent/content-sources.txt\n# 格式: <类型> | <路径> | <目标> | <描述>\n\nsession | /home/user/sessions/ | blog | 对话记录\nwiki-page | /home/user/wiki/concepts/ | both | Wiki 概念页\nskill | /home/user/.agent/skills/ | blog | 技能文档\n```\n\n---\n\n## 3. 技术实现\n\n### 3.1 注册表解析器\n\n```bash\n#!/bin/bash\n# 读取注册表并逐行处理\n\nREGISTRY="/home/user/.agent/github-repos.txt"\n\nwhile IFS='|' read -r path repo desc; do\n    # 跳过注释和空行\n    [[ "$path" =~ ^#.*$ ]] && continue\n    [[ -z "$path" ]] && continue\n    \n    # 去除首尾空格\n    path=$(echo "$path" | xargs)\n    repo=$(echo "$repo" | xargs)\n    desc=$(echo "$desc" | xargs)\n    \n    # 检查路径是否存在\n    if [[ ! -d "$path" ]]; then\n        echo "⚠️ 路径不存在: $path ($desc)"\n        continue\n    fi\n    \n    # 检查是否有变更\n    cd "$path"\n    changes=$(git status --porcelain 2>/dev/null)\n    \n    if [[ -n "$changes" ]]; then\n        echo "📝 有变更: $desc"\n        git add -A\n        git commit -m "auto-sync: $(date +%Y-%m-%d)"\n        git push origin master\n    else\n        echo "⏭️ 无变更: $desc"\n    fi\ndone < "$REGISTRY"\n```\n\n### 3.2 Cron 任务配置\n\n```yaml\n# cron 任务的 prompt 不再硬编码路径\nprompt: |\n  你是自动化同步 Agent。\n  \n  从 /home/user/.agent/github-repos.txt 读取仓库注册表。\n  格式: <本地路径> | <GitHub 仓库> | <描述>\n  以 # 开头的行是注释。\n  \n  对每个仓库:\n  1. 检查路径是否存在\n  2. 检查 git 变更\n  3. 有变更则 commit + push\n  4. 输出同步报告\n```\n\n### 3.3 注册表更新机制\n\n```bash\n# 新增仓库\necho "/home/user/new-project | user/new-project | 新项目" >> /home/user/.agent/github-repos.txt\n\n# 删除仓库（用 sed 删除匹配行）\nsed -i '|new-project|d' /home/user/.agent/github-repos.txt\n\n# 查看注册表\ncat /home/user/.agent/github-repos.txt | grep -v "^#"\n```\n\n---\n\n## 4. 实践案例\n\n### 4.1 GitHub 仓库每日同步\n\n**注册表**: `/home/user/.agent/github-repos.txt`\n\n```\n/home/user/wiki | user/llm-wiki | LLM Wiki 知识库\n/home/user/project | user/project | PageWise Chrome 扩展\n/home/user/autopilot | user/autopilot | 飞轮迭代引擎\n/home/user/.agent | user/config-backup | Hermes Agent 配置备份\n```\n\n**Cron 任务**: 每天 01:00 执行\n\n**工作流程**:\n```\n01:00 触发\n  → 读取 github-repos.txt（4 个仓库）\n  → 逐个检查:\n    - wiki: 有 2 个新页面 → commit + push ✅\n    - pagewise: 飞轮迭代已自动提交 → 跳过 ⏭️\n    - autopilot: 无变更 → 跳过 ⏭️\n    - hermes-backup: 有配置变更 → commit + push ✅\n  → 输出同步报告到飞书\n```\n\n**实际执行结果**:\n```\nlast_run_at: 2026-04-30T11:51:00\nlast_status: ok\n\n检测到 pagewise 仓库有未提交的迭代报告\n→ 自动 commit: "docs: 新增 R25 迭代报告"\n→ 自动 push: 成功\n```\n\n### 4.2 内容来源每日扫描\n\n**注册表**: `/home/user/.agent/content-sources.txt`\n\n```\nsession | .../claude-code-sessions/ | blog | Claude Code 对话记录\niter-report | .../pagewise/docs/reports/ | blog | 迭代报告\nwiki-page | .../wiki/concepts/ | both | Wiki 概念页面\nskill | /home/user/.agent/skills/ | blog | Skills\n```\n\n**Cron 任务**: 每天 02:00 执行\n\n**工作流程**:\n```\n02:00 触发\n  → 读取 content-sources.txt\n  → 扫描过去 24 小时的新增内容:\n    - session: 2 个新对话记录 → 撰写文章 → 发布博客\n    - wiki-page: 3 个新概念页 → 撰写「知识库更新」→ 发布博客\n    - iter-report: 5 个迭代报告 → 无重大功能 → 跳过\n    - skill: 1 个新 skill → 撰写经验总结 → 发布博客\n  → 脱敏检查 → 发布 → 清缓存\n```\n\n### 4.3 飞轮迭代与注册表的协作\n\n飞轮迭代引擎在完成一轮迭代后，会自动 commit 代码。但有些变更不是飞轮产生的（比如我在对话中手动修改了文件），这些变更就会遗漏。\n\n**注册表解决了这个问题**：每天 01:00 的 cron 任务会检查所有注册的仓库，把飞轮遗漏的变更也一并推送。\n\n```\n飞轮迭代（实时）     注册表 cron（每天一次）\n    │                      │\n    ▼                      ▼\n自动 commit + push     检查所有仓库\n    │                      │\n    ├─ 代码变更 ✅         ├─ 飞轮已提交的 → 跳过\n    ├─ 迭代报告 ✅         ├─ 飞轮遗漏的 → 补提交\n    └─ 文档更新 ✅         └─ 对话中新增的 → 补提交\n```\n\n---\n\n## 5. 注册表模式的优势\n\n### 5.1 集中管理\n\n所有需要同步的仓库、所有需要监控的内容来源，都集中在一两个文件里。不需要翻遍十几个 cron 任务来找到"哪个任务在检查哪个目录"。\n\n### 5.2 动态扩展\n\n新增一个项目？只需要往注册表加一行。不需要修改任何 cron 任务的 prompt。\n\n### 5.3 容错性\n\n注册表中的路径不存在时，cron 任务会记录警告但不会崩溃。不会因为一个仓库的问题影响其他仓库的同步。\n\n### 5.4 可审计\n\n注册表文件本身就在 Git 管理下。谁在什么时候添加了什么仓库，一目了然。\n\n### 5.5 解耦\n\ncron 任务的逻辑（怎么同步）和配置（同步什么）完全分离。修改配置不需要理解任务逻辑，修改任务逻辑不需要关心具体配置。\n\n---\n\n## 6. 适用场景\n\n| 场景 | 注册表内容 | Cron 任务 |\n|------|-----------|----------|\n| GitHub 仓库同步 | 本地路径 + 仓库地址 | 每天检查变更并 push |\n| 博客文章发布 | 内容来源路径 + 发布目标 | 每天扫描新内容并发布 |\n| 监控检查 | 服务地址 + 检查间隔 | 按间隔检查服务健康 |\n| 数据备份 | 数据目录 + 备份目标 | 每天增量备份 |\n| 日志收集 | 日志路径 + 归档策略 | 每天归档日志 |\n\n**共同特征**：有一组"目标"需要定期处理，且目标会动态增减。\n\n---\n\n## 7. 最佳实践\n\n### 7.1 注册表文件命名\n\n使用清晰的命名，让人一眼看出用途：\n\n```\n✅ github-repos.txt        — GitHub 仓库列表\n✅ content-sources.txt     — 内容来源列表\n✅ backup-targets.txt      — 备份目标列表\n\n❌ config.txt              — 太模糊\n❌ list.txt                — 不知道是什么列表\n```\n\n### 7.2 注册表格式\n\n```\n# 第一行写注释说明格式\n# 格式: <字段1> | <字段2> | <字段3>\n\n# 每个字段的含义在头部说明\n# 字段1: 本地路径（绝对路径）\n# 字段2: 远程仓库地址\n# 字段3: 描述（人类可读）\n```\n\n### 7.3 注册表放在哪\n\n放在 Agent 的配置目录下，和其他配置一起管理：\n\n```\n/home/user/.agent/\n├── github-repos.txt       — 仓库注册表\n├── content-sources.txt    — 内容来源注册表\n├── skills/                — 技能文档\n└── memory/                — 记忆文件\n```\n\n### 7.4 注册表的生命周期管理\n\n```bash\n# 新增\necho "新条目" >> /path/to/registry.txt\n\n# 删除\nsed -i '|要删除的关键词|d' /path/to/registry.txt\n\n# 修改\nsed -i 's|旧值|新值|' /path/to/registry.txt\n\n# 查看\ncat /path/to/registry.txt | grep -v "^#"\n\n# 统计\ncat /path/to/registry.txt | grep -v "^#" | grep -v "^$" | wc -l\n```\n\n---\n\n## 8. 与其他模式的对比\n\n| 模式 | 描述 | 优点 | 缺点 |\n|------|------|------|------|\n| **硬编码** | 路径写在 cron prompt 里 | 最简单 | 难维护、容易遗漏 |\n| **目录扫描** | 扫描固定目录下所有文件 | 自动发现 | 无法区分、可能误操作 |\n| **注册表** | 维护一个配置文件 | 精确控制、易维护 | 需要手动注册 |\n| **服务发现** | 通过 API 动态发现 | 全自动 | 复杂、需要额外服务 |\n\n**注册表模式是硬编码和全自动之间的最佳平衡点。**\n\n---\n\n## 9. 扩展方向\n\n### 9.1 注册表校验\n\n定期检查注册表的健康状态：\n\n```bash\n# 检查所有路径是否存在\nwhile IFS='|' read -r path repo desc; do\n    [[ "$path" =~ ^#.*$ ]] && continue\n    [[ -z "$path" ]] && continue\n    path=$(echo "$path" | xargs)\n    [[ ! -d "$path" ]] && echo "❌ 路径不存在: $path"\ndone < /home/user/.agent/github-repos.txt\n```\n\n### 9.2 注册表 Web UI\n\n如果注册表条目很多，可以做一个简单的 Web 界面来管理：\n\n- 查看所有注册的仓库/来源\n- 添加/删除条目\n- 查看每个条目的同步状态\n- 手动触发同步\n\n### 9.3 注册表继承\n\n支持"基础注册表 + 项目注册表"的继承模式：\n\n```\n/home/user/.agent/global-repos.txt        — 全局仓库\n/home/user/project/.project-repos.txt — 项目级仓库\n```\n\n---\n\n## 总结\n\n注册表模式的核心就一句话：**把"做什么"从"怎么做"中分离出来。**\n\n- **怎么做**（cron 任务逻辑）→ 写一次，很少改\n- **做什么**（注册表条目）→ 经常改，改起来很简单\n\n两个文本文件，解决了所有自动化任务的动态发现问题：\n\n```\ngithub-repos.txt     → GitHub 仓库同步\ncontent-sources.txt  → 博客文章发布\n```\n\n**不需要数据库，不需要配置中心，不需要 API。就是两个文本文件。**\n\n如果你的服务器上也跑着一堆定时任务，不妨试试这个模式。把硬编码的路径抽出来放到注册表里，你会发现自动化运维突然变得清爽了很多。\n
