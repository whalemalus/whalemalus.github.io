输出设计文档\n3. **实现编码** — Sub Agent（Claude Code）编写代码和测试\n4. **质量评审** — Guard Agent 五维评分（需求合规 30% + 代码质量 25% + 安全 20% + 性能 15% + 测试覆盖 10%）\n5. **复盘文档** — 记录发现、教训和改进点\n\n每轮迭代产生一个独立的测试文件和迭代报告，形成可追溯的工程记录。\n\n### Skill Engine 的桥接层设计\n\nPageWise 的技能系统有个容易被误解的设计：`saveSkill()` 不持久化 `parameters` 字段。\n\n这不是 bug。技能的参数分两层：\n- **定义层**：技能描述中用 `{{variable}}` 标记的模板变量\n- **运行时层**：通过 `toEngineSkill()` 桥接函数注入的实际参数值\n\n持久化的是定义层（模板），运行时参数由调用方提供。这种分离让同一个技能模板可以在不同上下文中复用。\n\n### 质量门机制\n\nGuard Agent 的五维评分系统：\n\n| 维度 | 权重 | 评估内容 |\n|------|------|----------|\n| 需求合规 | 30% | 测试覆盖了所有需求点吗？ |\n| 代码质量 | 25% | 命名清晰、结构合理、无重复？ |\n| 安全性 | 20% | 无注入风险、权限最小化？ |\n| 性能 | 15% | 无内存泄漏、无阻塞操作？ |\n| 测试覆盖 | 10% | 边界条件、异常路径覆盖？ |\n\n≥90 分自动通过，80-89 分修复 P1 后通过，<80 分需要重做（最多 3 轮）。\n\n---\n\n## 3. 实战指南\n\n> 从零开始的实操步骤\n\n### R42 Skill Engine E2E 测试\n\nR42 的目标是为 Skill Engine 编写端到端测试，覆盖技能的完整生命周期。\n\n#### 测试覆盖范围\n\n```javascript\n// tests/test-skill-engine-e2e.js — 23 个测试用例，8 个 `--print` 模式返回空输出（exit code 0）。\n\n**原因**：Shell 内联 echo 长字符串时，特殊字符（引号、反斜杠）会被 shell 解释，导致 prompt 内容被截断或转义。\n\n**解决**：用临时文件 + 管道替代 inline echo：\n```bash\n# ❌ 失败方式\nclaude -p "Write tests for the following very long prompt..."\n\n# ✅ 正确方式\ncat /tmp/prompt.txt | claude -p --max-turns 25 --bare --effort low\n```\n\n### 坑 2：迭代引擎脚本 API Key 占位符\n\n**现象**：自动化迭代引擎脚本 (`iteration-engine.py`) 调用 Claude Code 时返回 401 Unauthorized。\n\n**原因**：脚本第 103 行的 `ANTHROPIC_API_KEY` 被硬编码为 `***`（脱敏占位符），而不是从配置文件读取。\n\n**解决**：从 `/home/claude-user/.claude/settings.json` 读取实际 API Key：\n```python\nimport json\nsettings = json.load(open('/home/claude-user/.claude/settings.json'))\napi_key = settings.get('apiKey', '')\n```\n\n### 坑 3：TODO.md 多进程冲突\n\n**现象**：多个 cron 任务同时修改 TODO.md，导致 checkbox 状态不一致。\n\n**原因**：飞轮迭代引擎每天运行 3 次（09:00/14:00/20:00），加上周度复盘等任务，多个进程可能同时写入 TODO.md。\n\n**解决**：在修改前先 `git pull` 获取最新版本，修改后立即 `git push`。\n\n---\n\n## 总结与展望\n\n- **核心收获**：E2E 测试不仅能发现 bug，还能验证设计决策。R42 中发现的
