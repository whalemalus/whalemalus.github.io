# 飞轮迭代：让 AI Agent 持续改进项目的工程方法

> **摘要**：飞轮迭代是一种让 AI Agent 持续改进项目的方法。每个迭代形成「需求→设计→测试→编码→评审→文档」的闭环，前一轮的输出成为下一轮的输入。本文介绍这套方法论的核心理念、实践流程和踩坑经验。
>
> **关键词**：`飞轮迭代` `AI Agent` `工程方法` `Claude Code` `迭代开发`

---

## 楔子

一个 Chrome 扩展，34 轮迭代，不到 3 小时。从零到完整产品。

这不是 demo，不是 POC，是 PageWise 项目的实际迭代记录。飞轮转起来之后，每一轮只需要 2-10 分钟——Agent 不需要开会，不需要排期，不需要等评审。

## 引言

飞轮迭代的核心理念是**文档驱动迭代**：项目文档告诉 Agent 怎么工作，文档夹追踪做了什么和接下来做什么。

```
需求 → 设计 → 测试 → 编码 → 评审 → 文档 → 新需求
 ↑                                              │
 └──────────────────────────────────────────────┘
```

每轮迭代只做 1-2 个功能，但积累是复利式的。34 轮迭代下来，项目从零到完整产品。

## 全景地图

### 飞轮迭代 vs 其他方法

| 方法 | 特点 | 适用场景 |
|------|------|----------|
| Scrum | 人驱动、2 周冲刺 | 团队协作 |
| Kanban | 持续流动、WIP 限制 | 运维/支持 |
| **飞轮迭代** | **Agent 驱动、分钟级迭代** | **AI 辅助开发** |
| TDD | 测试先行、小步前进 | 单个开发者 |

飞轮迭代的差异：迭代周期从「周」压缩到「分钟」。Agent 可以 24/7 不间断地迭代。

### PageWise 项目的实际数据

以 PageWise Chrome 扩展为例，34 轮飞轮迭代的实际数据：

- **总迭代数**：34 轮

- **测试用例**：1873 行测试代码
- **单轮耗时**：2-10 分钟（自动化）
- **失败率**：早期较高，后期显著下降

## 核心概念

### 飞轮迭代（Flywheel Iteration）

飞轮是一种自强化循环：每一次转动都为下一次积蓄能量。

### 三层 Agent 架构

飞轮迭代需要三个角色协同工作：

| 层 | 角色 | 职责 |
|---|------|------|
| Plan Agent | 规划者 | 需求分析、设计文档、回顾总结 |
| Sub Agent | 执行者 | 编写代码、运行测试、提交变更 |
| Guard Agent | 守护者 | 验证测试、审查代码、质量把关 |

关键点：规划和验证由不同 Agent 完成，避免执行者既当运动员又当裁判。

### 项目模板

每个飞轮项目需要两个核心文件：

**CLAUDE.md — 项目指南**

告诉 Agent 这个项目是什么、怎么开发、有什么约束：

```markdown
# 项目指南

## 项目概述
[一句话描述]

## 技术栈
[语言、框架、存储、构建工具]

## 开发规范
[代码风格、命名规范、注释要求]

## 迭代流程
1. 更新需求文档
2. 更新设计文档
3. 编写测试
4. 实现代码
5. 运行测试
6. 更新变更日志
7. 更新待办清单
```

**docs/ 目录 — 迭代文档**

```
docs/
├── REQUIREMENTS.md    # 需求文档（持续更新）
├── DESIGN.md          # 设计文档（持续更新）
├── CHANGELOG.md       # 变更日志（仅追加）
├── TODO.md            # 待办清单（每轮更新）
├── reports/           # 迭代报告（每轮自动生成）
│   ├── R1.md
│   ├── R2.md
│   └── ...
└── verification/      # 验证报告
```

### 四文档系统

每轮迭代必须产出四份文档：

1. **REQUIREMENTS.md** — 这轮要做什么
2. **DESIGN.md** — 怎么做
3. **测试代码** — 做对了没有
4. **CHANGELOG.md** — 做了什么

这四份文档让每轮迭代可追溯。

## 实战指南

### 步骤 1：需求分析

Plan Agent 读取 TODO.md 和当前代码状态，产出本轮迭代的需求文档：

- 明确要做什么（功能点）
- 明确验收标准
- 明确影响范围

### 步骤 2：设计

Plan Agent 产出技术设计：

- 模块划分
- 接口定义
- 数据结构
- 边界条件

### 步骤 3：实现

Sub Agent（如 Claude Code）执行编码：

- 按设计文档实现
- 遵循 CLAUDE.md 中的开发规范
- 遇到设计问题时记录而非绕过

### 步骤 4：验证

Guard Agent 验证质量：

- 运行全部测试（必须全绿）
- 检查代码是否符合设计
- 检查是否引入回归
- 产出验证报告

### 步骤 5：回顾

Plan Agent 总结本轮：

- 更新 CHANGELOG.md
- 更新 TODO.md（标记完成、添加新发现的问题）
- 生成迭代报告

## 踩坑记录

### 坑 1：测试卡死

飞轮迭代中最常见的问题是测试阶段卡死。排查流程：

**Step 1: 快速定位**

```bash
# 找到高 CPU 进程
ps aux --sort=-%cpu | head -10

# 查看迭代日志
tail -30 /var/log/iteration.log
```

**Step 2: 检查代码完整性**

最常见的根因：Agent 只完成了一半就跑测试。检查被测模块是否完整实现了所有导出函数。

**Step 3: 单独运行测试**

```bash
# 单独运行失败的测试文件，而非全部
node --test tests/test-problematic.js
```

**Step 4: 检查测试本身**

测试代码可能有无限循环、未 resolve 的 Promise、或资源未释放。

### 坑 2：Agent 规划能力不足

早期迭代失败率较高，主要原因是 Agent 的规划能力不够。表现为：需求文档遗漏关键约束、设计方案存在逻辑漏洞、测试用例覆盖不全。

应对：在 CLAUDE.md 中写清楚约束条件，减少 Agent 自由发挥的空间。

### 坑 3：范围过大导致失败

单轮迭代如果塞入太多功能，失败率会明显上升。

应对：每轮只做 1-2 个功能。宁可多迭代几轮，也不要贪多。

## 总结

### 核心收获

1. **文档是契约** — CLAUDE.md 是 Agent 的工作手册，docs/ 是项目的记忆
2. **每轮只做一件事** — 范围越小，失败率越低
3. **测试必须全绿** — 没有「差不多通过」这回事
4. **失败要归因** — 每次失败都要记录原因，避免重复犯错
5. **复利式积累** — 前一轮的文档和测试是下一轮的基础

### 最佳实践

- 迭代周期从「周」压缩到「分钟」，Agent 不需要开会、排期、等评审
- 34 轮迭代、每轮 5 分钟，总共不到 3 小时就能从零构建完整的 Chrome 扩展
- 飞轮转起来了，就不会停
