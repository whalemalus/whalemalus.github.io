# 飞轮迭代：让 AI Agent 持续改进项目的工程方法\n\n> **摘要**：飞轮迭代（Flywheel Iteration）是一种让 AI Agent 持续改进项目的工程方法。每个迭代形成「需求→设计→测试→编码→评审→文档」的闭环，前一轮的输出成为下一轮的输入，项目像飞轮一样越转越快。本文介绍这套方法论的核心理念、实践流程和实战经验。\n\n> **标签**：`AI Agent` `工程方法` `迭代开发` `Claude Code` `飞轮`\n\n---\n\n## 什么是飞轮迭代\n\n飞轮是一种自强化循环：每一次转动都为下一次积蓄能量。在软件开发中，飞轮迭代的核心理念是 **文档驱动迭代** — 项目文档告诉 Agent 怎么工作，文档夹追踪做了什么和接下来做什么。\n\n```\n需求 → 设计 → 测试 → 编码 → 评审 → 文档 → 新需求\n ↑                                              │\n └──────────────────────────────────────────────┘\n```\n\n每轮迭代只做 1-2 个功能，但积累是复利式的。34 轮迭代下来，项目从零到完整产品。\n\n## 三层 Agent 架构\n\n飞轮迭代需要三个角色协同工作：\n\n| 层 | 角色 | 职责 |\n|---|------|------|\n| Plan Agent | 规划者 | 需求分析、设计文档、回顾总结 |\n| Sub Agent | 执行者 | 编写代码、运行测试、提交变更 |\n| Guard Agent | 守护者 | 验证测试、审查代码、质量把关 |\n\n这种分工的关键在于：**规划和验证由不同 Agent 完成，避免执行者既当运动员又当裁判**。\n\n## 项目模板\n\n每个飞轮项目需要两个核心文件：\n\n### CLAUDE.md — 项目指南\n\n告诉 Agent 这个项目是什么、怎么开发、有什么约束：\n\n```markdown\n# 项目指南\n\n## 项目概述\n[一句话描述]\n\n## 技术栈\n[语言、框架、存储、构建工具]\n\n## 开发规范\n[代码风格、命名规范、注释要求]\n\n## 迭代流程\n1. 更新需求文档\n2. 更新设计文档\n3. 编写测试\n4. 实现代码\n5. 运行测试\n6. 更新变更日志\n7. 更新待办清单\n```\n\n### docs/ 目录 — 迭代文档\n\n```\ndocs/\n├── REQUIREMENTS.md    # 需求文档（持续更新）\n├── DESIGN.md          # 设计文档（持续更新）\n├── CHANGELOG.md       # 变更日志（仅追加）\n├── TODO.md            # 待办清单（每轮更新）\n├── reports/           # 迭代报告（每轮自动生成）\n│   ├── R1.md\n│   ├── R2.md\n│   └── ...\n└── verification/      # 验证报告\n```\n\n## 单轮迭代流程\n\n### Phase 1: 需求分析\n\nPlan Agent 读取 TODO.md 和当前代码状态，产出本轮迭代的需求文档：\n\n- 明确要做什么（功能点）\n- 明确验收标准\n- 明确影响范围\n\n### Phase 2: 设计\n\nPlan Agent 产出技术设计：\n\n- 模块划分\n- 接口定义\n- 数据结构\n- 边界条件\n\n### Phase 3: 实现\n\nSub Agent（如 Claude Code）执行编码：\n\n- 按设计文档实现\n- 遵循 CLAUDE.md 中的开发规范\n- 遇到设计问题时记录而非绕过\n\n### Phase 4: 验证\n\nGuard Agent 验证质量：\n\n- 运行全部测试（必须全绿）\n- 检查代码是否符合设计\n- 检查是否引入回归\n- 产出验证报告\n\n### Phase 5: 回顾\n\nPlan Agent 总结本轮：\n\n- 更新 CHANGELOG.md\n- 更新 TODO.md（标记完成、添加新发现的问题）\n- 生成迭代报告\n\n## 实战数据：PageWise 项目\n\n以 PageWise Chrome 扩展为例，34 轮飞轮迭代的实际数据：\n\n- **总迭代数**：34 轮\n- **覆盖功能**：从基础对话到 LLM Wiki 知识编译系统\n- **测试用例**：1873 行测试代码\n- **单轮耗时**：2-10 分钟（自动化）\n- **失败率**：早期较高（Agent 规划能力不足），后期显著下降\n\n### 关键里程碑\n\n| 迭代 | 功能 | 阶段 |\n|------|------|------|\n| R4-R8 | 页面高亮、对话持久化、快捷键、导出、内容预览 | 补齐标配 |\n| R9-R14 | YouTube 字幕、多页面分析、技能系统、API 文档模式 | 差异化 |\n| R15-R20 | 知识关联、间隔复习、知识图谱、语义搜索 | 知识深化 |\n| R21-R26 | 多模态图片、PDF 阅读、Prompt 模板、对话分支 | 高级能力 |\n| R27-R33 | 性能优化、错误处理、新手引导、商店准备 | 打磨发布 |\n| R34-R40 | Markdown 导出、实体提取、交叉引用、知识编译 | LLM Wiki |\n\n## 调试经验：测试卡死怎么办\n\n飞轮迭代中最常见的问题是测试阶段卡死。从实战中总结的排查流程：\n\n### Step 1: 快速定位\n\n```bash\n# 找到高 CPU 进程\nps aux --sort=-%cpu | head -10\n\n# 查看迭代日志\ntail -30 /var/log/iteration.log\n```\n\n### Step 2: 检查代码完整性\n\n最常见的根因：Agent 只完成了一半就跑测试。检查被测模块是否完整实现了所有导出函数。\n\n### Step 3: 单独运行测试\n\n```bash\n# 单独运行失败的测试文件，而非全部\nnode --test tests/test-problematic.js\n```\n\n### Step 4: 检查测试本身\n\n测试代码可能有无限循环、未 resolve 的 Promise、或资源未释放。\n\n## 与其他迭代方法的对比\n\n| 方法 | 特点 | 适用场景 |\n|------|------|----------|\n| Scrum | 人驱动、2 周冲刺 | 团队协作 |\n| Kanban | 持续流动、WIP 限制 | 运维/支持 |\n| **飞轮迭代** | **Agent 驱动、分钟级迭代** | **AI 辅助开发** |\n| TDD | 测试先行、小步前进 | 单个开发者 |\n\n飞轮迭代的独特优势：**迭代周期从「周」压缩到「分钟」**。Agent 不需要开会、不需要排期、不需要等待评审 — 它可以 24/7 不间断地迭代。\n\n## 核心原则\n\n1. **文档是契约** — CLAUDE.md 是 Agent 的工作手册，docs/ 是项目的记忆\n2. **每轮只做一件事** — 范围越小，失败率越低\n3. **测试必须全绿** — 没有「差不多通过」这回事\n4. **失败要归因** — 每次失败都要记录原因，避免重复犯错\n5. **复利式积累** — 前一轮的文档和测试是下一轮的基础\n\n## 总结\n\n飞轮迭代不是银弹，但它为 AI Agent 驱动的软件开发提供了一套可操作的工程方法。关键洞察是：**Agent 最大的优势不是写代码快，而是可以不间断地迭代**。34 轮迭代、每轮 5 分钟 — 总共不到 3 小时，就能从零构建一个完整的 Chrome 扩展。\n\n飞轮转起来了，就不会停。\n\n---\n\n*本文基于 PageWise 项目的飞轮迭代实践总结。PageWise 是一个帮助用户在浏览技术网页时即时向 AI 提问的 Chrome 扩展。*\n
