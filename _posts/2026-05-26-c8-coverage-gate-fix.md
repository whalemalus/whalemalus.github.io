# 当测试覆盖率门禁"骗"了你：c8 配置踩坑实录

> **摘要**：PageWise 项目的覆盖率门禁一直报 24%，远低于 28% 的阈值。但实际被测试的代码覆盖率高达 75%。问题出在 c8 配置中的 `"all": true`，它把 171 个从未被任何测试导入的模块也计入了分母。本文记录了从发现到修复的完整过程。
>
> **关键词**：`c8` `代码覆盖率` `测试配置` `Node.js` `PageWise`

---

## 楔子

周五下午，PageWise 飞轮迭代引擎报告：覆盖率门禁没过。

我打开终端跑了一下 `npm run coverage:gate`，果然——**24%**，门禁要求 28%。差 4 个百分点。

但我明明记得上周刚写了一大波测试，覆盖率应该不低才对。于是我单独跑了一遍 `npm run test:coverage`，盯着输出看了半天：

```
Lines: 75.76% (13291/17543)
Branches: 78.2%
Functions: 82.1%
```

75%？那门禁为什么报 24%？

这两个数字之间的巨大鸿沟，就是今天的故事。

## 引言

测试覆盖率是代码质量的重要指标，但如果覆盖率工具配置不对，它给你的数字可能会严重误导你。一个错误的门禁阈值配置还可能阻塞整个 CI/CD 流水线，让团队对测试失去信心。

这篇文章记录了 PageWise 项目中 c8 覆盖率工具的一个常见配置陷阱：`"all": true` 选项。这个看起来没什么问题的配置项，把覆盖率从 75% 拉到了 24%，让门禁连续失败了好几个星期。

如果你也在用 c8 或者 Istanbul 系列工具，这个坑你大概率也会踩到。

## 核心概念

### c8 是什么？

c8 是 Node.js 的代码覆盖率工具，基于 V8 引擎内置的覆盖率数据。它是 Istanbul/nyc 的现代替代品，不需要源码插桩，直接使用 V8 原生数据，性能更好、配置更简单。

### `"all": true` 的含义

这是 c8 配置中最容易误解的选项：

```json
{
  "all": true,    // 把项目中所有源文件都计入覆盖率分母
  "src": ["lib"]  // 指定"所有源文件"的范围
}
```

**当 `all: true` 时**：c8 会扫描 `src` 目录下的所有文件，即使某个文件从未被任何测试导入，也会被计入覆盖率分母（覆盖率 = 0%）。

**当 `all: false` 时**（默认）：c8 只统计实际被执行到的代码的覆盖率。

### 为什么这很重要？

用一个类比来解释：

> 假设一个学校有 100 个学生，但只有 30 个学生参加了考试。如果算"全校平均分"，把没参加考试的学生也算作 0 分，那平均分会被严重拉低。
>
> `all: true` 就像是把没参加考试的学生也算作 0 分。`all: false` 则只统计实际参加考试的学生。

对于 PageWise 这样有 171 个 lib 模块（共 36,409 行）的项目，大量模块是工具库、插件系统等，不是每个模块都需要被主测试套件覆盖。`all: true` 让这些模块的 0% 覆盖率稀释了整体数字。

## 实战指南

### 步骤 1：确认问题

```bash
# 运行覆盖率门禁
npm run coverage:gate
# 输出: Lines: 24% (未通过 28% 门禁)

# 运行完整覆盖率报告
npm run test:coverage
# 输出: Lines: 75.76% (实际被测试代码的覆盖率)
```

两个数字差距这么大，说明配置肯定有问题。

### 步骤 2：检查 c8 配置

```bash
cat .c8rc.json
```

问题配置：

```json
{
  "all": true,
  "src": ["lib"],
  "reporter": ["text", "lcov", "json"],
  "include": ["lib/**/*.js"],
  "exclude": ["tests/**", "node_modules/**"]
}
```

`"all": true` + `"src": ["lib"]` = 把 lib 下所有 171 个文件都计入分母。

### 步骤 3：修正配置

```json
{
  "reporter": ["text", "lcov", "json"],
  "include": ["lib/**/*.js"],
  "exclude": ["tests/**", "node_modules/**"]
}
```

**删除了两行**：
- `"all": true` — 不再把未执行的文件计入分母
- `"src": ["lib"]` — 这是 `all` 的配套选项，没有 `all` 就不需要它

### 步骤 4：更新依赖测试

删除配置后，还需要更新那些断言 `all`/`src` 字段存在的测试：

```javascript
// 删除这类断言
assert.ok(config.all, 'all should be present');
assert.deepEqual(config.src, ['lib'], 'src should specify lib directory');

// 保留或调整为检查核心字段
const requiredFields = ['reporter', 'include', 'exclude'];
for (const field of requiredFields) {
  assert.ok(field in config, `${field} should be present`);
}
```

同时更新验证脚本 `scripts/validate-c8-config.sh`，移除对 `all`/`src` 的检查。

### 步骤 5：验证修复

```bash
# 运行完整测试套件
npm test
# 输出: 8089 pass / 0 fail ✅

# 运行覆盖率
npm run test:coverage
# 输出: Lines: 75.76%, Branches: 78.2%, Functions: 82.1% ✅

# 运行覆盖率门禁
npm run coverage:gate
# 输出: All thresholds passed ✅
```

| 指标 | 修复前 | 修复后 | 门禁阈值 |
|------|--------|--------|----------|
| Lines | 24% | 75.76% | ≥ 28% ✅ |
| Branches | 76% | 78.2% | ≥ 75% ✅ |
| Functions | 80% | 82.1% | ≥ 50% ✅ |

## 踩坑记录

### 坑 1：两个数字差距巨大却没人质疑

**现象**：覆盖率门禁报 24%，但测试报告显示 75%，持续了数周没人注意到差异。

**原因**：`npm run coverage:gate` 和 `npm run test:coverage` 虽然都调用 c8，但门禁脚本会读取 `.c8rc.json` 中的 `all` 配置，而直接运行 test:coverage 时可能有不同的默认行为。

**教训**：当两个相关指标出现不一致时，应该马上去查，别接受较低的那个数字就算了。

### 坑 2：`all: true` 是"好心办坏事"

**现象**：配置者可能觉得 `all: true` 是更严格的标准——"我们应该追求所有代码的覆盖率"。

**原因**：对于有大量工具模块的项目，`all: true` 会产生不切实际的分母。171 个模块中有大量是辅助性质的（工具函数、插件接口、类型定义等），不需要被主测试套件覆盖。

**教训**：覆盖率应该反映"被测试代码的质量"，而不是"项目中有多少代码没被测试"。这两件事是不一样的。

### 坑 3：基础设施测试耦合了配置细节

**现象**：4 个测试文件断言了 `.c8rc.json` 中存在 `all` 和 `src` 字段。

**原因**：这些测试是为了防止配置漂移（config drift），但它们断言了不该断言的字段。应该断言"配置文件包含必要字段"，而不是"配置文件长什么样"。

**教训**：配置验证测试应该检查"核心功能能不能正常工作"，而不是"配置是否长成某个特定样子"。

## 总结

### 核心收获

1. **`all: true` 不等于更严格**：它只是把更多文件拉进分母，稀释了真实覆盖率。大型项目用这个选项，几乎都是错的。

2. **配置即代码**：一行配置的差异可以导致 50 个百分点的覆盖率偏差。`.c8rc.json` 值得像代码一样被 review。

3. **门禁数字要可解释**：如果门禁报出的数字和你的直觉差距很大，大概率是配置问题，不是代码问题。

### 最佳实践

- **新建项目**：默认使用 `all: false`（c8 的默认值），除非你有明确理由
- **大型项目**：只统计实际被测试的代码，避免工具模块稀释覆盖率
- **配置审查**：CI 流水线中加入配置文件的静态检查，防止意外修改
- **数字验证**：定期对比门禁数字和详细报告，确保一致性
