# 当测试覆盖率门禁"骗"了你：c8 配置踩坑实录\n\n> **摘要**：PageWise 项目的覆盖率门禁一直报 24%，远低于 28% 的阈值。但实际被测试的代码覆盖率高达 75%。问题出在 c8 配置中的 `\\"all\\": true`，它把 171 个从未被任何测试导入的模块也计入了分母。本文记录了从发现到修复的完整过程。\n>\n> **关键词**：`c8` `代码覆盖率` `测试配置` `Node.js` `PageWise`\n\n---\n\n## 楔子\n\n周五下午，PageWise 飞轮迭代引擎报告：覆盖率门禁未通过。\n\n我打开终端跑了一下 `npm run coverage:gate`，果然——**24%**，门禁要求 28%。差 4 个百分点。\n\n但我明明记得上周刚写了一大波测试，覆盖率应该不低才对。于是我单独跑了一遍 `npm run test:coverage`，盯着输出看了半天：\n\n```\nLines: 75.76% (13291/17543)\nBranches: 78.2%\nFunctions: 82.1%\n```\n\n75%？那门禁为什么报 24%？\n\n这两个数字之间的巨大鸿沟，就是今天的故事。\n\n## 引言\n\n测试覆盖率是代码质量的重要指标，但如果你的覆盖率工具配置不当，它给出的数字可能会严重误导你。更糟糕的是，一个错误的门禁阈值配置可能会阻塞整个 CI/CD 流水线，让团队对测试失去信心。\n\n本文记录了 PageWise 项目中 c8 覆盖率工具的一个常见配置陷阱：`\\"all\\": true` 选项。这个看似无害的配置项，把覆盖率从 75% 拉到了 24%，让门禁连续失败了数周。\n\n如果你也在用 c8 或 Istanbul 系列工具，这个坑你大概率也会踩到。\n\n## 📖 目录\n\n1. 删除配置    75.76%     配置即代码\n```\n\n## 核心概念\n\n### c8 是什么？\n\nc8 是 Node.js 的代码覆盖率工具，基于 V8 引擎内置的覆盖率数据。它是 Istanbul/nyc 的现代替代品，不需要源码插桩，直接使用 V8 原生数据，性能更好、配置更简单。\n\n### `\\"all\\": true` 的含义\n\n这是 c8 配置中最容易误解的选项：\n\n```json\n{\n  "all": true,    // 把项目中所有源文件都计入覆盖率分母\n  "src": ["lib"]  // 指定"所有源文件"的范围\n}\n```\n\n**当 `all: true` 时**：c8 会扫描 `src` 目录下的所有文件，即使某个文件从未被任何测试导入，也会被计入覆盖率分母（覆盖率 = 0%）。\n\n**当 `all: false` 时**（默认）：c8 只统计实际被执行到的代码的覆盖率。\n\n### 为什么这很重要？\n\n用一个类比来解释：\n\n> 假设一个学校有 100 个学生，但只有 30 个学生参加了考试。如果算"全校平均分"，把没参加考试的学生也算作 0 分，那平均分会被严重拉低。\n>\n> `all: true` 就像是把没参加考试的学生也算作 0 分。`all: false` 则只统计实际参加考试的学生。\n\n对于 PageWise 这样有 171 个 lib 模块（共 36,409 行）的项目，大量模块是工具库、插件系统等，不是每个模块都需要被主测试套件覆盖。`all: true` 让这些模块的 0% 覆盖率稀释了整体数字。\n\n## 实战指南\n\n### 第一步：确认问题\n\n```bash\n# 运行覆盖率门禁\nnpm run coverage:gate\n# 输出: Lines: 24% (未通过 28% 门禁)\n\n# 运行完整覆盖率报告\nnpm run test:coverage\n# 输出: Lines: 75.76% (实际被测试代码的覆盖率)\n```\n\n两个数字差距巨大，说明配置有误。\n\n### 第二步：检查 c8 配置\n\n```bash\ncat .c8rc.json\n```\n\n问题配置：\n\n```json\n{\n  "all": true,\n  "src": ["lib"],\n  "reporter": ["text", "lcov", "json"],\n  "include": ["lib/**/*.js"],\n  "exclude": ["tests/**", "node_modules/**"]\n}\n```\n\n`\\"all\\": true` + `\\"src\\": [\\"lib\\"]` = 把 lib 下所有 171 个文件都计入分母。\n\n### 第三步：修正配置\n\n```json\n{\n  "reporter": ["text", "lcov", "json"],\n  "include": ["lib/**/*.js"],\n  "exclude": ["tests/**", "node_modules/**"]\n}\n```\n\n**删除了两行**：\n- `\\"all\\": true` — 不再把未执行的文件计入分母\n- `\\"src\\": [\\"lib\\"]` — 这是 `all` 的配套选项，没有 `all` 就不需要它\n\n### 第四步：更新依赖测试\n\n删除配置后，还需要更新那些断言 `all`/`src` 字段存在的测试：\n\n```javascript\n// 删除这类断言\nassert.ok(config.all, 'all should be present');\nassert.deepEqual(config.src, ['lib'], 'src should specify lib directory');\n\n// 保留或调整为检查核心字段\nconst requiredFields = ['reporter', 'include', 'exclude'];\nfor (const field of requiredFields) {\n  assert.ok(field in config, `${field} should be present`);\n}\n```\n\n同时更新验证脚本 `scripts/validate-c8-config.sh`，移除对 `all`/`src` 的检查。\n\n### 第五步：验证修复\n\n```bash\n# 运行完整测试套件\nnpm test\n# 输出: 8089 pass / 0 fail ✅\n\n# 运行覆盖率\nnpm run test:coverage\n# 输出: Lines: 75.76%, Branches: 78.2%, Functions: 82.1% ✅\n\n# 运行覆盖率门禁\nnpm run coverage:gate\n# 输出: All thresholds passed ✅\n```\n\n| 指标 | 修复前 | 修复后 | 门禁阈值 |\n|------|--------|--------|----------|\n| Lines | 24% | 75.76% | ≥ 28% ✅ |\n| Branches | 76% | 78.2% | ≥ 75% ✅ |\n| Functions | 80% | 82.1% | ≥ 50% ✅ |\n\n## 踩坑记录\n\n### 坑 1：两个数字差距巨大却没人质疑\n\n**现象**：覆盖率门禁报 24%，但测试报告显示 75%，持续了数周没人注意到差异。\n\n**原因**：`npm run coverage:gate` 和 `npm run test:coverage` 虽然都调用 c8，但门禁脚本会读取 `.c8rc.json` 中的 `all` 配置，而直接运行 test:coverage 时可能有不同的默认行为。\n\n**教训**：当两个相关指标出现不一致时，应该立即调查，而不是接受较低的那个数字。\n\n### 坑 2：`all: true` 是"好心办坏事"\n\n**现象**：配置者可能认为 `all: true` 是更严格的标准——"我们应该追求所有代码的覆盖率"。\n\n**原因**：对于有大量工具模块的项目，`all: true` 会产生不切实际的分母。171 个模块中有大量是辅助性质的（工具函数、插件接口、类型定义等），不需要被主测试套件覆盖。\n\n**教训**：覆盖率应该反映"被测试代码的质量"，而不是"项目中有多少代码没被测试"。两者是不同的问题。\n\n### 坑 3：基础设施测试耦合了配置细节\n\n**现象**：4 个测试文件断言了 `.c8rc.json` 中存在 `all` 和 `src` 字段。\n\n**原因**：这些测试是为了防止配置漂移（config drift），但它们断言了错误的字段。应该断言"配置文件包含必要字段"，而不是"配置文件包含特定值"。\n\n**教训**：配置验证测试应该检查"核心功能是否可工作"，而不是"配置是否长成某个特定样子"。\n\n## 总结与展望\n\n### 核心收获\n\n1. **`all: true` 不等于更严格**：它只是把更多文件拉入分母，稀释了真实覆盖率。对于大型项目，这几乎总是错误的选择。\n\n2. **配置即代码**：一行配置的差异可以导致 50 个百分点的覆盖率偏差。`.c8rc.json` 值得像代码一样被 review。\n\n3. **门禁数字要可解释**：如果门禁报出的数字和你的直觉差距很大，大概率是配置问题，不是代码问题。\n\n### 最佳实践\n\n- **新建项目**：默认使用 `all: false`（c8 的默认值），除非你有明确理由\n- **大型项目**：只统计实际被测试的代码，避免工具模块稀释覆盖率\n- **配置审查**：CI 流水线中加入配置文件的静态检查，防止意外修改\n- **数字验证**：定期对比门禁数字和详细报告，确保一致性\n\n### 延伸阅读\n\n- [c8
