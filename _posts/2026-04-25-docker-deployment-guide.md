24）\n- 应用代码\n- 依赖库\n- 配置文件\n\n**在我们的 OpenClaw 项目中：**\n```\n镜像：1panel/openclaw:2026.4.21\n基础镜像：node:24-bookworm（基于 Debian Bookworm + Node.js 24）\n```\n\n镜像是**不可变的**——一旦创建就不能修改。如果需要更新，就创建一个新版本的镜像。镜像名后面的 `:2026.4.21` 就是**标签（tag）**，表示版本号。\n\n### 2. 📦 容器（Container）—— 镜像的运行实例\n\n**类比：** 如果镜像是菜谱，那容器就是**照着菜谱做出来的一道菜**🍲。\n\n从一个镜像可以创建多个容器，就像同一张菜谱可以做出很多盘菜。每个容器都是独立运行的，有自己的文件系统、网络和进程空间。\n\n**在我们的项目中：**\n```\n容器名：1Panel-openclaw-gFtw\n来自镜像：1panel/openclaw:2026.4.21\n状态：运行中（Up）\n```\n\n> 💡 **一句话总结：** 镜像是静态的模板，容器是动态的运行实例。镜像可以类比为"类（Class）"，容器可以类比为"对象（Object）"。\n\n### 3. 📁 卷（Volume）—— 持久化存储\n\n**类比：** 容器就像是酒店房间🏨，退房后房间会被清空。卷就像是**保险箱**🔒，即使退房，保险箱里的东西还在。\n\n容器默认是临时的——删除容器后，里面的文件就没了。但我们的配置文件、数据不能丢啊！所以需要**卷挂载**，把容器内的目录映射到宿主机的磁盘上。\n\n**在我们的项目中：**\n```yaml\nvolumes:\n  - ./data/conf:/home/node/.openclaw          # 配置文件持久化\n  - ./data/workspace:/home/node/.openclaw/workspace  # 工作区数据持久化\n  - /etc/localtime:/etc/localtime             # 同步宿主机时区\n```\n\n这意味着：\n- 容器里 `/home/node/.openclaw` 目录的内容，实际上保存在宿主机的 `./data/conf` 中\n- 即使容器被删除重建，数据依然存在\n- 第三个挂载是把宿主机的时区信息传给容器，保证时间一致\n\n### 4. 🔌 端口映射（Port Mapping）—— 网络桥梁\n\n**类比：** 容器有自己的"内网电话号码"，但外面的人打不进来。端口映射就像是前台总机📞，把外部来电转接到内部分机。\n\n**在我们的项目中：**\n```\n端口映射：0.0.0.0:18789 → 18789\n```\n\n这表示：\n- 宿主机所有网卡（0.0.0.0）的 18789 端口\n- 转发到容器内部的 18789 端口\n- 任何人通过 `http://服务器IP:18789` 都能访问到容器内的 OpenClaw 服务\n\n### 5. 📝 docker-compose.yml —— 编排说明书\n\n**类比：** 如果 Docker 命令是**单个食材的处理**（切菜、炒菜），那 docker-compose 就是**整桌宴席的菜谱**📋——一道命令搞定所有菜。\n\n`docker-compose.yml` 用声明式的方式定义了：\n- 用什么镜像\n- 怎么配置网络\n- 挂载什么目录\n- 映射什么端口\n- 设置什么环境变量\n- 健康检查怎么做\n\n---\n\n## 认识我们的实战项目：OpenClaw\n\nOpenClaw 是一个运行在 Node.js 上的网关服务。我们的部署方式是通过 **1Panel** 面板进行 Docker 容器化管理。\n\n### 📋 OpenClaw 容器信息一览\n\n| 项目 | 值 实战详解：OpenClaw 的 docker-compose.yml\n\n让我们逐行解析这个 docker-compose 配置文件：\n\n```yaml\nversion: "3.8"\n\nservices:\n  openclaw:\n    image: 1panel/openclaw:2026.4.21          # 使用的 Docker 镜像及版本标签\n    container_name: 1Panel-openclaw-gFtw      # 容器的名称（便于识别）\n    restart: unless-stopped                    # 重启策略\n    user: node                                 # 以 node 用户运行（非 root）\n    networks:\n      - 1panel-network                         # 加入 1panel-network 网络\n    ports:\n      - "0.0.0.0:18789:18789"                 # 端口映射\n    volumes:\n      - ./data/conf:/home/node/.openclaw       # 配置目录挂载\n      - ./data/workspace:/home/node/.openclaw/workspace  # 工作区挂载\n      - /etc/localtime:/etc/localtime          # 时区同步\n    entrypoint:\n      - /usr/local/bin/docker-entrypoint.sh    # 入口点脚本\n    command:\n      - node\n      - openclaw.mjs\n      - gateway\n      - --allow-unconfigured                  # 启动命令及参数\n    healthcheck:\n      test: ["CMD", "fetch", "http://127.0.0.1:18789/healthz"]\n      interval: 30s                            # 每 30 秒检查一次\n      timeout: 10s                             # 超时时间 10 秒\n      retries: 3                               # 连续失败 3 次视为不健康\n      start_period: 10s                        # 启动后等 10 秒再开始检查\n\nnetworks:\n  1panel-network:\n    external: true                             # 使用已存在的外部网络\n```\n\n### 📌 逐项深度解析\n\n#### `image: 1panel/openclaw:2026.4.21`\n\n指定使用的 Docker 镜像。这个镜像的完整名称包含三部分：\n- **仓库地址**：省略了默认的 `docker.io/`（Docker Hub）\n- **镜像名**：`1panel/openclaw`\n- **标签**：`:2026.4.21`（版本号，类似 Git 的 tag）\n\n镜像的基础构建层：\n```\n1panel/openclaw:2026.4.21\n    └── 基于 node:24-bookworm\n            └── 基于 debian:bookworm\n                    └── 基于 scratch（Docker 最小基础镜像）\n```\n\nDocker 镜像采用**分层存储**，每一层只记录与上一层的差异，大大节省磁盘空间。\n\n#### `restart: unless-stopped`\n\n重启策略决定了容器在退出后是否自动重启：\n\n| 策略 | 说明 |\n|------|------|\n| `no` | 默认值，不自动重启 |\n| `always` | 总是自动重启（包括手动停止后 Docker 重启时） |\n| `on-failure` | 只在非正常退出（退出码非0）时重启 |\n| `unless-stopped` | 总是自动重启，除非是手动 `docker stop` 停止的 |\n\nOpenClaw 使用 `unless-stopped` 是一个很好的选择——服务器重启后容器会自动恢复运行，但如果你手动停止了它，Docker 不会自作主张地帮你重启。\n\n#### `user: node`\n\n以非 root 用户运行容器内的进程。这是一个**安全最佳实践** ✅：\n- 即使容器被攻破，攻击者也只有普通用户的权限\n- 防止容器内的恶意进程修改系统文件\n- OpenClaw 镜像内置了 `node` 用户\n\n#### `ports: "0.0.0.0:18789:18789"`\n\n端口映射格式：`[宿主机IP:]宿主机端口:容器端口[/协议]`\n\n```\n0.0.0.0:18789:18789\n│        │      │\n│        │      └── 容器内部端口\n│        └───────── 宿主机暴露端口\n└────────────────── 绑定所有网络接口\n```\n\n- `0.0.0.0` 表示监听所有网络接口（所有人都能访问）\n- 如果写成 `127.0.0.1:18789:18789`，则只有本机可以访问\n- 如果写成 `18789:18789`，默认等同于 `0.0.0.0:18789:18789`\n\n#### `volumes` 卷挂载详解\n\n```yaml\nvolumes:\n  - ./data/conf:/home/node/.openclaw\n  - ./data/workspace:/home/node/.openclaw/workspace\n  - /etc/localtime:/etc/localtime\n```\n\n**前两个是"绑定挂载"（Bind Mounts）**：\n\n```\n宿主机目录                容器目录\n./data/conf      ←→      /home/node/.openclaw\n./data/workspace  ←→      /home/node/.openclaw/workspace\n```\n\n这是**双向映射**——在容器内修改文件，宿主机上也能看到；反之亦然。这样即使容器被删除重建，只要指向同一个宿主机目录，数据就还在。\n\n**第三个是只读时区同步**：\n```\n/etc/localtime   ←→      /etc/localtime\n```\n让容器和宿主机使用相同的时区设置，确保日志时间、定时任务等时间一致。\n\n> ⚠️ **注意**：`./data/conf` 这里的 `.` 指的是 docker-compose.yml 所在的目录，即 `/opt/your-app/`。\n\n#### `entrypoint` 与 `command`\n\n```yaml\nentrypoint:\n  - /usr/local/bin/docker-entrypoint.sh\ncommand:\n  - node\n  - openclaw.mjs\n  - gateway\n  - --allow-unconfigured\n```\n\n**entrypoint** 是容器启动时**必定执行**的入口程序（通常是初始化脚本）。\n**command** 是传给 entrypoint 的参数。\n\n最终执行的命令相当于：\n```bash\n/usr/local/bin/docker-entrypoint.sh node openclaw.mjs gateway --allow-unconfigured\n```\n\n`docker-entrypoint.sh` 通常负责：\n- 环境变量初始化\n- 权限设置\n- 依赖检查\n- 最后 `exec` 执行真正的业务命令\n\n`--allow-unconfigured` 参数允许服务在未完成配置时也能启动，方便初次部署。\n\n#### `healthcheck` 健康检查\n\n```yaml\nhealthcheck:\n  test: ["CMD", "fetch", "http://127.0.0.1:18789/healthz"]\n  interval: 30s\n  timeout: 10s\n  retries: 3\n  start_period: 10s\n```\n\nDocker 会定期向容器内的 `http://127.0.0.1:18789/healthz` 发送请求：\n- 如果返回成功（HTTP 200），容器状态为 `healthy` ✅\n- 如果连续 3 次失败，状态变为 `unhealthy` ❌\n- 启动后有 10 秒的缓冲期，这段时间内不计入检查\n\n> 💡 健康检查不影响容器运行，但可以被其他监控系统（如 1Panel）利用来自动处理故障。\n\n#### `networks: 1panel-network (external: true)`\n\n```yaml\nnetworks:\n  1panel-network:\n    external: true\n```\n\n- `1panel-network` 是一个预创建的 Docker 网络\n- `external: true` 表示这个网络不是由本 compose 文件创建的，而是已经存在的\n- 同一网络内的容器可以通过**容器名互相访问**\n\n例如，如果还有另一个容器 `nginx` 也在 `1panel-network` 中，它可以访问 `http://1Panel-openclaw-gFtw:18789`。\n\n---\n\n## 常用 Docker 命令大全\n\n### 🔍 查看类命令\n\n#### 查看运行中的容器\n```bash\ndocker ps\n```\n\n输出示例：\n```\nCONTAINER ID   IMAGE                          COMMAND denied\n```\n**原因：** 容器内用户（node）没有权限访问某些文件\n**解决：**\n```bash\n# 在宿主机上修改文件权限\nchown -R 1000:1000 /opt/your-app/data/\n# 1000 通常是容器内 node 用户的 UID\n```\n\n#### 坑 5：Docker 磁盘空间不足\n```bash\n# 查看 Docker
