`80` 和 `443`。\n\n这意味着即使 Cloudflare 把流量送到了服务器的 443 端口，也没有服务在监听！\n\n```\n# 期望的映射\n服务器:80  → NPM 容器:80\n服务器:443 → NPM 容器:443\n\n# 实际的映射（1Panel 默认）\n服务器:30080 → NPM 容器:80\n服务器:30443 → NPM 容器:443\n```\n\n### ❌ 问题 2：Cloudflare DNS 中有错误的 A 记录\n\nCloudflare 的 DNS 配置中存在多个 A 记录，其中包含了不属于我们的 IP：\n\n```\nwhalemalus.com → <你的服务器公网IP>   ✅ 正确（我们的 VPS）\nwhalemalus.com → <示例IP-A>   ❌ 错误（不知道哪里来的）\nwhalemalus.com → <示例IP-B>   ❌ 错误（不知道哪里来的）\n```\n\nCloudflare 会对这些 A 记录进行 **轮询 (Round Robin)**，导致大约 1/3 的请求被发送到错误的服务器。\n\n### ❌ 问题 3：SSL 证书配置\n\nCloudflare 提供了 **Universal SSL**（Let's Encrypt 颁发），但 Cloudflare 到源站服务器之间也需要加密。需要生成 **Cloudflare Origin Certificate** 并配置到 NPM。\n\n### ❌ 问题 4：NPM API 更新后配置未自动生成\n\n通过 NPM 的 API 创建代理主机和上传证书后，Nginx 的实际配置文件并没有自动生成！需要手动创建。\n\n---\n\n## 四、完整解决步骤\n\n### 步骤 1：修改 NPM 端口映射\n\n#### 1.1 停用主机上的 Nginx（如果有的话）\n\n首先检查服务器上是否有独立安装的 Nginx 在占用 80/443 端口：\n\n```bash\n# 检查是否有 Nginx 在运行\nsystemctl status nginx\n\n# 如果在运行，停止并禁用\nsudo systemctl stop nginx\nsudo systemctl disable nginx\n```\n\n#### 1.2 查找 NPM 的 docker-compose 文件\n\nNPM 通常由 1Panel 安装，配置文件在 1Panel 的应用目录中：\n\n```bash\n# 查找 docker-compose 文件\nfind / -name "docker-compose.yml" -path "*/nginx*" 2>/dev/null\n\n# 1Panel 的典型路径\nls /opt/nginx-proxy-manager/\n```\n\n#### 1.3 修改 .env 文件\n\n找到 `.env` 文件，修改端口配置：\n\n```bash\n# 进入 NPM 目录\ncd /opt/nginx-proxy-manager/\n\n# 编辑 .env\nvi .env\n```\n\n修改以下内容：\n\n```env\n# 修改前（1Panel 默认）\nPANEL_APP_PORT_HTTP1=30080\nPANEL_APP_PORT_HTTP2=30443\n\n# 修改后（标准端口）\nPANEL_APP_PORT_HTTP1=80\nPANEL_APP_PORT_HTTP2=443\n```\n\n#### 1.4 重建容器\n\n```bash\ndocker compose down\ndocker compose up -d --force-recreate\n```\n\n验证端口是否正确映射：\n\n```bash\ndocker ps | grep nginx-proxy-manager\n# 应该看到类似：0.0.0.0:80->80/tcp, 0.0.0.0:443->443/tcp\n```\n\n> ⚠️ **注意**：1Panel 在更新或重启应用时可能会覆盖 `.env` 的修改。如果发现端口又变回去了，需要再次修改。也可以通过 1Panel 的界面直接修改端口映射。\n\n---\n\n### 步骤 2：重置 NPM 管理员密码\n\n如果忘记了 NPM 的管理密码，或者需要通过 API 操作但无法登录，可以通过修改数据库来重置密码。\n\n#### 2.1 安装 sqlite3\n\n```bash\n# Debian/Ubuntu\nsudo apt update && sudo apt install -y sqlite3\n\n# CentOS/RHEL\nsudo yum install -y sqlite\n```\n\n#### 2.2 找到 NPM 数据库文件\n\n```bash\n# 查找数据库文件\nfind / -name "*.sqlite" 2>/dev/null\n# 或者\nfind / -name "database.sqlite" 2>/dev/null\n```\n\n典型路径：`/opt/nginx-proxy-manager/data/database.sqlite`\n\n#### 2.3 查看当前用户\n\n```bash\nsqlite3 /path/to/database.sqlite "SELECT id, email FROM user;"\n```\n\n输出类似：\n\n```\n1|admin@example.com\n```\n\n#### 2.4 生成 bcrypt 密码哈希\n\n```bash\n# 方法一：使用 Python\npython3 -c "import bcrypt; print(bcrypt.hashpw(b'your-new-password', bcrypt.gensalt()).decode())"\n\n# 方法二：使用 Node.js（如果有的话）\nnode -e "const bcrypt = require('bcryptjs'); console.log(bcrypt.hashSync('your-new-password', 10));"\n\n# 方法三：在线工具生成 bcrypt hash（注意安全，生成后立即使用）\n```\n\n#### 2.5 更新数据库中的密码\n\n```bash\nsqlite3 /path/to/database.sqlite "UPDATE auth SET secret='$2b$10$YOUR_HASH_HERE' WHERE user_id=1;"\n```\n\n> 💡 **注意**：在 shell 中使用 bcrypt hash 时，注意 `$` 符号需要转义！\n\n#### 2.6 重启 NPM 容器\n\n```bash\ndocker restart nginx-proxy-manager\n# 或者\ndocker compose restart\n```\n\n现在可以使用新密码登录 NPM 管理界面（`http://YOUR_IP:81`）。\n\n---\n\n### 步骤 3：通过 NPM API 创建代理主机\n\n#### 3.1 获取 API Token\n\n```bash\ncurl -X POST http://localhost:81/api/tokens \n  -H "Content-Type: application/json" \n  -d '{\n    "identity": "admin@example.com",\n    "secret": "***"\n  }'\n```\n\n响应：\n\n```json\n{\n  "token": "***",\n  "expires": "2026-05-25T00:00:00.000Z",\n  "user_id": 1\n}\n```\n\n保存这个 token：\n\n```bash\nexport NPM_TOKEN="***"\n```\n\n#### 3.2 创建代理主机\n\n```bash\ncurl -X POST http://localhost:81/api/nginx/proxy-hosts \n  -H "Authorization: Bearer *** \n  -H "Content-Type: application/json" \n  -d '{\n    "domain_names": ["whalemalus.com", "www.whalemalus.com"],\n    "forward_host": "172.18.0.1",\n    "forward_port": 2222,\n    "forward_scheme": "http",\n    "ssl_forced": true,\n    "block_exploits": true,\n    "allow_websocket_upgrade": true,\n    "http2_support": true,\n    "caching_enabled": false\n  }'\n```\n\n> 🔑 **关键点**：`forward_host` 使用 `172.18.0.1` 而非 `host.docker.internal`！\n>\n> `172.18.0.1` 是 Docker 默认桥接网络的网关 IP，相当于宿主机在 Docker 网络中的地址。而 `host.docker.internal` 在 Linux 上不一定可用（macOS/Windows 通常支持）。\n\n#### 3.3 验证代理主机创建成功\n\n```bash\ncurl -s http://localhost:81/api/nginx/proxy-hosts \n  -H "Authorization: Bearer *** | python3 -m json.tool\n```\n\n你应该能看到刚创建的代理主机配置，`id` 应该是 `1`（或下一个递增的数字）。\n\n---\n\n### 步骤 4：上传 Cloudflare Origin Certificate\n\n#### 4.1 在 Cloudflare 生成 Origin Certificate\n\n1. 登录 [Cloudflare Dashboard](https://dash.cloudflare.com)\n2. 选择你的域名 `whalemalus.com`\n3. 进入 **SSL/TLS** → **Origin Server**\n4. 点击 **Create Certificate**\n5. 保持默认设置：\n   - 私钥类型：RSA (2048)\n   - 主机名：`whalemalus.com`, `*.whalemalus.com`\n   - 有效期：15 年\n6. 点击 **Create**\n7. **复制保存**生成的证书和私钥（私钥只显示一次！）\n\n```\n-----BEGIN CERTIFICATE-----\nMIIE...（你的 Origin Certificate）\n-----END CERTIFICATE-----\n```\n\n```\n[REDACTED PRIVATE KEY]\n```\n\n#### 4.2 通过 NPM API 上传证书\n\n```bash\ncurl -X POST http://localhost:81/api/nginx/certificates \n  -H "Authorization: Bearer *** \n  -H "Content-Type: application/json" \n  -d '{\n    "provider": "other",\n    "nice_name": "Cloudflare Origin - whalemalus.com",\n    "domain_names": ["whalemalus.com", "*.whalemalus.com"],\n    "meta": {\n      "certificate": "-----BEGIN CERTIFICATE-----\nMIIE...（完整证书内容）\n-----END CERTIFICATE-----",\n      "certificate_key": "[REDACTED PRIVATE KEY]"\n    }\n  }'\n```\n\n> ⚠️ **注意**：证书和私钥中的换行符需要用 `\n` 替代。可以使用以下命令处理：\n\n```bash\n# 将证书文件内容转为一行（替换换行为 \n）\nCERT=$(cat origin-cert.pem | tr '\n' '\n' | sed ':a;N;$!ba;s/\n/\\\\\\\\\n/g')\nKEY=$(cat origin-key.pem | tr '\n' '\n' | sed ':a;N;$!ba;s/\n/\\\\\\\\\n/g')\n\n# 然后在 JSON 中使用 $CERT 和 $KEY\n```\n\n#### 4.3 验证证书上传成功\n\n```bash\n# 检查证书文件是否生成\nls -la /opt/nginx-proxy-manager/data/custom_ssl/npm-*/\n\n# 应该看到类似\n# cert.pem\n# key.pem\n```\n\n---\n\n### 步骤 5：手动创建 NPM Nginx 配置\n\n这是最关键的一步！通过 API 创建代理主机和上传证书后，**Nginx 的配置文件可能不会自动生成**。我们需要手动创建。\n\n#### 5.1 找到 NPM 的 Nginx 配置目录\n\n```bash\nls /opt/nginx-proxy-manager/data/nginx/proxy_host/\n# 如果为空或不存在配置文件，就需要手动创建\n```\n\n#### 5.2 创建代理主机配置\n\n```bash\ncat > /opt/nginx-proxy-manager/data/nginx/proxy_host/1.conf << 'EOF'\n# whalemalus.com 代理配置\nserver {\n    listen 443 ssl http2;\n    listen [::]:443 ssl http2;\n\n    server_name whalemalus.com www.whalemalus.com;\n\n    # Cloudflare Origin Certificate\n    ssl_certificate /data/custom_ssl/npm-<ID>/cert.pem;\n    ssl_certificate_key /data/custom_ssl/npm-<ID>/key.pem;\n\n    # SSL 配置\n    ssl_protocols TLSv1.2 TLSv1.3;\n    ssl_ciphers HIGH:!aNULL:!MD5;\n    ssl_prefer_server_ciphers on;\n    ssl_session_cache shared:SSL:10m;\n    ssl_session_timeout 5m;\n\n    # HSTS\n    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;\n\n    # 代理配置\n    location / {\n        proxy_pass http://172.18.0.1:2222;\n        proxy_set_header Host $host;\n        proxy_set_header X-Real-IP $remote_addr;\n        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;\n        proxy_set_header X-Forwarded-Proto $scheme;\n\n        # WebSocket 支持\n        proxy_http_version 1.1;\n        proxy_set_header Upgrade $http_upgrade;\n        proxy_set_header Connection "upgrade";\n\n        # 超时设置\n        proxy_connect_timeout 60s;\n        proxy_send_timeout 60s;\n        proxy_read_timeout 60s;\n    }\n\n    # 访问日志和错误日志\n    access_log /data/logs/proxy-host-1_access.log proxy;\n    error_log /data/logs/proxy-host-1_error.log warn;\n}\nEOF\n```\n\n#### 5.3 创建默认服务器配置\n\n为了防止没有匹配到域名的请求暴露信息，需要创建一个默认服务器：\n\n```bash\n# 先生成一个自签名证书用于默认服务器\nopenssl req -x509 -nodes -days 3650 \n  -newkey rsa:2048 \n  -keyout /opt/nginx-proxy-manager/data/nginx/default_key.pem \n  -out /opt/nginx-proxy-manager/data/nginx/default_cert.pem \n  -subj "/CN=default"\n\ncat > /opt/nginx-proxy-manager/data/nginx/default.conf << 'EOF'\n# 默认服务器 - 拒绝未知域名的 SSL 连接\nserver {\n    listen 443 ssl default_server;\n    listen [::]:443 ssl default_server;\n\n    server_name _;\n\n    # 使用自签名证书\n    ssl_certificate /data/nginx/default_cert.pem;\n    ssl_certificate_key /data/nginx/default_key.pem;\n\n    # 直接拒绝连接\n    ssl_reject_handshake on;\n}\n\nserver {\n    listen 80 default_server;\n    listen [::]:80 default_server;\n\n    server_name _;\n\n    return 444;\n}\nEOF\n```\n\n> 💡 **`ssl_reject_handshake on;`** 的作用：当请求的 SNI（Server Name Indication）不匹配任何已配置的域名时，直接拒绝 TLS 握手，而不是返回默认证书。这样可以防止通过 IP
