# 周一早上的定时炸弹：一个隐藏了 102 轮迭代的日期 Bug\n\n> **摘要**：PageWise 的 `getWeeklyStats` 测试在周一突然全红，根因是测试代码用 `Date.now() - N天` 构造时间戳，但生产代码以周一 00:00:00 作为一周起点。当 `now - 1天` 落到周日时，数据被归入上周，断言自然失败。本文还原整个排查和修复过程。\n>\n> **关键词**：`飞轮迭代` `Flaky Test` `日期处理` `Chrome Extension` `自动化测试`\n\n---\n\n## 楔子\n\n周一早上九点，飞轮迭代引擎准时启动。按照惯例，它会扫描 TODO 清单、跑一遍测试、找到下一个任务开始迭代。但今天不一样——5,887 个测试里有 1 个挂了。\n\n不是新写的代码出了问题。是那行存在了几个月、跑了上百次的测试，突然在今天这个特定的周一，安静地失败了。\n\n## 引言\n\nFlaky Test（不稳定测试）是自动化测试体系里最隐蔽的敌人。它不像编译错误那样大声宣告自己的存在，而是安静地在某个特定条件下才露出真面目。更可怕的是，当你的飞轮迭代引擎已经完成了 102 轮迭代、所有 TODO 都打上了勾，这种"偶发"失败很容易被当成噪音忽略掉。\n\n这次的故事，就是关于一个在周一才会爆炸的日期炸弹，以及飞轮引擎如何在"无事可做"的状态下，依然找到了值得修复的问题。\n\n## 📖 目录\n\n1. // 0=Sunday, 1=Monday, ...\n  const diff = day === 0 ? 6 : day - 1; // 周日回退6天，其他回退 day-1 天\n  d.setDate(d.getDate() - diff);\n  d.setHours(0, 0, 0, 0);\n  return d;\n}\n```\n\n这个算法保证了：无论今天是周几，`weekStart` 始终指向本周一的 00:00:00。\n\n---\n\n## 实战指南\n\n### 第一步：发现问题\n\n飞轮迭代引擎运行全量测试：\n\n```bash\nnode --test tests/test-*.js\n```\n\n输出显示 5,886 通过，1 失败：\n\n```\n✗ 应汇总本周的会话数据 (tests/test-review-session.js:327)\n  AssertionError: expected 1 >= 2\n```\n\n关键信息：\n- 失败文件：`tests/test-review-session.js`\n- 失败行号：327\n- 断言内容：期望 `totalSessions >= 2`，实际只有 1\n\n### 第二步：定位根因\n\n查看失败的测试代码：\n\n```javascript\n// 测试代码（有问题的版本）\nconst now = Date.now();\nconst dayMs = 24 * 60 * 60 * 1000;\n\n// 创建 3 个会话，分别在今天、昨天、前天\ncreateSession({ timestamp: now });\ncreateSession({ timestamp: now - dayMs });      // 周一 - 1天 = 周日\ncreateSession({ timestamp: now - dayMs * 2 });  // 周一 - 2天 = 周六\n\nconst stats = getWeeklyStats();\nassert(stats.totalSessions >= 2);  // 期望至少 2 个在本周\n```\n\n问题在于：测试假设 `now - 1天` 和 `now - 2天` 都在"本周"内。但当 `now` 是周一时：\n\n| 时间戳 | 对应日期 | 是否在本周（周一为起点） |\n|--------|----------|------------------------|\n| `now` | 周一 | ✅ 是 |\n| `now - 1天` | 周日 | ❌ 否（属于上周） |\n| `now - 2天` | 周六 | ❌ 否（属于上周） |\n\n只有 1 个会话在本周，`totalSessions = 1`，断言 `>= 2` 失败。\n\n### 第三步：修复\n\n核心思路：**让测试使用和生产代码相同的 weekStart 算法**，而不是用相对偏移。\n\n```javascript\n// 修复后的测试代码\nconst now = Date.now();\nconst dayMs = 24 * 60 * 60 * 1000;\n\n// 计算本周起始点（与生产代码一致）\nconst weekStart = getWeekStart(now);\n\n// 将会话锚定到 weekStart 的相对位置\ncreateSession({ timestamp: weekStart });              // 周一\ncreateSession({ timestamp: weekStart + dayMs });       // 周二\ncreateSession({ timestamp: weekStart + dayMs * 2 });   // 周三\n\nconst stats = getWeeklyStats();\nassert.strictEqual(stats.totalSessions, 3);  // 精确断言\n```\n\n两个关键改动：\n\n1. **时间锚点从 `now` 改为 `weekStart`**：无论今天是周几，`weekStart + 0/1/2天` 始终在本周内\n2. **断言从 `>= 2` 改为 `=== 3`**：精确断言比模糊断言更能暴露问题\n\n### 第四步：验证\n\n```bash\nnode --test tests/test-*.js\n# 5,887 pass / 0 fail\n```\n\n全量回归通过，提交：\n\n```bash\ngit commit -m "fix: flaky getWeeklyStats test - day-of-week dependency (飞轮迭代 R1)"\ngit push origin master\n```\n\nCommit: `9f716fe`\n\n---\n\n## 踩坑记录\n\n### 坑 1：`now - N天` 不等于"N 天前的同一时刻"\n\n**现象**：测试在周二到周日都通过，唯独周一失败。\n\n**原因**：`Date.now() - N * dayMs` 做的是毫秒级减法，不考虑日历边界。在周一减去 1 天的毫秒数，得到的是周日同一时刻的时间戳——这在"以周一为起点"的周统计中属于上周。\n\n**教训**：涉及"本周"、"本月"等日历概念时，永远先计算边界（weekStart、monthStart），再用边界做偏移，而不是用 `now` 做偏移。\n\n### 坑 2：模糊断言掩盖了问题\n\n**现象**：测试写的是 `>= 2` 而不是 `=== 3`，所以当只有 1 个会话在本周时才失败。如果有 2 个会话在本周，测试会通过，但逻辑上可能仍然不对。\n\n**教训**：当你确切知道期望值时，用精确断言（`===`、`==`）。模糊断言（`>=`、`<=`）只在确实存在合理范围时才使用。\n\n### 坑 3：飞轮引擎的测试解析器有 Bug\n\n**现象**：引擎脚本的 `run_tests()` 函数用 `output.count("✓")` 统计通过数，但 `node --test` 输出的是 TAP 格式（`# pass` / `# fail`），导致引擎始终报告 0/0。\n\n**影响**：飞轮引擎无法正确感知测试状态，可能把失败的测试误判为通过。\n\n**状态**：已记录为已知缺陷，待修复。\n\n---\n\n## 总结与展望\n\n### 核心收获\n\n1. **日期测试必须考虑周一效应**：任何以"周"为单位的统计，测试数据的构造都必须基于 `weekStart` 而非 `now`\n2. **飞轮引擎的"空闲"状态也有价值**：即使没有新的 TODO 任务，全量测试套件本身就是待维护资产\n3. **精确断言优于模糊断言**：`=== 3` 比 `>= 2` 更能暴露边界问题\n\n### 最佳实践\n\n- **日期测试三原则**：\n  1. 用 `weekStart`/`monthStart` 做锚点，不用 `now`\n  2. 测试数据覆盖所有星期几（至少跑一次周一的场景）\n  3. 断言精确值，不断言范围\n\n- **Flaky Test 处理流程**：\n  1. 复现：确认在什么条件下失败\n  2. 定位：找到对环境的隐式依赖\n  3. 修复：消除隐式依赖，使测试确定性通过\n  4. 验证：全量回归，确保不引入新问题\n\n### 延伸阅读\n\n- PageWise 飞轮迭代引擎的完整架构：从 R1 到 R102 的 102 轮迭代实践\n- Chrome Extension 的自动化测试方案：node:test + E2E 的混合策略\n- 飞轮引擎已知缺陷的修复计划：测试解析器、Phase 2 稳定性、Claude Code CLI 认证\n
