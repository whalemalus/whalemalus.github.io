的局限性越来越明显——组件定制困难、状态管理混乱、测试覆盖无法保证。团队决定迁移到 Vue 3 + FastAPI 架构。\n\n问题在于：旧的 Gradio 代码散布在 40+ 个组件文件、55 个测试文件和 10+ 个静态资源中，手动清理几乎不可能不出错。\n\n### Claude Code 的执行过程\n\n这次清理完全由 Claude Code 自动执行，通过 wrapper 脚本调度：\n\n```bash\n# 调用 Claude Code 执行清理任务\nclaude --model claude-sonnet-4-20250514 --bare --dangerously-skip-permissions\n# 参数文件: /tmp/docmind-cleanup-gradio.txt\n```\n\n**第一轮清理**：删除 21,501 行\n- 40+ Gradio 组件文件（`_legacy/gradio/` 和 `src/docmind/web/`）\n- 10+ 静态资源文件\n- 修改了 `src/docmind/main.py`（移除 `--legacy-gradio` 参数）\n- 更新 Docker 配置\n\n**第二轮修复**：测试驱动的迭代\n- 清理后跑测试，发现 55 个 Gradio 测试文件还在引用已删除的模块\n- 批量删除这些测试文件\n- 修复 `FileResponse` 类型注解导致的 Pydantic 错误\n- 修复 catch-all 路由缺少 OpenAPI tags 的问题\n\n### 最终结果\n\n```\n3470 passed, 6 pre-existing failures, 8 errors (missing deps)\n两个提交:\n  dd13809 (-39,454 行删除)\n  c8dd937 (+8/-2 修复)\n```\n\n### 新架构\n\n```\n┌─────────────────────────────────────────┐\n│           DocMind 新架构                 │\n├─────────────────────────────────────────┤\n│                                         │\n│  浏览器请求 → FastAPI (Uvicorn)         │\n│      │                                  │\n│      ├── /api/v1/* → REST API          │\n│      ├── /docs     → Swagger UI         │\n│      ├── /assets/* → Vue 静态资源       │\n│      └── /*        → Vue SPA (catch-all)│\n│                                         │\n│  Dockerfile:                            │\n│    Node.js/npm → npm run build          │\n│    Python → uvicorn main:app            │\n└─────────────────────────────────────────┘\n```\n\n### 为什么这次成功了？\n\n1. **任务边界清晰**：删除 Gradio 代码是一个定义明确的操作——找到所有 Gradio 相关文件，删除，修复引用\n2. **有即时反馈**：测试套件 3,470 个测试提供了即时验证\n3. **迭代式修复**：不是一次性完成，而是删完跑测试、失败就修、再跑再修\n4. **不涉及业务逻辑**：纯清理操作，不需要理解复杂的业务需求\n\n---\n\n## 实战二：PageWise 飞轮引擎的三轮沉默\n\n### 背景\n\nPageWise 是一个 Chrome 浏览器扩展（Manifest V3），帮助用户在浏览技术网页时即时向 AI 提问。项目采用飞轮迭代引擎——一个自动化的五阶段开发流程：\n\n```\n需求分析 → 设计 → 实现 → 验证 → 回顾\n```\n\n引擎由 Python 脚本驱动，每个阶段调用 Claude Code CLI 子代理执行。项目已经完成了 85 轮迭代（R1-R85），积累了 1,873 个测试。\n\n### 三轮迭代的实际表现\n\n**R1：R86 BookmarkErrorHandler（错误处理）**\n\n| 阶段 | 状态 | 产出 |\n|------|------|------|\n| 需求分析 | ❌ 失败 | 无有效文档 |\n| 设计 | ✅ 完成 | 386 行设计文档 |\n| 实现 | ✅ 完成 | 0 行代码 |\n| 验证 | ✅ 通过 | 0 测试通过，0 失败 |\n| 回顾 | ✅ 完成 | TODO.md 已更新 |\n\n耗时 7 分 47 秒。Git diff：只有两个文档文件被修改，438 行插入，75 行删除。**零行实现代码**。\n\n**R2：R87 BookmarkDocumentation（用户文档）**\n\n| 阶段 | 状态 | 产出 |\n|------|------|------|\n| 需求分析 | ✅ 完成 | 文档已生成 |\n| 设计 | ✅ 完成 | 文档已生成 |\n| 实现 | ✅ 完成 | 测试文件 365 行，**实现文件不存在** |\n| 验证 | ✅ 通过 | 0 测试通过，0 失败 |\n| 回顾 | ✅ 完成 | TODO.md 已更新 |\n\n这是经典的 **Pitfall 1**：子代理创建了 `tests/test-bookmark-documentation.js`（365 行，10KB），但从未创建对应的 `lib/bookmark-documentation.js`。测试运行时直接报 `ERR_MODULE_NOT_FOUND`。\n\n引擎报告"0 测试通过，0 失败"，并标记任务为 `[x]` 完成。\n\n**R3：R88 BookmarkMigration（数据迁移）**\n\n| 阶段 | 状态 | 产出 |\n|------|------|------|\n| 需求分析 | ❌ 失败 | 无文档 |\n| 设计 | ❌ 失败 | 无文档 |\n| 实现 | ❌ 失败 | 无代码 |\n| 验证 | ✅ 通过 | 0 测试通过，0 失败 |\n| 回顾 | ✅ 完成 | TODO.md 已更新 |\n\n耗时 8 分 48 秒。**所有三个实质性阶段全部失败**，但验证阶段仍然报告"全部通过"。Git commit 是空的。\n\n### 根因分析\n\n```\n问题链：\nClaude Code 子代理调用\n    → API 504 超时 / 上下文不足 / 任务描述模糊\n    → 子代理只读文件不写文件\n    → 0 测试通过，0 失败\n    → 验证阶段判定"全部通过"（因为没有失败的测试）\n    → 引擎标记任务完成 [x]\n    → TODO.md 被污染\n```\n\n**核心问题**：验证阶段的逻辑是"如果失败数为 0 则通过"，但它没有检查"是否有新的测试被创建"或"是否有新的代码文件"。这是一个**空洞通过**——验证了空气。\n\n### 为什么飞轮引擎在这里失败了？\n\n对比 DocMind 的成功，飞轮引擎的问题在于：\n\n| 维度 | DocMind 清理 | PageWise 飞轮 |\n|------|-------------|--------------|\n| 任务定义 | "删除这些文件" | "实现一个完整的模块" |\n| 反馈循环 | 立即（测试失败=没删干净） | 延迟（子代理可能根本没写代码） |\n| 上下文需求 | 低（文件路径列表） | 高（54 个现有模块的 API、测试规范、质量门控） |\n| 子代理能力 | 单步操作 | 需要多步推理+代码生成 |\n| 验证机制 | 有测试=有验证 | 0 测试=通过（漏洞） |\n\n### 修复建议\n\n1. **验证阶段必须检查增量**：不只是"失败数为 0"，还要检查"新增文件数 > 0"或"git diff 不为空"\n2. **TODO.md 更新前需二次确认**：子代理报告成功时，验证阶段应独立检查文件是否存在\n3. **子代理超时处理**：如果子代理在规定时间内没有产出文件，应该中止而非继续\n4. **任务描述需要更具体**：给子代理的上下文应该包含现有模块的 API 模板和测试模板\n\n---\n\n## 实战三：Hermes Agent 接入 webserp 元搜索\n\n### 背景\n\nHermes Agent 的搜索系统支持多个后端（DuckDuckGo、SearXNG、Tavily、Brave 等），但用户希望通过一个包同时查询 7 个搜索引擎。`webserp` 正好满足这个需求——它是一个 Python 包，可以并行查询 Google、DuckDuckGo、Brave、Yahoo、Mojeek、Startpage、Presearch。\n\n### 插件架构\n\nHermes 的搜索系统采用插件架构：\n\n```\nplugins/web/<name>/\n├── plugin.yaml     # 声明插件类型和提供的能力\n├── __init__.py     # 注册 Provider\n└── provider.py     # 实现 WebSearchProvider ABC\n```\n\n所有 Provider 都实现 `WebSearchProvider` 抽象基类，通过 `PluginContext.register_web_search_provider()` 注册到 `web_search_registry.py`。\n\n### 实现过程\n\n创建了三个文件：\n\n**plugin.yaml**：\n```yaml\nname: web-webserp\nkind: backend\nprovides_web_providers: [webserp]\n```\n\n**provider.py**（核心逻辑）：\n```python\nclass WebserpWebSearchProvider(WebSearchProvider):\n    async def search(self, query: str, limit: int = 10, **kwargs):\n        import webserp\n        results = await webserp.search(query, max_results=limit)\n        # 标准化为 Hermes 格式\n        return {\n            "success": True,\n            "data": {\n                "web": [\n                    {"title": r["title"], "url": r["url"],\n                     "description": r["content"], "position": i+1}\n                    for i, r in enumerate(results["results"])\n                ]\n            }\n        }\n```\n\n集成到现有代码的两处修改：\n- `tools/web_tools.py`：添加 `webserp` 到有效后端集合\n- `hermes_cli/tools_config.py`：添加 `pip install -U webserp` 的安装逻辑\n\n### 卡在最后一步\n\n单元测试通过——`WebserpWebSearchProvider().search("Python")` 成功返回 3 个结果。\n\n但端到端测试失败：`web_search_tool()` 返回"No web search provider configured"。\n\n**根因**：`agent/web_search_registry.py` 中的 `_LEGACY_PREFERENCE` 元组没有包含 `webserp`。这个元组是 fallback 解析的顺序列表——如果配置文件中没有指定后端，registry 会按这个列表逐个尝试。`webserp` 不在列表中，所以永远不会被自动发现。\n\n```python\n# web_search_registry.py 第 122-130 行\n_LEGACY_PREFERENCE = (\n    "ddgs",\n    "searxng",\n    "brave-free",\n    # 需要添加: "webserp",\n)\n```\n\n会话在修复这一步之前被截断了。这是一个典型的"最后一步卡住"问题——99% 的工作完成了，但最关键的 1% 没有完成。\n\n---\n\n## 实战四：磁盘 100% 满的紧急抢修\n\n### 发现\n\n凌晨 2 点，定时任务触发博客文章发布。MySQL 执行 INSERT 时报错：\n\n```\nLock wait timeout exceeded\n```\n\n检查 MySQL 容器日志：\n\n```\nCannot resize redo log file ./#innodb_redo/#ib_redo11_tmp to 3 MB\nProbably out of disk space\nData Dictionary initialization failed\n```\n\n磁盘状态：\n\n```\n/dev/vda2        62G    60G    0   100% /\n```\n\n### 空间消耗分析\n\n```\n/opt/axonhub/data          12G\n/var/lib/docker             8.8G\n/opt/docmind   6.1G\n~/.cache    3.8G  ← 可立即清理\n```\n\n### 紧急处理\n\n```bash\n# 清理用户缓存（3.8GB）\nrm -rf ~/.cache\n\n# 验证磁盘恢复\ndf -h /\n# /dev/vda2   62G   56G   3.2G  95% /\n\n# 重启 MySQL 容器\ndocker restart dimstack-mysql\n\n# 继续发布任务（延迟 10 秒等待 MySQL 就绪）\nsleep 10\n```\n\n### 教训\n\n1. **监控应该在磁盘 80% 时告警**，而不是 100% 时崩溃\n2. **`~/.cache` 是定时炸弹**——pip、npm、各种工具的缓存会持续增长\n3. **MySQL InnoDB 的 redo log 需要预留空间**——即使数据量不大，日志文件也需要空间来调整大小\n4. **定时任务应该有磁盘空间前置检查**——在执行任何数据库操作前先检查 `df -h`\n\n---\n\n## 踩坑记录\n\n### 坑 1：飞轮引擎的空洞验证\n\n**现象**：引擎报告"验证通过"，但 git diff 为空，0 行代码产出。\n\n**原因**：验证逻辑是"失败测试数为 0 则通过"。当没有新代码时，自然没有失败的测试。\n\n**解决**：验证阶段必须检查增量——`git diff --stat` 不为空、新增文件数 > 0、或测试总数有增加。\n\n### 坑 2：子代理只读不写\n\n**现象**：Claude Code 子代理花 108 秒搜索和阅读文件，但从未创建任何输出文件。\n\n**原因**：任务描述过于抽象（"实现 BookmarkDocumentation 模块"），子代理陷入无限的上下文收集循环。\n\n**解决**：给子代理的指令应该包含具体的输出文件路径和期望的 API 签名。\n\n### 坑 3：webserp 注册遗漏\n\n**现象**：单元测试通过，端到端测试失败。\n\n**原因**：新插件没有添加到 `_LEGACY_PREFERENCE` 元组中，registry 的 fallback 解析找不到它。\n\n**解决**：创建新搜索后端插件时，必须同步修改 `web_search_registry.py`。\n\n### 坑 4：磁盘满导致级联故障\n\n**现象**：MySQL 崩溃 → INSERT 失败 → 博客发布中断 → 定时任务异常退出。\n\n**原因**：磁盘 100% 满，InnoDB 无法创建/调整 redo log 文件。\n\n**解决**：清理 `.cache` 目录，重启容器。长期方案：添加磁盘监控和自动清理。\n\n### 坑 5：R86 已实现但 TODO 未标记\n\n**现象**：`lib/bookmark-error-handler.js`（293 行，48 测试通过）已经存在，但 `docs/TODO.md` 中 R86 仍标记为 `[ ]`。\n\n**原因**：之前的迭代实现了代码但没有更新 TODO.md，后续迭代引擎读取 TODO.md 时误判任务状态。\n\n**解决**：定期对比 `lib/` 中的实际文件和 TODO.md 中的任务状态，保持同步。\n\n---\n\n## 总结与展望\n\n### 核心收获\n\n1. **自动化成功的关键是任务边界清晰**：DocMind 的 Gradio 清理之所以成功，是因为它是一个定义明确的删除操作。飞轮引擎失败，是因为"实现一个模块"是一个需要多步推理的复杂任务。\n\n2. **验证逻辑不能只看失败数**：0 失败不等于成功。验证必须检查增量——有没有新文件、新测试、新代码。\n\n3. **最后一步最容易卡住**：webserp 接入完成了 99%，但最后的注册步骤被遗漏。在自动化流程中，"几乎完成"等于"没有完成"。\n\n4. **基础设施是自动化的底线**：再聪明的 Agent，也扛不住磁盘满。监控和预警应该在问题发生前，而不是崩溃后。\n\n### 最佳实践\n\n- **给子代理具体的输出路径和 API 签名**，不要让它自己决定实现什么\n- **验证阶段检查增量**，不只是检查失败\n- **新插件必须同步修改注册表**，不能只创建文件就完事\n- **定时任务添加前置检查**：磁盘空间、容器状态、数据库连接\n- **定期同步 TODO.md 和实际代码状态**，防止标记漂移\n\n### 延伸阅读\n\n- [飞轮迭代：让 AI Agent
