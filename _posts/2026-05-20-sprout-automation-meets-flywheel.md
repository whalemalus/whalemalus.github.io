## 核心概念

### 概念1：Ingest = Iteration（同构操作）

LLM Wiki 的 ingest 操作和飞轮迭代的单次迭代惊人地相似。两者的核心都是：**输入 → 编译 → 输出 → 记录**。

区别在于编译的对象不同：
- LLM Wiki 编译的是**知识**（从原始来源到结构化页面）
- 飞轮编译的是**代码**（从需求到实现）

这意味着两个领域的最佳实践可以互相借鉴。飞轮的"产出物验证"可以直接应用到 Wiki 的 ingest 流程。

### 概念2：Silent Failure 是知识管理的隐性杀手

飞轮迭代中最危险的失败模式是静默失败：Agent 返回"成功"但实际没完成。

在 LLM Wiki 中同样存在：
- **Ingest 静默失败**：LLM 说"已创建 5 个页面"但实际只创建了 3 个
- **Cross-reference 静默失败**：wikilinks 指向不存在的页面
- **Synthesis 静默失败**：发芽报告只是拼凑了页面摘要，不是真正的"新洞察"

### 概念3：Wiki Lint = 知识的 Guard Agent

飞轮用 Guard Agent 检测引擎健康。Wiki 用 Lint 检测知识库健康。两者的检查项完全对应：

| Wiki Lint | Guard Agent | 检查什么 |
|-----------|-------------|---------|
| 孤立页面检查 | TODO 未变化检测 | 是否有"被遗忘"的内容 |
| 断链检查 | 文件未修改检测 | 是否有"断裂"的引用 |
| 索引完整性 | CI 状态监控 | 是否有"遗漏"的记录 |
| 矛盾标记 | 资源使用监控 | 是否有"冲突"的内容 |

## 踩坑记录

### 踩坑1：发芽报告 ≠ 博客文章

发芽报告的结构（知识网络概览 → 意外连接 → 矛盾 → 洞察 → 行动建议）和博客文章的结构（楔子 → 引言 → 全景地图 → 核心概念 → 实战指南 → 踩坑 → 总结）完全不同。

**解决方案**：发芽报告生成后，需要一个"格式转换"步骤，把知识碰撞的内容重新组织成博客读者能消化的结构。

### 踩坑2：理论和实践的碰撞需要"翻译"

LLM Wiki 的术语（ingest、query、lint、synthesis）和飞轮的术语（iteration、guard、silent failure）说的是同一件事，但用的是不同的语言。

**解决方案**：在碰撞报告中建立术语对照表，让两个领域的读者都能理解。

### 踩坑3：自动化 vs 质量的永恒矛盾

飞轮追求速度（快速循环），Wiki 追求质量（深度交叉引用）。把飞轮的速度逻辑应用到 Wiki 会产生大量低质量页面。

**解决方案**：分层处理 — 飞轮负责"粗加工"（快速创建草稿），Wiki 的 Lint 负责"精加工"（定期清理和提升质量）。

## 总结与展望

### 核心收获

1. **LLM Wiki 和飞轮迭代是同构的** — 两者的操作可以互相借鉴
2. **静默失败是知识管理的隐性杀手** — 需要独立的验证机制
3. **知识管理需要自己的 CI/CD** — Ingest 后自动 Lint，发芽后自动发布

### 最佳实践

- 每次 ingest 后运行 lint（CI）
- 发芽报告自动发布到博客（CD）
- 用脚本而不是 LLM 验证产出物（Guard Agent）
- 分层处理：快速粗加工 + 定期精加工
