# 当飞轮引擎自己也需要修 Bug：自动化迭代的信任危机\n\n> **摘要**：PageWise 项目的飞轮迭代引擎连续两次运行失败——API 密钥占位符未替换、语法错误导致所有 Claude Code 调用静默失败，引擎却报告"成功"。本文记录从发现到修复的全过程，以及自动化系统中"假成功"的防范经验。\n>\n> **关键词**：`飞轮迭代` `Claude Code` `自动化` `Bug 排查` `BookmarkLinkChecker`\n\n---\n\n## 楔子\n\n5 月 4 日下午两点，飞轮迭代引擎准时启动。日志显示五个阶段全部完成——需求、设计、实现、验证、回顾，一行行绿色的 ✅ 让人安心。但当我打开项目目录，发现一个新文件都没有。\n\n引擎说它成功了，代码却不存在。\n\n这不是第一次。四个小时前的 R2 轮次，同样的事情已经发生过一次。引擎跑完了所有阶段，git 也提交了，但提交的只有一个需求文档——没有一行代码。\n\n两次"成功"，两次空跑。飞轮引擎本身，成了最需要修的那个 Bug。\n\n## 引言\n\nPageWise 是一个 Chrome 浏览器扩展，用于 AI 辅助网页阅读和知识管理。项目的开发采用"飞轮迭代"方法——一个自动化引擎驱动 Claude Code 完成需求分析、设计、编码、测试、回顾五个阶段，循环往复。\n\n截至 5 月初，飞轮已经跑完了 51 轮迭代（R43-R51），积累了 2000 多个测试用例。引擎脚本 `pagewise-iteration-engine.py` 被设计成"放着不管就能跑"的自动化系统。\n\n但"放着不管"恰恰是问题所在。当引擎本身出现 Bug，而你又没有验证机制时，自动化就变成了一个制造虚假进度的机器。\n\n## 全景地图：飞轮迭代系统的架构\n\n```\n┌─────────────────────────────────────────────────────┐\n│              飞轮迭代系统                              │\n├─────────────────────────────────────────────────────┤\n│                                                     │\n│  ┌──────────────┐    ┌──────────────┐               │\n│  │ 迭代引擎      │───→│  Claude Code  │               │\n│  │ (Python 脚本) │    │  (子代理)     │               │\n│  └──────┬───────┘    └──────┬───────┘               │\n│         │                   │                        │\n│         ▼                   ▼                        │\n│  ┌──────────────┐    ┌──────────────┐               │\n│  │ TODO.md      │    │ 项目源码      │               │\n│  │ (任务队列)    │    │ (lib/tests/)  │               │\n│  └──────────────┘    └──────────────┘               │\n│                                                     │\n│  五个阶段：                                           │\n│  需求 → 设计 → 实现 → 验证 → 回顾                     │\n│                                                     │\n│  ⚠️ 故障点：API 密钥传递                              │\n│  ⚠️ 故障点：错误检测逻辑                              │\n└─────────────────────────────────────────────────────┘\n```\n\n本文的学习路径：\n1. **引擎怎么工作的** — 理解五阶段流程\n2. **哪里出了问题** — 两个隐蔽的 Bug\n3. **为什么没被发现** — 错误检测的盲区\n4. **怎么修的** — 手动修复 + 补充验证\n\n## 核心概念\n\n### 飞轮迭代引擎\n\n飞轮迭代引擎是一个 Python 脚本，负责自动化驱动 Claude Code 完成开发任务。它从 `TODO.md` 读取下一个待完成的任务，然后依次调用 Claude Code 执行五个阶段：\n\n1. **需求分析**：生成 `docs/REQUIREMENTS-ITER{n}.md`\n2. **设计**：生成 `docs/DESIGN-ITER{n}.md`\n3. **实现**：生成代码文件到 `lib/` 和 `tests/`\n4. **验证**：运行测试套件\n5. **回顾**：生成迭代报告，更新 TODO.md\n\n每个阶段通过 `subprocess` 调用 Claude Code CLI，传入系统提示词和上下文文件。\n\n### "假成功"现象\n\n引擎的每个阶段调用 Claude Code 后，只检查 `returncode == 0`（进程退出码）。但 Claude Code 即使因认证失败返回 401，进程退出码也是 0。引擎看到 0，就标记为"成功"，继续下一个阶段。\n\n这就是"假成功"——系统报告一切正常，实际上什么都没发生。\n\n### API 密钥传递机制\n\n引擎需要将 Anthropic API 密钥传递给 Claude Code。脚本中有一段逻辑从 `/home/claude-user/.claude/settings.json` 读取密钥，然后通过环境变量 `ANTHROPIC_API_KEY` 传给子进程。\n\n问题出在具体的代码实现上。\n\n## 实战指南：从发现到修复\n\n### 第一步：R2 轮次的异常信号\n\n5 月 4 日下午 2 点，R2 轮次执行。引擎报告五个阶段全部完成，git 也自动提交了。但查看提交内容：\n\n```bash\ngit log --oneline -1\n# 7605cff docs: R63 iteration — REQUIREMENTS only\n\ngit show --stat 7605cff\n# docs/REQUIREMENTS-ITER3.md  (208 lines)\n# docs/TODO.md                (updated)\n# 没有 lib/ 或 tests/ 的变更\n```\n\n只有需求文档和 TODO 更新，没有代码文件。R52（BookmarkGraph MVP E2E 测试）的任务没有被实际完成。\n\n### 第二步：定位引擎脚本的 Bug\n\n检查引擎脚本 `/root/scripts/pagewise-iteration-engine.py`，发现两个关键问题：\n\n**Bug 1：API 密钥占位符（第 106 行）**\n\n```python\n# 错误代码\nexport ANTHROPIC_API_KEY=***\n\n# 正确应该是\nexport ANTHROPIC_API_KEY={api_key}\n```\n\n脚本上方有读取 `settings.json` 的逻辑，但读取后的 `api_key` 变量没有被正确插入到环境变量设置中。`***` 是一个占位符，本应在后续修改中替换，但被遗漏了。\n\n**Bug 2：语法错误（第 125 行）**\n\n```python\n# 错误代码\n{api_key}("运行测试套件")\n\n# 正确应该是\nlog("运行测试套件")\n```\n\n`{api_key}` 被错误地插入到了 `log()` 函数调用的位置。这是一个 Python 语法错误，会导致 `NameError`。\n\n### 第三步：理解为什么错误检测失效\n\n引擎的错误检测逻辑只检查进程退出码：\n\n```python\nresult = subprocess.run(cmd, shell=True, capture_output=True)\nif result.returncode == 0:\n    log("✅ 阶段完成")\nelse:\n    log("❌ 阶段失败")\n```\n\n但 Claude Code 的 401 认证失败不会改变退出码。引擎需要更严格的验证：\n\n```python\n# 改进后的检测逻辑\nresult = subprocess.run(cmd, shell=True, capture_output=True)\n\n# 退出码不为 0 → 明确失败\nif result.returncode != 0:\n    log("❌ 阶段失败：退出码 " + str(result.returncode))\n    return False\n\n# 退出码为 0，但输出为空 → 可能是假成功\nif not result.stdout.strip() and not result.stderr.strip():\n    log("⚠️ 警告：无输出，可能是认证失败")\n    return False\n\n# 检查 stderr 中是否有认证错误\nif "401" in result.stderr or "Unauthorized" in result.stderr:\n    log("❌ 认证失败")\n    return False\n\nlog("✅ 阶段完成")\n```\n\n### 第四步：手动修复 R63\n\n由于引擎无法正常工作，R63（BookmarkLinkChecker 模块）需要手动实现。\n\n创建 `lib/bookmark-link-checker.js`：\n\n```javascript\nclass BookmarkLinkChecker {\n  constructor(options = {}) {\n    this.concurrency = options.concurrency || 5;\n    this.timeout = options.timeout || 8000;\n    this.rateLimitPerDomain = options.rateLimitPerDomain || 2; // QPS\n    this.results = [];\n    this.controller = null;\n  }\n\n  async checkAll(bookmarks) {\n    this.controller = new AbortController();\n    this.results = [];\n\n    // 按域名分组，实现速率限制\n    const domainGroups = this._groupByDomain(bookmarks);\n    const tasks = [];\n\n    for (const [domain, urls] of Object.entries(domainGroups)) {\n      for (const url of urls) {\n        tasks.push(this._rateLimitedCheck(domain, url));\n      }\n    }\n\n    // 并发执行，受 concurrency 限制\n    await this._runWithConcurrency(tasks, this.concurrency);\n    return this.results;\n  }\n\n  async checkOne(url) {\n    return this._checkUrl(url);\n  }\n\n  async _checkUrl(url) {\n    try {\n      // HEAD 请求优先，节省带宽\n      let response = await fetch(url, {\n        method: 'HEAD',\n        signal: AbortSignal.timeout(this.timeout),\n        redirect: 'follow'\n      });\n\n      // HEAD 被拒绝时回退到 GET\n      if (response.status === 405 || response.status === 501) {\n        response = await fetch(url, {\n          method: 'GET',\n          signal: AbortSignal.timeout(this.timeout),\n          redirect: 'follow'\n        });\n      }\n\n      const result = {\n        url,\n        status: response.ok ? 'alive' : (response.redirected ? 'redirect' : 'dead'),\n        statusCode: response.status,\n        redirectedTo: response.redirected ? response.url : null,\n        checkedAt: new Date().toISOString()\n      };\n\n      this.results.push(result);\n      return result;\n    } catch (error) {\n      const result = {\n        url,\n        status: 'unknown',\n        error: error.message,\n        checkedAt: new Date().toISOString()\n      };\n      this.results.push(result);\n      return result;\n    }\n  }\n\n  cancel() {\n    if (this.controller) {\n      this.controller.abort();\n    }\n  }\n\n  getDeadLinks() {\n    return this.results.filter(r => r.status === 'dead');\n  }\n\n  getRedirectLinks() {\n    return this.results.filter(r => r.status === 'redirect');\n  }\n\n  getReport() {\n    const alive = this.results.filter(r => r.status === 'alive').length;\n    const dead = this.results.filter(r => r.status === 'dead').length;\n    const redirect = this.results.filter(r => r.status === 'redirect').length;\n    const unknown = this.results.filter(r => r.status === 'unknown').length;\n    return { total: this.results.length, alive, dead, redirect, unknown };\n  }\n}\n```\n\n创建对应的测试文件 `tests/test-bookmark-link-checker-e2e.js`，覆盖：\n- 单个 URL 检查（alive / dead / redirect / timeout）\n- 批量检查的并发控制\n- 域名速率限制\n- 取消操作\n- 报告生成\n\n## 踩坑记录\n\n### 踩坑 1：grep 检查遗漏了 `***`\n\n在 R3 运行前，执行了 `grep '\\*\\*\\*' pagewise-iteration-engine.py` 来检查是否还有占位符。结果返回空——没有找到。\n\n但实际情况是第 106 行确实有 `***`。原因可能是 grep 的通配符转义问题，或者脚本在检查后被修改过。\n\n**教训**：不要只依赖 grep，直接用 `cat -n` 查看关键行号附近的内容。\n\n### 踩坑 2：退出码 ≠ 成功\n\nClaude Code CLI 即使遇到 401 认证错误，进程退出码仍然是 0。引擎只检查退出码，导致所有失败的调用都被标记为"成功"。\n\n**教训**：自动化系统的错误检测必须多层验证——退出码、输出内容、文件存在性，缺一不可。\n\n### 踩坑 3：部分修复导致更隐蔽的 Bug\n\n脚本的第 106 行之前有读取 API 密钥的代码，说明有人尝试过修复。但修复不完整——读取了密钥却没有替换掉占位符。这种"修了一半"的状态比完全没修更危险，因为它给人一种"已经修好了"的错觉。\n\n**教训**：修复 Bug 后必须端到端验证，不能只看代码逻辑"看起来对"。\n\n## 总结与展望\n\n**核心收获**：\n- 自动化系统的最大风险不是"跑不了"，而是"跑了但没跑对"\n- 错误检测必须验证实际产出，不能只看退出码\n- "部分修复"比"完全没修"更危险\n\n**最佳实践**：\n- 飞轮引擎运行后，检查 git diff 确认实际变更\n- 每个阶段的产出必须有对应的文件存在性检查\n- API 密钥等敏感配置不要硬编码，从环境变量或配置文件读取\n\n**延伸阅读**：\n- PageWise 飞轮迭代方法论：从 R1 到 R42 的完整记录\n- Claude Code 子代理工作流：四角色质量门控管道\n- 注册表模式：让自动化任务不再硬编码\n
