## 楔子

周五下午，书签模块的代码已经写了上千行，功能测试也跑通了。按理说可以松口气了，但 Chrome Web Store 的审核不会因为"功能能用"就放过你。存储结构是否合理？权限声明是否最小化？性能在万级书签下会不会崩？这些问题不解决，提交审核就是碰运气。

## 核心概念

### 2.1 存储预处理（BookmarkStorePrep）

Chrome Extension 的存储层需要特别考虑：

- **IndexedDB 限制**：Chrome 对扩展的 IndexedDB 有配额限制，需要设计合理的数据分片策略
- **存储迁移**：版本升级时的数据结构变更必须向前兼容
- **导入导出**：用户需要能备份和恢复书签数据

R83 迭代新增了 406 行代码，建立了书签存储的基础设施层，包括数据验证、格式转换和批量操作接口。

### 2.2 安全审计（BookmarkSecurityAudit）

Chrome Web Store 的安全审核重点关注：

- **权限最小化**：manifest.json 中只声明必要权限
- **CSP 合规**：Content Security Policy 不能有 `unsafe-eval`
- **XSS 防护**：用户输入必须经过转义处理
- **注入检测**：Content Script 不能被页面脚本影响

R84 迭代完成了 44 项安全测试，覆盖了权限审计、CSP 合规性检查和 XSS 防护验证。

### 2.3 性能基准（BookmarkPerformanceBenchmark）

性能是用户体验的基础：

- **读写延迟**：单次操作应在毫秒级完成
- **搜索性能**：万级书签的全文搜索应在 100ms 内响应
- **内存占用**：不能因为书签数量增长导致内存泄漏
- **批量操作**：导入导出大量书签时不能阻塞 UI

R85 迭代建立了完整的性能基准测试套件，30 项测试全部通过，质量评分 95/100。

## 实战指南

### 3.1 飞轮迭代流程

每轮迭代遵循五阶段流程：

```
需求分析 → 设计 → 实现 → 验证 → 回顾
(Plan)    (Plan)  (Sub)  (Guard) (Plan)
```

关键设计：
- **Plan Agent** 负责需求分析和设计文档
- **Sub Agent（Claude Code）** 负责代码实现
- **Guard Agent** 负责质量验证和测试
- **自动门控**：测试不通过不能进入下一阶段

### 3.2 安全审计的实战经验

R84 迭代遇到了一个典型问题：**测试与实现的 API 不匹配**。

Claude Code 生成的测试使用了 class-based API：
```javascript
// Claude Code 生成的测试
const audit = new BookmarkSecurityAudit()
audit.auditPermissions()
```

但实际实现是函数式 API：
```javascript
// 实际实现
export function auditPermissions() { ... }
export function auditCSP() { ... }
```

解决方案：手动重写测试文件，匹配实际的函数式 API。44 项测试全部通过。

**教训**：自动化生成代码时，务必确认接口风格的一致性。Class-based 和函数式 API 混用是最常见的坑之一。

### 3.3 性能基准测试的设计

R85 的性能基准测试覆盖四个维度：

| 维度 | 测试项 | 达标标准 |
|------|--------|----------|
| 读写性能 | 单条 CRUD | < 10ms |
| 搜索性能 | 全文搜索 | < 100ms |
| 内存占用 | 万级书签 | < 50MB |
| 批量操作 | 1000 条导入 | < 5s |

测试结果：30 项测试全部通过，全量回归 4238 项测试无失败。

### 3.4 质量门控实践

本次冲刺的质量门控标准：

- **测试通过率**：100%（不允许跳过任何测试）
- **全量回归**：4238 项测试全部通过
- **质量评分**：≥ 90%（R85 获得 95 分）
- **安全合规**：无高危漏洞

## 踩坑记录

### 坑 1：Phase 2 失败但继续执行

飞轮脚本在 Phase 2（设计阶段）标记为失败，但仍然继续执行了后续阶段。

**原因**：脚本未检查中间阶段的失败状态。

**修复**：在脚本中增加阶段检查，失败时终止流程。

### 坑 2：测试计数不准确

`run_tests()` 函数使用 `tail -20` 截断输出，导致 `✓`/`✗` 计数不准确。

**原因**：测试输出格式与解析逻辑不匹配。

**修复**：改用更可靠的测试结果解析方式。

### 坑 3：脚本 timeout 导致中断

迭代脚本在执行过程中遇到 timeout，需要手动恢复。

**解决方案**：Hermes 直接执行飞轮生命周期，绕过脚本的 timeout 限制。

## 总结与展望

### 核心收获

1. **一天三轮迭代是可行的**：飞轮流程的自动化程度足够高，可以在一天内完成从存储到安全到性能的全维度验证
2. **安全审计不能跳过**：API 不匹配这种问题只有在实际测试中才能发现
3. **性能基准要提前建立**：等到用户反馈卡顿就晚了

### 最佳实践

- **先审计再优化**：安全问题比性能问题更致命
- **全量回归不能省**：每次改动都要跑完整测试套件
- **质量门控要硬性执行**：90 分以下不能提交
