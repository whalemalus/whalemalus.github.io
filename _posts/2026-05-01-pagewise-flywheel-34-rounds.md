TODO.md: 0 unchecked items remaining." 飞轮引擎安静地停了下来。\n\n这不是崩溃，不是超时，不是 API 报错。它只是……做完了。\n\n34 轮迭代，从知识库性能优化到语义搜索，从 API 响应缓存到服务端同步，一个 LLM Wiki 知识编译系统的全部蓝图，在 24 小时内被逐一实现。没有人工干预，没有"下一步做什么"的犹豫，只有 TODO.md 上一个接一个被打勾的任务。\n\n直到最后一个。\n\n## 引言\n\n这篇文章记录的是 PageWise 项目在 2026 年 4 月 30 日这一天发生的事情。PageWise 是一个 Chrome 扩展，让你在浏览技术网页时可以直接向 AI 提问，并把答案存入本地知识库。\n\n这一天发生了三件事：\n\n1. **v2.1.0 的 3 个关键 Bug 被修复**，测试套件从 59 个失败归零\n2. **飞轮迭代引擎完成了 34 轮迭代**，构建了完整的 LLM Wiki 系统\n3. **飞轮引擎发现自己无事可做**，因为 TODO.md 的燃料耗尽了\n\n如果你对"如何用自动化迭代引擎在一天内完成一个月的工作量"感兴趣，这篇文章会告诉你整个过程——包括那些差点翻车的时刻。\n\n## 📖 目录\n\n1. TODO.md 是燃料\n\n这是最关键的洞察：**飞轮引擎的动力来源不是代码，是任务清单**。\n\n```markdown\n# TODO.md — 飞轮的燃料\n\n- [x] 知识库性能优化        ← R1 完成\n- [x] AI 响应缓存           ← R2 完成\n- [x] 技能推荐优化          ← R3 完成\n- [x] 用户行为分析          ← R4 完成\n...\n- [x] 服务端同步            ← R34 完成\n- [ ] (空)                  ← 引擎停止\n```\n\n当最后一个 `[ ]` 变成 `[x]`，飞轮就失去了动力。这不是 Bug——这是设计。\n\n### 4. 四文档系统\n\n每轮迭代必须产出四份文档：\n\n1. **REQUIREMENTS.md** — 这轮要做什么\n2. **DESIGN.md** — 怎么做\n3. **测试代码** — 做对了没有\n4. **CHANGELOG.md** — 做了什么\n\n这四份文档让每轮迭代可追溯、可复现、可审计。\n\n---\n\n## 实战指南\n\n### 第一步：修复 v2.1.0 的 3 个关键 Bug\n\n在飞轮引擎全速运转之前，先要解决用户报告的 3 个 Bug。\n\n**Bug 1：Q&A 页面 `insertBefore` 错误**\n\n现象：每次打开问答页面都报 `Failed to execute 'insertBefore' on 'Node'`。\n\n根因：`chatArea.innerHTML = ''` 清空了 DOM，但 MessageRenderer 的 sentinel 元素也被一起删除了。\n\n```javascript\n// 修复：添加 reset() 方法\nreset() {\n    this.chatArea.innerHTML = '';\n    // 重建 sentinel 元素\n    const sentinel = document.createElement('div');\n    sentinel.id = 'chatEndSentinel';\n    this.chatArea.appendChild(sentinel);\n}\n```\n\n**Bug 2：学习路径超时**\n\n现象：点击"生成学习路径"后 30 秒报超时。\n\n根因：用 `Promise.race` 包装的 fetch 请求无法真正取消——超时后请求仍在后台运行。\n\n```javascript\n// 修复：用 AbortSignal.timeout 替代 Promise.race\nconst response = await fetch(url, {\n    signal: AbortSignal.timeout(90000),  // 90 秒，真实超时\n    // ...\n});\n```\n\n**Bug 3：Wiki 页面完全不可用**\n\n现象：Wiki 页面没有样式，按钮点击无反应。\n\n根因：WikiStore 未初始化，事件未绑定，CSS 缺失 283 行。\n\n修复：初始化 WikiStore，绑定事件监听器，补充 283 行 CSS。\n\n### 第二步：修复测试套件的 59 个失败\n\nBug 修复后跑测试，发现 59 个测试失败。逐一排查：\n\n| 模块 | 问题 | 修复 |\n|------|------|------|\n| `batch-summary.js` | `minSectionChars` 默认值 50 太大 | 改为 10 |\n| `plugin-system.js` | `^` 和 `~` 版本匹配逻辑错误 | 重写 `satisfiesVersion` |\n| `git-repo.js` | 模块不存在 | 新建完整模块 |\n| `test-embedding.js` | API 改名 `preprocess` → `tokenize` | 对齐新 API |\n| `test-shortcuts.js` | 导出不匹配 | 对齐实际模块 |\n| 性能测试 | 阈值过严（250ms） | 放宽到 500ms |\n\n最终结果：**1873/1873 测试全部通过**。\n\n```bash\n$ node --test tests/*.js\n✓ 361 suites, 1873 tests, 0 failures\n```\n\n### 第三步：飞轮引擎全速运转\n\n测试绿灯后，飞轮引擎开始自动执行。以下是 34 轮迭代的完整记录：\n\n| 轮次 | 任务 | 产出 |\n|------|------|------|\n| R1 | 知识库性能优化 | 索引优化，查询缓存 |\n| R2 | AI 响应缓存 | 本地缓存层 |\n| R3 | 技能推荐优化 | 推荐算法改进 |\n| R4 | 用户行为分析 | 行为追踪模块 |\n| R5 | 页面内容提取增强 | 新增 4 个提取方法 |\n| R6 | 对话存储优化 | 会话持久化 |\n| R7 | 语义搜索（Embedding） | embedding-engine.js |\n| R8-R34 | LLM Wiki 知识编译系统 | 完整三级架构 |\n| R35 | (跳过) | TODO.md 无剩余任务 |\n\n**R7 的意外收获**：在实现语义搜索时，Claude Code 自动为 `page-sense.js` 添加了 4 个新的内容提取方法（`extractContent`、`extractImages`、`extractMetadata`、`extractHeadings`）。这是飞轮效应的典型体现——做一件事的过程中，顺带完成了另一件。\n\n### 第四步：飞轮停止\n\n当 R34 完成服务端同步后，引擎检查 TODO.md：\n\n```\nChecked items: 71\nUnchecked items: 0\nStatus: ALL_TASKS_COMPLETE\n```\n\n飞轮安静地停了下来。34 轮迭代，一天之内，整个 LLM Wiki 系统从蓝图变成了代码。\n\n---\n\n## 踩坑记录\n\n### 坑 1：API Key 占位符未替换\n\n**现象**：R30-R34 的 Phase 1（需求）和 Phase 2（设计）全部失败，Claude Code 返回 401 Unauthorized。\n\n**排查过程**：\n```bash\n# 检查迭代引擎脚本\ngrep -n "API_KEY" /root/scripts/pagewise-iteration-engine.py\n# 第 103 行：apiKey = "***"  ← 占位符！\n```\n\n**根因**：迭代引擎脚本第 103 行的 API Key 是 `***` 占位符，没有替换为真实密钥。Claude Code 每次启动都因认证失败而退出。\n\n**教训**：自动化脚本中的密钥管理必须有验证步骤。建议在引擎启动时检查 API Key 是否为占位符，提前报错而不是静默失败。\n\n### 坑 2：文件权限导致 Claude Code "变慢"\n\n**现象**：Claude Code 每次调用都用满 12 轮，看起来"极慢"。\n\n**根因**：`/home/claude-user/pagewise/tests/` 目录属于 `root`，而 Claude Code 以 `claude-user` 身份运行。每一轮都在报 `Permission denied`，12 轮全部浪费。\n\n**修复**：\n```bash\nchown -R claude-user:claude-user /home/claude-user/pagewise/\n```\n\n**教训**：在委托 Claude Code 之前，必须检查文件权限。建议在飞轮引擎的 Guard Agent 中加入权限检查步骤。\n\n### 坑 3：大文件消耗 Max Turns\n\n**现象**：Claude Code 处理 `sidebar.js`（6500 行）和 `sidebar.css`（4800 行）时，30 轮全部用于读文件，没有产出任何代码。\n\n**根因**：Claude Code 的 `--bare` 模式下，每读一个大文件就消耗 1-2 轮。两个文件加起来就用完了所有轮次。\n\n**应对**：\n- 使用 `--effort low` 减少不必要的文件读取\n- 将相关代码片段直接嵌入 prompt，避免 Claude Code 自己去读\n- 将大文件拆分为更小的模块\n\n### 坑 4：`git diff` 为空的"成功"迭代\n\n**现象**：R30、R32 的 Phase 3（实现）报告"成功"，但 `git diff` 为空——没有实际代码变更。\n\n**根因**：Claude Code 因 API Key 问题无法启动，但脚本没有检查子进程是否真正产出了代码变更就标记为"成功"。\n\n**教训**：迭代引擎必须验证产出——至少检查 `git diff --stat` 是否非空。\n\n---\n\n## 总结与展望\n\n### 核心收获\n\n1. **飞轮需要燃料**：TODO.md 不只是任务清单，它是整个自动化系统的动力源。没有任务，飞轮就停转。\n\n2. **自动化不等于无人值守**：API Key 占位符、文件权限、大文件处理——这些"小问题"在自动化放大镜下变成了系统性故障。\n\n3. **测试是飞轮的轴承**：1873 个测试不是装饰，它们是让飞轮平稳转动的轴承。每一轮迭代都依赖测试来验证产出。\n\n4. **Claude Code 是优秀的 Sub Agent**：在权限正确、prompt 精确的条件下，Claude Code 能在一轮迭代中完成令人惊讶的工作量——比如 R5 顺带添加了 4 个新的内容提取方法。\n\n### 最佳实践\n\n- **Guard Agent 必须检查**：文件权限、API Key、磁盘空间、测试状态\n- **验证每轮产出**：`git diff --stat` 非空才能标记成功\n- **prompt 中嵌入代码片段**：避免 Claude Code 浪费轮次读大文件\n- **TODO.md 用完就停**：不要为了"保持运转"而添加低价值任务\n- **飞轮节奏要适度**：3 次/天是合理频率，太快会撞上 API 限流\n\n### 延伸阅读\n\n- Chrome Web Store 发布准备——项目已功能完备，下一步是上架\n- 性能基准测试——为 1873 个测试建立性能基线\n- 用户反馈系统——从真实用户获取新燃料（新任务）\n- Phase 6 路线图——国际化、协作功能、离线支持\n\n飞轮停了，不是因为坏了，而是因为做完了。这大概是自动化系统最好的结局。\n
