让出主线程，让浏览器有机会处理渲染和用户事件。\n\n```javascript\nasync buildGraphBatched(bookmarks, onProgress) {\n  const batchSize = 100\n  for (let i = 0; i < bookmarks.length; i += batchSize) {\n    const batch = bookmarks.slice(i, i + batchSize)\n    this._processGraphBatch(batch)\n    \n    // 让出主线程，浏览器可以处理渲染\n    await new Promise(resolve => setTimeout(resolve, 0))\n    \n    if (onProgress) {\n      onProgress({ phase: 'graph', current: i + batch.length, total: bookmarks.length })\n    }\n  }\n}\n```\n\n**类比**：想象你在搬 8000 箱货物。一次性搬完会累死（阻塞），分批搬、每搬 100 箱休息一下（yield），虽然总时间差不多，但中间可以处理其他事情（UI 响应）。\n\n### 2.2 LRU 缓存：最近最少使用淘汰\n\n缓存是空间换时间的经典策略。但缓存不能无限增长，需要一个淘汰机制。\n\n**LRU (Least Recently Used)** 策略：当缓存满时，淘汰最久没被访问的条目。\n\n```javascript\ntrimCache(cache, maxSize) {\n  if (cache.size <= maxSize) return cache\n  \n  // Map 的插入序 = 迭代序\n  // 最先插入的 = 最久没更新的 = 应该被淘汰的\n  const keysToDelete = [...cache.keys()].slice(0, cache.size - maxSize)\n  for (const key of keysToDelete) {\n    cache.delete(key)\n  }\n  return cache\n}\n```\n\n**关键洞察**：ES6 的 `Map` 保持插入顺序，这意味着遍历 `Map.keys()` 时，最先遍历到的就是最早插入的。利用这个特性，我们可以用简洁的代码实现 LRU 淘汰。\n\n### 2.3 视口裁剪：只渲染用户看到的\n\n书签图谱可能有 8000 个节点，但用户屏幕一次只能看到 50-100 个。渲染全部节点是巨大的浪费。\n\n```javascript\ngetVisibleNodes(nodes, viewport, padding = 50) {\n  return nodes.filter(node => {\n    return (\n      node.x >= viewport.x - padding &&\n      node.x <= viewport.x + viewport.width + padding &&\n      node.y >= viewport.y - padding &&\n      node.y <= viewport.y + viewport.height + padding\n    )\n  })\n}\n```\n\n`padding` 参数预加载视口边缘外的节点，避免滚动时出现空白闪烁。\n\n### 2.4 Worker 卸载：真正的多线程\n\nWeb Worker 运行在独立线程，不阻塞主线程。适合 CPU 密集型计算（如大规模相似度计算、TF-IDF 向量化）。\n\n```javascript\ncreateWorker() {\n  // Worker 不可用时优雅降级到主线程\n  if (typeof Worker === 'undefined') return null\n  try {\n    return new Worker('worker.js')\n  } catch {\n    return null\n  }\n}\n\nasync runInWorker(operation, data) {\n  const worker = this.createWorker()\n  if (!worker) {\n    // 降级：主线程执行\n    return this._executeOnMainThread(operation, data)\n  }\n  return new Promise((resolve, reject) => {\n    worker.postMessage({ operation, data })\n    worker.onmessage = (e) => resolve(e.data)\n    worker.onerror = (e) => reject(e)\n  })\n}\n```\n\n**设计要点**：Worker 创建失败时（比如在某些受限环境）自动降级到主线程，保证功能可用性。\n\n---\n\n## 3. 实战指南\n\n### 3.1 性能基准测试\n\n在优化前，先建立基准：\n\n```javascript\nconst stats = optimizer.getPerformanceStats()\nconsole.log(`图谱构建: ${stats.buildTime}ms`)\nconsole.log(`缓存命中: ${stats.cacheHits}/${stats.cacheTotal}`)\nconsole.log(`批次数: ${stats.batchCount}`)\n```\n\n### 3.2 批处理的批次大小选择\n\n| 批次大小 | 适用场景 | 单批耗时 |\n|----------|----------|----------|\n| 50 | 高频交互（拖拽、缩放） | ~2ms |\n| 100 | 一般操作（搜索、过滤） | ~5ms |\n| 200 | 初始化加载 | ~10ms |\n\n经验法则：**单批耗时不超过 16ms**（60fps 的一帧时间），用户就不会感知到卡顿。\n\n### 3.3 视口裁剪与 Canvas 结合\n\n在 Canvas 渲染的图谱中，视口裁剪效果最显著：\n\n```javascript\n// 渲染循环\nfunction render() {\n  const visibleNodes = optimizer.getVisibleNodes(allNodes, camera.viewport)\n  // 只绘制可见节点，8000 → ~80\n  ctx.clearRect(0, 0, canvas.width, canvas.height)\n  for (const node of visibleNodes) {\n    drawNode(ctx, node)\n  }\n  requestAnimationFrame(render)\n}\n```\n\n从渲染 8000 个节点降到 ~80 个，帧率从卡顿回到 60fps。\n\n### 3.4 测试策略\n\n20 个测试用例覆盖：\n\n- **批处理**：空数组、单批、多批、进度回调、取消\n- **LRU 缓存**：未满不淘汰、满时淘汰最旧、更新刷新顺序\n- **视口裁剪**：空视口、全包含、部分包含、padding 扩展\n- **Worker**：创建成功/失败、降级执行、取消\n- **性能统计**：零值、多次操作累积\n\n---\n\n## 4. 踩坑记录\n\n### 4.1 setTimeout(0) 不是真正的 0ms\n\n`setTimeout(0)` 实际延迟约 4ms（浏览器最小延迟）。在高频场景下，累积延迟不可忽略。\n\n**解决**：对于需要精确控制的场景，考虑 `requestIdleCallback` 或 `requestAnimationFrame`。\n\n### 4.2 Map 的迭代序在删除后不变\n\n删除 Map 中间的条目不会改变其他条目的迭代顺序。这是 LRU 缓存正确性的基础，但容易被误以为「删除后重新排序」。\n\n### 4.3 Worker 的序列化开销\n\n向 Worker 传递大数据时，`postMessage` 会进行结构化克隆（structured clone），这个过程可能很慢。\n\n**解决**：使用 `Transferable` 对象（如 `ArrayBuffer`）实现零拷贝传输，但传输后原对象不可用。\n\n---\n\n## 5. 总结与展望\n\n### 核心收获\n\n1. **批处理是万级数据的基本功**：不改变算法复杂度，但让 UI 保持响应\n2. **缓存要有上限**：无限增长的缓存比没有缓存更危险\n3. **视口裁剪是最高效的优化**：从 O(n) 渲染降到 O(可见区域)\n4. **降级设计很重要**：Worker 不可用时要能回到主线程\n\n### 测试数据\n\n- 全量回归：3112 个测试用例，0 失败\n- 新增模块：20
