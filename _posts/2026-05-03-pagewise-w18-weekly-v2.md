核心概念\n\n### 飞轮迭代（Flywheel Iteration）\n\n飞轮迭代的核心思想：**每一轮做一小步，但每一步都让下一步更容易**。\n\n```\n第1轮: 写功能 → 测试不完善 → 敢改但容易踩坑\n第10轮: 功能多了 → 测试覆盖提升 → 改动更安全\n第30轮: 测试覆盖广 → 重构有信心 → 代码质量提升\n第42轮: 2111 测试 → 任何改动都能验证 → 持续交付\n```\n\n飞轮一旦转起来，惯性会帮你。关键是要**持续推动**——每天至少一轮迭代，即使只写一个测试。\n\n### LLM Wiki 三层架构\n\nLLM Wiki 是 PageWise 的知识编译系统，把零散的 Q&A 整理成结构化的 Wiki：\n\n```\nL1: 基础层 — 导出 → 实体提取 → 交叉引用 → Git\n         ↓\nL2: 编译层 — 分类 → 关联 → 矛盾检测 → 增量编译\n         ↓\nL3: 系统层 — 浏览 → 图谱 → 查询 → 同步\n```\n\n每一层解决一个不同层次的问题：L1 解决"怎么存"，L2 解决"怎么理解"，L3 解决"怎么用"。\n\n### E2E 测试策略\n\nE2E（End-to-End）测试不测单元函数，而是测**完整流程**：\n\n| 模块 | 测试重点 | 用例数 |\n|------|----------|--------|\n| AI Client | API 调用、流式解析、重试、超时 | 多场景 |\n| AI Cache | 缓存命中/未命中、LRU 淘汰 | 多场景 |\n| Knowledge Base | 增删改查、全文搜索、标签过滤 | 51 |\n| Conversation Store | 创建/恢复/分支/合并 | 多场景 |\n| Page Sense | 页面内容提取、动态页面 | 多场景 |\n| PDF Extractor | 各类 PDF 解析、大文件、错误处理 | 多场景 |\n| Skill Engine | 技能全链路 | 23 |\n\n---\n\n## 实战指南\n\n### 如何用飞轮迭代推进项目\n\n**Step 1: 确定最小可交付单元**\n\n不要试图一次做完一个大功能。把功能拆成可以独立测试和验证的小块：\n\n```\n❌ "实现 LLM Wiki 系统"     → 太大，不知道从哪开始\n✅ "实现知识库导出为 Wiki 格式" → 小到可以一轮做完\n```\n\n**Step 2: 每轮必须有测试**\n\n```\n迭代 = 写代码 + 写测试 + 验证通过\n```\n\n没有测试的迭代不算完成。测试是飞轮的润滑剂——有了测试，下一轮改动才敢放手。\n\n**Step 3: 记录每轮的发现**\n\n每轮迭代结束写一份报告：做了什么、遇到什么坑、下一轮做什么。这是飞轮的"记忆"，防止重复踩坑。\n\n### E2E 测试编写模式\n\n以 Knowledge Base E2E 为例（51 个用例）：\n\n```javascript\n// 使用 node:test 框架\nimport { describe, it, beforeEach } from 'node:test'\nimport assert from 'node:assert/strict'\n\ndescribe('Knowledge Base E2E', () => {\n  beforeEach(() => {\n    // 每个测试前重置 IndexedDB mock\n    setupIndexedDBMock()\n  })\n\n  it('should add and retrieve a knowledge entry', async () => {\n    const kb = new KnowledgeBase()\n    await kb.add({ title: '测试条目', content: '内容' })\n    const result = await kb.search('测试')\n    assert.equal(result.length, 1)\n  })\n\n  it('should filter by tags', async () => {\n    const kb = new KnowledgeBase()\n    await kb.add({ title: 'A', tags: ['docker'] })\n    await kb.add({ title: 'B', tags: ['nginx'] })\n    const result = await kb.filterByTag('docker')\n    assert.equal(result.length, 1)\n  })\n})\n```\n\n关键点：\n- 用 `node:test` 内置框架，不引入额外依赖\n- IndexedDB 用 mock，不依赖浏览器环境\n- 每个测试独立，不依赖执行顺序\n\n---\n\n## 踩坑记录\n\n### 坑 1: 测试数量增长后变慢\n\n**现象**：测试从 1000 增长到 2000+ 后，运行时间从 10 秒变成了 30 秒。\n\n**原因**：IndexedDB mock 在每个 `beforeEach` 中重新初始化，大量测试累积后成为瓶颈。\n\n**解决**：优化 mock 初始化逻辑，用共享实例替代每次都重建。运行时间回到 15 秒。\n\n### 坑 2: LLM Wiki 增量编译遗漏\n\n**现象**：新增 Q&A 后，Wiki 页面没有更新。\n\n**原因**：增量编译只检查了"新增"，没有检查"关联变更"——一个 Q&A 可能影响多个 Wiki 页面。\n\n**解决**：增量编译时不仅标记新增 Q&A，还反向查找所有引用了相关实体的 Wiki 页面，标记为"需更新"。\n\n### 坑 3: 统一错误处理的兼容性\n\n**现象**：引入 `ErrorHandler` 后，部分旧代码的 try-catch 失效。\n\n**原因**：`ErrorHandler` 的错误格式和旧代码期望的格式不一致。\n\n**解决**：在 `ErrorHandler` 中增加 `toLegacyFormat()` 方法，保持向后兼容。\n\n---\n\n## 总结与展望\n\n### W18 核心收获\n\n1. **飞轮真的能转起来** — 42 轮迭代不是靠加班，是靠每轮小步快跑\n2. **测试是信心的来源** — 2111 个测试让每次重构都有底气\n3. **三层架构降低复杂度** — LLM Wiki 的
