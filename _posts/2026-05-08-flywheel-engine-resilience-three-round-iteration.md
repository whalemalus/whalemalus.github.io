# 飞轮引擎的韧性：一次连续三轮迭代的实战复盘\n\n> **摘要**：在一天之内，PageWise 的飞轮迭代引擎连续跑完三轮迭代——R70 暗色主题（93.6 分）、R71 快捷键系统（94.0 分）、R72 E2E 测试（超时后恢复）。本文记录了引擎超时的排查过程、Guard Agent 手动接管的恢复策略，以及从这些实战中提炼出的工程经验。\n>\n> **关键词**：`飞轮迭代` `自动化引擎` `超时恢复` `AI Agent`\n\n---\n\n## 楔子\n\n凌晨两点，引擎日志里跳出一行红色的 `subprocess.TimeoutExpired`。三轮迭代中的最后一轮在 Phase 4 卡住了——Guard Agent 调用 Claude Code 做质量审查时，600 秒的超时限制被撞了个正着。\n\n代码已经提交，测试已经通过，但验证报告还没写。如果在传统 CI/CD 流水线里，这意味着整条流水线失败，需要人工介入重跑。但飞轮引擎的架构设计里，有一个叫做「Guard Agent 手动接管」的安全网。\n\n## 引言\n\nPageWise 是一个 Chrome 浏览器扩展，帮助用户在浏览技术网页时即时向 AI 提问。它的开发采用了「飞轮迭代」方法论——一套自动化的迭代引擎，把需求分析、设计、编码、验证、回顾五个阶段串成一个闭环。\n\n5 月 7 日这一天，引擎连续执行了三轮迭代，产出了两个完整功能模块和一套 E2E 测试。过程中遇到了两次引擎超时，但都被自动恢复机制化解。这个过程值得记录下来，作为 AI Agent 自动化开发的一个实战样本。\n\n---\n\n## 📖 目录\n\n1. [全景地图](#1-全景地图)\n2. [三轮迭代实录](#2-三轮迭代实录)\n3. [引擎超时与恢复机制](#3-引擎超时与恢复机制)\n4. [踩坑记录](#4-踩坑记录)\n5. [总结与展望](#5-总结与展望)\n\n---\n\n## 1. 全景地图\n\n飞轮引擎的执行架构分为三层：\n\n```\n┌─────────────────────────────────────────────┐\n│              Plan Agent (Hermes)             │\n│  需求分析 → 设计 → 任务拆解 → Prompt 编写    │\n└──────────────────┬──────────────────────────┘\n                   │ 委派任务\n                   ▼\n┌─────────────────────────────────────────────┐\n│           Sub Agent (Claude Code)            │\n│  编码 → 测试 → 文档更新 → Git Commit         │\n└──────────────────┬──────────────────────────┘\n                   │ 交付结果\n                   ▼\n┌─────────────────────────────────────────────┐\n│              Guard Agent (Hermes)            │\n│  测试运行 → 代码审查 → 0-100 评分 → 决策     │\n└─────────────────────────────────────────────┘\n```\n\n每轮迭代按五个 Phase 推进：\n\n```\nPhase 1 (需求) → Phase 2 (设计) → Phase 3 (编码) → Phase 4 (验证) → Phase 5 (回顾)\n     ~100s           ~70s           ~280s           ~120s            ~60s\n```\n\n**关键设计**：Phase 3 是唯一真正需要 Claude Code 的阶段。Phase 1/2/4/5 都可以由 Guard Agent 或 Plan Agent 直接完成。这个设计在超时恢复时发挥了关键作用。\n\n## 2. 三轮迭代实录\n\n### R70: 暗色主题 BookmarkDarkTheme\n\n**任务**：为书签图谱系统添加暗色主题支持。\n\n**新增模块** `lib/bookmark-dark-theme.js`（273 行）：\n- 三模式切换：`light` / `dark` / `system`\n- `getTheme()` 解析 system 模式，通过 `matchMedia` 检测系统偏好\n- `getColors()` / `getGraphColors()` / `getPanelColors()` 分层色板\n- `getGroupColors()` 15 色分组方案（明暗各一）\n- `getCSSVariables()` 18 个 CSS 变量\n- 纯 ES Module，不依赖 DOM 或 Chrome API\n\n**测试**：43 个用例，全部通过。\n\n**Guard 评分**：93.6/100\n\n```\n功能完整性  30%  95  →  28.5\n代码质量    25%  93  →  23.25\n测试覆盖    25%  92  →  23.0\n文档同步    10%  95  →  9.5\n安全合规    10%  95  →  9.5\n───────────────────────────\n总计               93.75\n```\n\n**引擎问题**：Phase 5 超时（600s），但所有关键工作在 Phase 3 已完成。\n\n### R71: 快捷键 BookmarkKeyboardShortcuts\n\n**任务**：为书签系统添加可自定义的快捷键管理。\n\n**新增模块** `lib/bookmark-keyboard-shortcuts.js`（385 行）：\n- 5 个默认快捷键：搜索 (Ctrl+F)、放大 (=)、缩小 (-)、重置 (0)、刷新 (F5)\n- `on(action, cb)` / `off(action, cb)` / `dispatch(action)` 回调驱动架构\n- `getBindings()` / `setBinding()` / `resetBindings()` 自定义绑定（chrome.storage.sync 持久化）\n- `detectConflict()` 快捷键冲突检测\n- `formatBinding()` / `getShortcutsSummary()` 格式化显示\n\n**测试**：48 个用例，全部通过。\n\n**Guard 评分**：94.0/100\n\n```\n功能完整性  30%  95  →  28.5\n代码质量    25%  92  →  23.0\n测试覆盖    25%  95  →  23.75\n文档同步    10%  95  →  9.5\n安全合规    10%  92  →  9.2\n───────────────────────────\n总计               94.0\n```\n\n**引擎问题**：Phase 4 超时（Guard Agent 调用 Claude Code 做审查时 600s 限制被撞）。Guard Agent 手动接管，直接读代码 + 评分。\n\n### R72: BookmarkGraph V2.0 E2E 测试\n\n**任务**：为 9 个 V2.0 模块编写端到端集成测试。\n\n**引擎状态**：Phase 3 超时（540s），Claude Code 只创建了一个 0 字节的空文件。\n\n**Guard Agent 恢复**：\n1. 验证 REQUIREMENTS 文档——发现 3 个模块名不匹配\n2. 用 `grep` 提取所有 9 个模块的 API 签名（Pre-Reading Pattern）\n3. 直接编写 E2E 测试文件（Escape Hatch 策略）\n4. 运行测试、修复失败、提交\n\n**关键发现**：Claude Code 在 REQUIREMENTS 文档中写的模块名与实际文件名不一致。例如 `bookmark-content-preview.js` 实际是 `bookmark-preview.js`。如果不做恢复而直接重跑引擎，这些错误会再次出现。\n\n## 3. 引擎超时与恢复机制\n\n### 超时分布\n\n| 轮次 | 超时阶段 | 超时原因 | 耗时 |\n|------|----------|----------|------|\n| R70 | Phase 5 | `run_tests()` 在回顾阶段阻塞 | 600s+ |\n| R71 | Phase 4 | Guard Agent 调用 Claude Code 审查 | 520s+ |\n| R72 | Phase 3 | Claude Code 实现阶段卡住 | 540s |\n\n### 恢复策略\n\n飞轮引擎的恢复策略基于一个核心观察：**Phase 3 是唯一真正需要 Claude Code 的阶段**。\n\n**Phase 5 超时（R70）**：影响最小。Phase 3 已经自动更新了 TODO.md、CHANGELOG.md、IMPLEMENTATION.md。Guard Agent 只需要独立运行测试、写 VERIFICATION.md。\n\n**Phase 4 超时（R71）**：影响中等。Guard Agent 跳过 Claude Code 调用，直接读代码 + git diff + 评分。这个经验直接推动了一个架构改进：**Phase 4 应该是 Hermes 侧操作，不需要调用 Claude Code**。\n\n**Phase 3 超时（R72）**：影响最大。但 Guard Agent 通过 Escape Hatch 机制——当 Claude Code 不可用时，Guard Agent 直接写代码——成功恢复。关键步骤是先验证 REQUIREMENTS 文档的准确性（发现 3 个模块名错误），再用 Pre-Reading Pattern 收集 API 签名。\n\n### 引擎改进方向\n\n从这三轮迭代中提炼出三个改进点：\n\n1. **Phase 4 去除 Claude Code 调用**：Guard Agent 完全有能力独立完成验证（运行测试、读 diff、评分、写报告），节省 100-300 秒\n2. **Phase 3 超时阈值调整**：当前 540s 对 Complex 任务不够，建议区分 Simple/Medium/Complex 设置不同阈值\n3. **空文件检测**：引擎超时后应检查是否有 0 字节的产物文件，这类「幽灵文件」需要被清理\n\n## 4. 踩坑记录\n\n### 坑 1：模块名不匹配\n\nClaude Code 在 REQUIREMENTS 文档中写的模块名与实际文件名不一致。\n\n```\n文档写的：bookmark-content-preview.js\n实际文件：bookmark-preview.js\n```\n\n**原因**：Claude Code 根据功能描述推断文件名，没有实际检查 `ls lib/*.js`。\n\n**解决**：Guard Agent 在恢复时先用 `ls lib/*.js` 验证模块名，发现 3 个不匹配并修正。\n\n### 坑 2：旧版文档未被覆盖\n\nPhase 1/2 生成的 REQUIREMENTS-ITER2.md 和 DESIGN-ITER2.md 来自之前的迭代，没有被新内容覆盖。\n\n**原因**：Claude Code 的 `write_file` 可能在某些情况下没有正确覆盖旧文件（可能是权限或路径问题）。\n\n**解决**：Guard Agent 检查文件时间戳，发现是旧版后手动更新。\n\n### 坑 3：Phase 3 的 `--effort low` 意外好用\n\nR70 使用了 `--bare --effort low` 参数调用 Claude Code，本以为会降低质量，结果 Phase 3 不仅完成了代码，还自动更新了所有文档（之前一直认为 `--effort low` 会跳过文档更新）。\n\n**可能原因**：Simple 任务（1 个文件）让 Claude Code 有剩余的 turn 来处理文档；prompt 明确列出了 8 个要求包括文档更新。\n\n## 5. 总结与展望\n\n### 核心收获\n\n1. **Guard Agent 是安全网**：当引擎超时时，Guard Agent 的手动接管能力是整个系统可靠性的基石。Escape Hatch 机制（Guard Agent 直接写代码）在 Claude Code 不可用时挽救了整轮迭代。\n\n2. **Phase 3 是瓶颈**：只有编码阶段真正需要 Claude Code。其他四个阶段都可以由 Hermes Agent 独立完成。这意味着引擎的超时风险集中在 Phase 3，优化应聚焦于此。\n\n3. **文档验证不可省略**：Claude Code 生成的 REQUIREMENTS 文档可能包含错误的模块名。Guard Agent 在恢复时做的模块名验证，避免了「带着错误需求重跑引擎」的恶性循环。\n\n### 最佳实践\n\n- **超时后不要盲目重跑引擎**：先检查哪些产物已经完成，只补做缺失的部分\n- **Guard Agent 应主动接管**：Phase 4 超时时，Guard Agent 直接读代码评分，比等引擎重跑更快更可靠\n- **Pre-Reading Pattern 收集 API 签名**：Guard Agent 直接写代码时，先用 `grep` 收集所有相关模块的 API 签名，确保测试覆盖完整\n\n### 下一步\n\n- R72 的 E2E 测试已恢复并提交\n- 引擎脚本将更新：Phase 4 去除 Claude Code 调用、Phase 3 超时阈值区分任务复杂度\n- 继续推进 R73（书签-知识库联动）\n\n---\n\n*本文记录了 PageWise 项目在 2026 年 5 月 7 日的三轮飞轮迭代实战。文中所有服务器路径和配置信息已做脱敏处理。*\n
