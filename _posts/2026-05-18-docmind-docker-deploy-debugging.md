# DocMind Docker 部署调试实录：四个让人抓狂的 Bug 与解法\n\n> **摘要**：在将 DocMind 文档管理系统部署到 Docker 容器时，连续遭遇四个诡异 Bug——静态服务器不代理 API 导致白屏、源码与容器认证状态不同步、上游 LLM 网关 504 超时、SSE 流式响应无限阻塞。本文记录每个问题的排查过程和最终解法。\n>\n> **关键词**：`Docker` `Vue 3` `FastAPI` `反向代理` `LLM 网关`\n\n---\n\n## 楔子\n\n周五凌晨三点，我盯着浏览器里一片空白的 DocMind 页面，心里只有一个念头：代码明明是对的，为什么跑起来就不行？\n\n本地开发一切正常，`npm run dev` 启动后功能完好。可一旦放进 Docker 容器，所有 API 请求都像石沉大海——没有报错，没有 404，只有无尽的白屏。这不是代码的错，是部署的错。而部署的错，往往比代码的错更难找。\n\n## 引言\n\nDocMind 是一个基于 Vue 3 + FastAPI 的文档管理系统，支持文档摘要、问答、知识图谱等 AI 功能。本地开发阶段一切顺利，但当我把它打包进 Docker 容器部署到服务器后，接连踩了四个坑。\n\n这四个坑各有各的诡异之处：一个是静态文件服务器的"善意"行为，一个是容器内外代码版本不一致，一个是上游服务的间歇性故障，还有一个是流式协议的兼容性问题。它们看似独立，实则共同指向一个道理——**Docker 部署不是简单的"打包运行"，环境差异会放大每一个被忽视的细节**。\n\n---\n\n## 📖 目录\n\n1. [总结与展望](#总结与展望)\n\n---\n\n## 全景地图\n\n> 鸟瞰 DocMind Docker 部署的完整架构，理解各组件之间的关系\n\n### 架构图\n\n```\n┌─────────────────────────────────────────────────────────┐\n│                    用户浏览器                            │\n│              http://localhost:3000                       │\n└──────────────────────┬──────────────────────────────────┘\n                       │\n           ┌───────────▼───────────┐\n           │   Vite Dev Server     │\n           │   (前端开发服务器)     │\n           │   port 3000           │\n           │   /api → :7860        │\n           └───────────┬───────────┘\n                       │ proxy\n           ┌───────────▼───────────┐\n           │   FastAPI Backend     │\n           │   (Docker Container)  │\n           │   port 7860           │\n           │   docmind-docmind     │\n           └───┬───────────────┬───┘\n               │               │\n    ┌──────────▼──┐    ┌───────▼────────┐\n    │ Redis Cache  │    │  LLM Gateway   │\n    │ port 6379    │    │ vs npx serve -s  ← 前端代理层\nBug 2: auth.py 源码 vs 容器版本        ← 认证层\nBug 3: LLM Gateway 504                 ← 上游服务层\nBug 4: SSE 流式阻塞                    ← 协议层\n```\n\n### 本文的学习路径\n\n```\nBug 1 (白屏) → Bug 2 (401) → Bug 3 (504) → Bug 4 (挂起)\n     ↓              ↓             ↓              ↓\n  代理配置      容器同步       网关降级       协议兼容\n```\n\n---\n\n## 核心概念\n\n### 1. 静态服务器 vs 开发服务器\n\n`npx serve -s` 是一个纯静态文件服务器，它的工作就是把文件原样返回给浏览器。当请求 `/api/v1/status` 时，它找不到这个文件，于是返回 `index.html`（SPA 的 fallback 机制）。浏览器拿到的是 HTML，而不是 JSON。\n\nVite Dev Server 不同，它支持代理配置。在 `vite.config.ts` 里可以写：\n\n```javascript\n// vite.config.ts\nserver: {\n  proxy: {\n    '/api': {\n      target: 'http://127.0.0.1:7860',\n      changeOrigin: true,\n    },\n    '/health': {\n      target: 'http://127.0.0.1:7860',\n    }\n  }\n}\n```\n\n这样 `/api/*` 的请求会被正确转发到后端，而不是返回 HTML。\n\n**类比**：静态服务器像一个只会递文件的图书管理员，你要什么它就给什么，找不到就给你一本空白书。Vite Dev Server 像一个有分拣能力的前台，知道哪些请求该转给后端处理。\n\n### 2. Docker 容器的代码同步问题\n\n当你在本地修改了 `auth.py`，Docker 容器里的代码不会自动更新。容器有自己的文件系统，它是从镜像构建时复制进去的。\n\n```bash\n# 本地代码改了，但容器还是旧的\n# 查看容器里的 auth.py\ndocker exec docmind cat /app/src/docmind/web/api/auth.py\n\n# 对比本地的\ncat src/docmind/web/api/auth.py\n```\n\n如果不 rebuild 镜像或重新 `docker cp`，容器永远运行旧代码。\n\n### 3. SSE（Server-Sent Events）流式协议\n\nSSE 是一种服务器向客户端推送数据的协议。客户端发起请求后，服务器通过一个长连接持续发送数据块：\n\n```\nHTTP/1.1 200 OK\nContent-Type: text/event-stream\n\ndata: {"chunk": "你"}\ndata: {"chunk": "好"}\ndata: {"chunk": "世"}\ndata: {"chunk": "界"}\ndata: [DONE]\n```\n\n问题在于：不是所有 LLM 供应商都支持 SSE。如果供应商返回的是普通 JSON（不是流式），而客户端用 `httpx.iter_lines()` 去读，它会一直等待更多的行，直到超时。\n\n---\n\n## 实战指南\n\n### Bug 1：白屏——静态服务器不代理 API\n\n**现象**：浏览器打开页面一片空白，控制台没有报错。\n\n**排查过程**：\n\n```bash\n# 1. 检查后端是否正常\ncurl http://localhost:7860/health\n# → {"status": "ok"}\n\n# 2. 检查前端到后端的连通性\ncurl http://localhost:3000/api/v1/status\n# → <!DOCTYPE html>...（返回了 HTML！）\n\n# 3. 确认前端服务器类型\nps aux | grep serve\n# → npx serve -s\n```\n\n**根因**：`npx serve -s` 不支持 API 代理，所有 `/api/*` 请求都返回了 `index.html`。\n\n**修复**：\n\n```bash\n# 停止 serve，改用 Vite dev server\n# 在 Dockerfile 或启动脚本中：\nnpx vite --port 3000 --host\n```\n\n同时加固路由守卫，验证 API 响应格式：\n\n```typescript\n// frontend/src/router/index.ts\ntry {\n  const status = await getOnboardingStatus()\n  // 验证响应是布尔值，不是 HTML\n  if (typeof status.onboarded === 'boolean') {\n    onboarded.value = status.onboarded\n  } else {\n    console.error('Invalid API response:', status)\n    onboarded.value = true // 降级处理\n  }\n} catch (e) {\n  onboarded.value = true\n}\n```\n\n### Bug 2：401 Unauthorized——容器认证状态不同步\n\n**现象**：所有 API 请求返回 401，但本地代码已经禁用了认证。\n\n**排查过程**：\n\n```bash\n# 1. 本地代码确认\ngrep -A5 "def verify_api_key" src/docmind/web/api/auth.py\n# → return APIKeyInfo(...)  # 已禁用\n\n# 2. 容器内代码确认\ndocker exec docmind grep -A5 "def verify_api_key" /app/src/docmind/web/api/auth.py\n# → if not api_key: raise HTTPException(401)  # 还是旧的！\n```\n\n**根因**：本地 `auth.py` 已经改成了"永远返回默认值"（禁用认证），但 Docker 容器里还是旧版本，仍然要求 `Authorization` 头。\n\n**修复**：\n\n```bash\n# 方法1：复制文件到容器\ndocker cp src/docmind/web/api/auth.py docmind:/app/src/docmind/web/api/auth.py\ndocker restart docmind\n\n# 方法2：重新构建镜像（更彻底）\ndocker build -t docmind-docmind .\ndocker stop docmind && docker rm docmind\ndocker run -d --name docmind ... docmind-docmind\n```\n\n### Bug 3：504 Gateway Timeout——LLM 网关不稳定\n\n**现象**：知识图谱生成、文档摘要等 AI 功能间歇性失败，后端日志显示 `所有渠道均失败`。\n\n**排查过程**：\n\n```bash\n# 1. 直接测试 LLM 端点\ncurl -s -o /dev/null -w "%{http_code}" \\\n  -X POST https://<LLM服务地址>/v1/chat/completions \\\n  -H "Content-Type: application/json" \\\n  -d '{"model":"mimo-2.5-pro","messages":[{"role":"user","content":"hi"}]}'\n# → 504\n\n# 2. 检查网关配置\ndocker exec docmind cat /app/data/settings.json\n# → 只配置了一个 channel，没有 fallback\n```\n\n**根因**：LLM 网关只有一个通道，当上游返回 504 时没有备用通道可以切换。\n\n**修复**（代码层面）：\n\n```python\n# src/docmind/gateway/core.py\nasync def chat_stream(self, messages, **kwargs):\n    # 给 stream 路由也加上 fallback\n    return await self.route_with_fallback(messages, stream=True, **kwargs)\n```\n\n```typescript\n// frontend/src/api/client.ts - 改善错误信息\naxios.interceptors.response.use(\n  (response) => response,\n  (error) => {\n    // 用后端返回的具体错误替换 Axios 泛型错误\n    const detail = error.response?.data?.detail\n    if (detail) {\n      error.message = detail\n    }\n    return Promise.reject(error)\n  }\n)\n```\n\n**建议**：在 LLM 网关配置中添加备用通道，避免单点故障。\n\n### Bug 4：挂起——SSE 流式响应阻塞\n\n**现象**：点击"摘要"按钮后，页面一直转圈，永远不会返回结果。\n\n**排查过程**：\n\n```bash\n# 1. 测试后端流式端点\ncurl -N http://localhost:7860/api/v1/chat/stream \\\n  -H "Content-Type: application/json" \\\n  -d '{"file_id": "test"}'\n# → 没有 SSE 事件流，直接返回 JSON\n```\n\n**根因**：前端用 `httpx.Client.stream()` 配合 `iter_lines()` 读取 SSE 数据，但 LLM 供应商不支持流式返回，直接回了一个普通 JSON。`iter_lines()` 会一直等待更多的行，永远不会结束。\n\n**修复**：\n\n```typescript\n// frontend/src/components/tabs/SummaryTab.vue\n// 从流式改为非流式调用\nimport { summarizeFile } from '@/api/summary'  // 非流式版本\n\n// 添加超时保护\nconst controller = new AbortController()\nsetTimeout(() => controller.abort(), 120_000) // 120秒超时\n\nconst result = await summarizeFile(fileId, { signal: controller.signal })\n```\n\n流式代码保留在后端，等 LLM 供应商支持 SSE 后再启用。\n\n---\n\n## 踩坑记录\n\n### 坑 1：SPA Fallback 的双刃剑\n\nSPA（单页应用）的 fallback 机制——找不到文件就返回 `index.html`——在开发时很方便，但在部署时会掩盖真正的 404 错误。API 请求返回 HTML 而不是 JSON，Axios 不会抛异常（因为 HTTP 状态码是 200），导致错误被静默吞掉。\n\n**教训**：永远在 API 响应拦截器中验证 Content-Type 或数据结构。\n\n### 坑 2：Docker 容器是"快照"\n\nDocker 容器的文件系统是在构建时固化的。`docker run` 之后，容器内的代码不会跟随宿主机变化。除非使用 volume 映射或重新构建，否则改了代码等于没改。\n\n**教训**：每次修改代码后，要么 `docker cp` + `docker restart`，要么重新构建镜像。没有第三种选择。\n\n### 坑 3：上游服务不可控\n\nLLM 网关是第三方服务，它的稳定性不在你的控制范围内。单通道配置意味着上游一挂，所有 AI 功能全废。\n\n**教训**：关键服务必须有 fallback 通道。代码层面用 `route_with_fallback()` 兜底，运维层面配置多个供应商。\n\n### 坑 4：流式 ≠ 通用\n\nSSE 是一个很好的用户体验优化，但前提是服务端真的支持。如果服务端返回普通 JSON，客户端的流式读取代码会变成一个无限等待的死循环。\n\n**教训**：流式调用必须有超时保护。同时保留非流式版本作为降级方案。\n\n---\n\n## 总结与展望\n\n### 核心收获\n\n1. **环境差异是 Docker 部署的第一杀手**——本地 `npm run dev` 和容器内 `npx serve -s` 的行为完全不同\n2. **容器代码不会自动同步**——修改本地代码后，容器里的文件不会变\n3. **上游服务必须有 fallback**——单通道配置等于单点故障\n4. **流式协议需要降级方案**——不是所有服务端都支持 SSE\n\n### 最佳实践\n\n- 前端开发服务器用 Vite（支持代理），不要用纯静态服务器\n- 认证逻辑变更后，必须同步更新 Docker 容器\n- LLM 网关至少配置两个通道\n- 流式调用必须有超时保护和非流式降级\n- API 响应拦截器要验证数据结构，不能只看 HTTP 状态码\n\n### 延伸阅读\n\n- Vite 代理配置：https://vitejs.dev/config/server-options.html#server-proxy\n- Docker 容器文件系统：https://docs.docker.com/storage/\n- SSE 规范：https://html.spec.whatwg.org/multipage/server-sent-events.html\n
