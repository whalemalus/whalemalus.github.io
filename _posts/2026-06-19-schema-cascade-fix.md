# 当修 Schema 时忘了通知消费者：一次配置演进的连锁修复

> **摘要**：PCEC 自进化引擎在 15:00 重建了服务清单的 Schema，3 小时后发现诊断脚本还在读旧格式，19 个任务全部误报。本文记录这次连锁故障的发现和修复过程，以及 Schema 演进时更新消费者的教训。
>
> **关键词**：`PCEC` `Schema 演进` `消费者同步` `自进化系统` `配置管理`

---

去年我们团队改过一次数据库字段名，上线后发现有三个下游服务还在用旧字段名，导致线上告警炸了半小时。当时我们定了一条规矩：改 schema 之前先 grep 所有消费者。

这条规矩执行了大概两个月，然后就没人记得了。

今天 PCEC 引擎自己踩了同一个坑。区别是它在 3 小时内自己发现了，自己修了。

## PCEC 是什么

PCEC（Periodic Cognitive Expansion Cycle）是我们自进化系统的核心引擎，每 3 小时运行一次。它做的事情很简单：扫描系统状态，发现问题，修复问题，记录进化。

到今天为止，它累计进化了 89 次。

它管理着一个叫 `expected-services.json` 的清单文件——记录了系统里 19 个定时任务应该处于什么状态。诊断脚本每次运行时会拿这份清单和实际状态做对比，发现不一致就告警。

## 事故经过

### 15:00 — 发现问题

PCEC 的第 5 轮扫描发现，清单文件有两套 Schema 并存：

```yaml
# 旧格式
cron_jobs:
  required: [job1, job2, ...]
  optional: [job3, ...]

# 新格式
jobs: [{id: "job1", ...}, ...]
```

诊断脚本读的是 `jobs[]`，但清单里的 19 个任务都写在 `cron_jobs.required` 里。结果：脚本认为 19 个任务全部"不在清单中"，每次扫描都报告 18 个假差异。

修复方案：把清单重建为统一的 `jobs[]` 格式，直接从 `jobs.json` 生成。

### 15:06 — 修复清单

PCEC 重建了 `expected-services.json`，统一为 `jobs[]` 格式。验证通过：19/19 个任务全部匹配。

### 18:00 — 发现连锁问题

3 小时后，PCEC 的下一轮扫描发现：诊断脚本的比较逻辑还在读旧的 `cron_jobs.required/optional` 字段。清单已经没有这些字段了，所以脚本再次报告 19 个任务全部"不在清单中"。

修清单的人（也就是 PCEC 自己）没有通知消费者。

### 18:06 — 修复消费者

PCEC 更新了诊断脚本的比较逻辑，改为自动检测 Schema 版本：优先读 `jobs[]`，fallback 到旧格式。

```bash
# 修复前（只读旧格式）
required=$(jq -r '.cron_jobs.required[]' "$manifest")

# 修复后（自动检测）
if jq -e '.jobs' "$manifest" >/dev/null 2>&1; then
    # 新格式：直接比对 jobs[] 数组
    jq -r '.jobs[].id' "$manifest"
else
    # 旧格式 fallback
    jq -r '.cron_jobs.required[]' "$manifest"
fi
```

验证通过：19/19 个任务匹配，0 个假差异。

### 21:00 — 清理收尾

最后一轮扫描确认系统 ALL GREEN，做了一些维护性清理：关闭了 1 个孤立会话，清理了 25 个过期的输出文件，精简了状态文件。

## 为什么会发生

根本原因：我们有两套 Schema（旧的 `cron_jobs.required/optional` 和新的 `jobs[]`），清单文件和诊断脚本各用一套。修清单的时候，只想着"统一格式"，没 grep 一下谁在读旧格式。

这在软件工程里太常见了。你改了一个配置文件的结构，觉得"改完了"，但忘了通知所有读这个配置的服务。

## PCEC 的自我修复链

这次事件最有意思的地方不是 Schema 本身，而是 PCEC 的处理方式：

| 时间 | 发生了什么 | PCEC 做了什么 |
|------|-----------|-------------|
| 15:00 | 清单双 Schema 并存 | 重建清单为统一格式 |
| 18:00 | 诊断脚本读旧格式 | 更新脚本的比较逻辑 |
| 21:00 | 系统恢复干净 | 执行维护性清理 |

整个过程没有人工干预。PCEC 在 15:00 修了清单，在 18:00 发现自己修出了新问题，在 21:00 确认一切恢复正常。

不是说它多聪明。它只是每 3 小时扫一次，发现问题就修，修完再扫。如果 18:00 那轮没发现，21:00 会发现。如果 21:00 还没发现，明天 00:00 会发现。

持续扫描的价值不在于每次都能发现问题，而在于问题不会永远藏着。

## Schema 演进的教训

这次连锁故障提炼出几条规则：

改配置格式之前，先 grep 谁在读它。不是"应该有人读"，是实际 grep 一下。PCEC 的诊断脚本引用了 `cron_jobs.required` 这个路径，grep 一下就能找到。

统一 Schema 后，验证的不只是"新格式能用"，还有"所有消费者都切到了新格式"。PCEC 在 15:00 验证了前者，但没有验证后者。

如果可以的话，保留旧格式的兼容读取。PCEC 在 18:00 的修复方案就是加了 fallback：优先读新格式，读不到就用旧格式。这样即使有遗漏的消费者，也不会直接报错。

## 这次进化产生的反模式

PCEC 把这次事件记录为 AP-029 `consumer_schema_desync`：修数据 Schema 时未同步更新所有消费者，是前一个修复（AP-028 `dual_representation_drift`）的连锁后果。

加上之前发现的 AP-028，这次连锁故障产出了两个反模式：

- **AP-028**：双格式 Schema 并存——清单文件同时有新旧两种格式，导致不同消费者看到不同数据
- **AP-029**：Schema 修了但消费者没跟上——修数据格式时没有通知所有读取方

## 总结

### 这次事件的教训

Schema 演进不是改一个文件的事。改之前 grep 消费者，改之后验证消费者，能加 fallback 就加 fallback。

PCEC 引擎在这次事件中展现了自进化系统的一个特点：它不怕犯错，因为它会自己发现错误并修复。15:00 的修复产生了 18:00 的新问题，但 18:00 的修复让 21:00 恢复了正常。这个循环本身就是进化。
