# 3.8GB 小内存 VPS 的四层资源防护体系\n\n> 当你的服务器只有 3.8GB 内存，却跑着 8 个 Docker 容器时，随时可能因为内存耗尽而卡死。本文分享一套实战验证的四层防护方案。\n\n## 问题背景\n\n一台 3.8GB 内存的 VPS，部署了多个服务：\n\n| 服务 | 用途 | 内存占用 |\n|------|------|----------|\n| DocMind | AI 文档助手 | ~60MB（空闲）→ 峰值可达 500MB+ |\n| AxonHub | API 网关 | ~50MB（空闲）→ CPU 密集时飙升 |\n| DimStack | 博客系统 | ~300MB + MySQL + Redis |\n| PinchTab | 浏览器自动化 | ~170MB |\n| Nginx Proxy Manager | 反向代理 | ~85MB |\n| Hermes Agent | AI Agent | 由系统管理 |\n| Claude Code | AI 编码助手 | 由系统管理 |\n\n**核心矛盾**：所有容器都没有内存限制，任何一个服务的内存峰值都可能吃光所有内存，导致 OOM Killer 随机杀死进程（可能是关键服务），甚至服务器直接卡死需要重启。\n\n## 解决方案：四层防护\n\n### L1: earlyoom — 优雅的 OOM 守护进程\n\n**问题**：Linux 默认的 OOM Killer 在内存完全耗尽时才触发，此时系统往往已经卡死，连 SSH 都连不上。\n\n**方案**：[earlyoom](https://github.com/rfjakob/earlyoom)（⭐ 4000+）在内存还有余量时就主动终止最耗内存的进程，避免系统完全卡死。\n\n```bash\n# 安装\napt-get install -y earlyoom\n\n# 配置保护名单（这些进程不会被杀死）\ncat > /etc/default/earlyoom << 'EOF'\n# 内存剩余 <10% 且 Swap 剩余 <10% 时触发\nEARLYOOM_ARGS="-m 10 -s 10 --avoid 'hermes|claude|node|python.*hermes' --prefer '.*'"\nEOF\n\n# 启动并设为开机自启\nsystemctl enable --now earlyoom\n```\n\n**关键参数**：\n- `-m 10`：内存剩余 <10% 时触发\n- `-s 10`：Swap 剩余 <10% 时触发\n- `--avoid`：保护名单，匹配的进程不会被优先杀死\n- `--prefer`：优先回收名单\n\n### L2: Docker 容器内存硬限制\n\n**问题**：Docker 容器默认不限制内存，可以无限制地消耗宿主机内存。\n\n**方案**：为每个容器设置 `--memory` 和 `--memory-swap` 限制。\n\n```bash\n# 为容器设置内存限制（运行时修改，无需重建）\ndocker update --memory=512m --memory-swap=768m docmind\ndocker update --memory=800m --memory-swap=1g axonhub-app\ndocker update --memory=512m --memory-swap=768m dimstack-app\ndocker update --memory=256m --memory-swap=512m dimstack-mysql\ndocker update --memory=128m --memory-swap=256m dimstack-redis\ndocker update --memory=512m --memory-swap=768m pinchtab\ndocker update --memory=256m --memory-swap=512m nginx-proxy-manager\ndocker update --memory=128m --memory-swap=256m docmind-redis\n```\n\n**内存分配策略**（总 3.8GB）：\n\n| 容器 | 内存上限 | Swap 上限 | 设计思路 |\n|------|----------|-----------|----------|\n| 系统预留 | ~800MB | — | OS + Hermes + Claude Code |\n| AxonHub | 800MB | 1GB | CPU 密集型，需要较多内存 |\n| DocMind | 512MB | 768MB | 不常用时可暂停 |\n| PinchTab | 512MB | 768MB | 测试完可暂停 |\n| DimStack App | 512MB | 768MB | 博客应用 |\n| MySQL | 256MB | 512MB | 数据库 |\n| Nginx | 256MB | 512MB | 反向代理 |\n| Redis ×2 | 128MB | 256MB | 缓存，占用小 |\n\n**关键点**：\n- `--memory` 是硬限制，超过会触发容器内 OOM Kill\n- `--memory-swap` 包含内存 + Swap 总量\n- 设置后**立即生效**，不需要重启容器\n- 容器被 OOM Kill 后，Docker 会根据重启策略自动重启\n\n### L3: 智能看门狗 — 自动暂停/恢复容器\n\n**问题**：即使设了内存限制，多个容器同时运行仍然可能逼近系统极限。而且很多容器（如 DocMind、PinchTab）并非 24 小时都需要运行。\n\n**方案**：一个 cron 定时任务，每 5 分钟检查资源使用率，自动暂停非核心容器。\n\n```bash\n#!/bin/bash\n# /opt/scripts/resource-watchdog.sh\n# 每 5 分钟由 cron 执行\n\nMEM_THRESHOLD=85    # 内存使用率阈值\nSWAP_THRESHOLD=70   # Swap 使用率阈值\nCPU_THRESHOLD=90    # CPU 使用率阈值\n\n# 获取使用率\nget_mem_usage() { free | awk '/Mem:/ {printf "%.0f", $3/$2 * 100}'; }\nget_swap_usage() { free | awk '/Swap:/ {if($2>0) printf "%.0f", $3/$2 * 100; else print 0}'; }\n\nMEM=$(get_mem_usage)\nSWAP=$(get_swap_usage)\n\n# 保护名单：这些容器绝不暂停\nis_protected() {\n    case "$1" in\n        *nginx-proxy*|*mysql*|*redis*) return 0 ;;\n        *) return 1 ;;\n    esac\n}\n\n# 需要回收时，暂停最大可回收容器\nif [ "$MEM" -ge "$MEM_THRESHOLD" ] || [ "$SWAP" -ge "$SWAP_THRESHOLD" ]; then\n    for container in $(docker stats --no-stream --format "{{.Name}}\\t{{.MemPerc}}" | \\\n        while IFS=$'\\t' read name mem; do\n            is_protected "$name" || echo "$(echo $mem | tr -d '%') $name"\n        done | sort -rn | awk '{print $2}'); do\n        docker pause "$container"\n        # 暂停一个后检查是否够了\n        [ "$(get_mem_usage)" -lt 75 ] && break\n    done\nfi\n\n# 内存充足时恢复暂停的容器\nif [ "$MEM" -lt 65 ] && [ -f /tmp/watchdog-paused-containers ]; then\n    while read container; do\n        docker unpause "$container" 2>/dev/null\n    done < /tmp/watchdog-paused-containers\n    rm -f /tmp/watchdog-paused-containers\nfi\n```\n\n设置 cron：\n```bash\nchmod +x /opt/scripts/resource-watchdog.sh\n(crontab -l; echo "*/5 * * * * /opt/scripts/resource-watchdog.sh") | crontab -\n```\n\n**自动行为**：\n- 内存 >85% → 暂停最大可回收容器（按内存排序）\n- 内存 <65% → 自动恢复暂停的容器\n- 保护名单内的容器（Nginx、MySQL、Redis）永远不会被暂停\n\n### L4: Docker 日志限制\n\n**问题**：Docker 容器日志默认不限制大小，长时间运行可能吃满磁盘。\n\n**方案**：\n\n```json\n// /etc/docker/daemon.json\n{\n  "log-driver": "json-file",\n  "log-opts": {\n    "max-size": "10m",\n    "max-file": "3"\n  }\n}\n```\n\n每个容器最多保留 3 个 10MB 的日志文件（共 30MB），超过自动轮转。\n\n```bash\n# 重启 Docker 生效（会短暂中断所有容器）\nsystemctl restart docker\n```\n\n## 整体架构\n\n```\n┌─────────────────────────────────────────────────┐\n│                   应用层                          │\n│  DocMind  AxonHub  DimStack  PinchTab  Nginx    │\n│  (512M)   (800M)   (512M)   (512M)   (256M)    │\n├─────────────────────────────────────────────────┤\n│ L2: Docker Memory Limit (硬限制)                 │\n│  每个容器有独立内存上限，超限触发容器内 OOM Kill    │\n├─────────────────────────────────────────────────┤\n│ L3: Resource Watchdog (智能看门狗, cron 5min)    │\n│  内存>85% → 暂停非核心容器                        │\n│  内存<65% → 自动恢复                              │\n├─────────────────────────────────────────────────┤\n│ L1: earlyoom (OOM 守护进程)                      │\n│  内存<10% → 杀死最耗内存进程(保护 hermes/claude)  │\n├─────────────────────────────────────────────────┤\n│ L4: Docker Log Rotation (日志轮转)               │\n│  每容器最大 30MB，防磁盘写满                       │\n├─────────────────────────────────────────────────┤\n│                   宿主机                          │\n│  3.8GB RAM + 2GB Swap                            │\n│  系统预留 ~800MB (OS + Hermes + Claude Code)      │\n└─────────────────────────────────────────────────┘\n```\n\n## 验证效果\n\n部署后，通过 `docker stats` 观察：\n\n```\nNAME                    CPU %   MEM USAGE / LIMIT    MEM %\ndocmind                 0.18%   62.89MiB / 512MiB    12.3%\naxonhub-app             52.25%  496MiB / 800MiB      62.0%\ndimstack-app            0.30%   306.8MiB / 512MiB    59.9%\npinchtab                4.50%   172.4MiB / 512MiB    33.7%\nnginx-proxy             0.14%   84.54MiB / 256MiB    33.0%\ndimstack-mysql          1.01%   56.88MiB / 256MiB    22.2%\n```\n\n内存使用率从之前的 90%+ 降到稳定在 50-60%，Swap 使用也明显下降。\n\n## 关键经验\n\n1. **Docker 容器必须设内存限制**：默认不限制是最大的隐患\n2. **earlyoom 比默认 OOM Killer 更可靠**：在系统卡死前就介入\n3. **暂停比杀死更优雅**：`docker pause` 保留容器状态，恢复后无缝继续\n4. **保护名单很重要**：Nginx、MySQL、Redis 这类基础设施不能被暂停\n5. **日志必须限制**：Docker 日志是隐形的磁盘杀手\n6. **Swap 是最后防线**：2GB Swap 配合 3.8GB RAM，给了系统足够的缓冲\n\n## 相关工具\n\n- [earlyoom](https://github.com/rfjakob/earlyoom) — Early OOM Daemon for Linux（⭐ 4000+）\n- [Docker resource constraints](https://docs.docker.com/config/containers/resource_constraints/) — Docker 官方资源限制文档\n- [cgroups v2](https://www.kernel.org/doc/html/latest/admin-guide/cgroup-v2.html) — Linux 内核资源控制\n\n---\n\n*本文基于真实生产环境实践，服务器配置：3.8GB RAM / 2GB Swap / Ubuntu 24.04。方案已在生产环境稳定运行。*\n
