# 当 AI Agent 学会「优雅地犯错」：PageWise 统一错误处理实战\n\n> 一个 Chrome 扩展如何从 31 处散乱的 try-catch 进化到统一的错误分类体系，顺便为 Chrome Web Store 上架扫清了国际化障碍。\n\n## 背景：错误处理的「历史债务」\n\nPageWise 项目迭代到第 34 轮时，一个痛点浮出水面——核心模块的错误处理各自为政：\n\n- `ai-client.js` 有 **31 处**错误处理，但没有使用统一的错误分类\n- `knowledge-base.js` 有 **50 处**错误处理，同样缺乏标准化\n- `service-worker.js`（后台 Service Worker）完全没有全局错误捕获\n\n与此同时，虽然 `manifest.json` 中引用了 `__MSG_extName__` 国际化标记，但实际的 `_locales` 目录并不存在——这意味着提交 Chrome Web Store 时，扩展名称会显示为原始 key 而非本地化文本。\n\n这两个问题被合并为迭代 #35（R35），标记为 **Complex** 级别。\n\n## 设计决策\n\n### 统一错误分类\n\n项目中已经存在一个 `error-handler.js` 模块（393 行），包含完整的错误分类体系（`ErrorType` 枚举和分类函数）。但这个模块只在 `sidebar.js` 中被使用，核心模块完全没有集成。\n\n设计决策：\n- `ai-client.js` 所有错误对象附加 `.classified` 属性，调用方可直接获取错误类型\n- `knowledge-base.js` 的 IndexedDB 操作使用 `classifyStorageError()` 统一分类\n- `service-worker.js` 安装全局错误捕获（`self.onerror` + `unhandledrejection`）\n\n### 国际化基础\n\n创建 `_locales/en/messages.json` 和 `_locales/zh_CN/messages.json`，消除 Chrome Web Store 提交障碍。\n\n## 实现过程\n\n### 第一轮：核心模块集成\n\nClaude Code 接手 `ai-client.js` 和 `knowledge-base.js` 的改造。关键改动：\n\n```javascript\n// 改造前：散乱的错误处理\ntry {\n  const response = await fetch(url, options);\n  // ...\n} catch (error) {\n  console.error('AI request failed:', error);\n  throw error;\n}\n\n// 改造后：统一分类\ntry {\n  const response = await fetch(url, options);\n  // ...\n} catch (error) {\n  const classified = classifyAIError(error);\n  error.classified = classified;\n  throw error;\n}\n```\n\n`knowledge-base.js` 的改造更为复杂——涉及 50 处 IndexedDB 操作，每一处都需要用 `classifyStorageError()` 包装。\n\n### 第二轮：全局错误捕获\n\n`service-worker.js` 作为 Chrome 扩展的后台入口，需要捕获所有未处理的异常：\n\n```javascript\n// 全局错误捕获\nself.onerror = (message, source, lineno, colno, error) => {\n  const classified = error ? classifyAIError(error) : { type: 'UNKNOWN', message };\n  console.error('[Global Error]', classified);\n};\n\nself.addEventListener('unhandledrejection', (event) => {\n  const classified = classifyAIError(event.reason);\n  console.error('[Unhandled Rejection]', classified);\n});\n```\n\n### 第三轮：国际化文件\n\n创建标准的 Chrome 扩展 locale 文件：\n\n```json\n// _locales/zh_CN/messages.json\n{\n  "extName": { "message": "智阅 PageWise" },\n  "extDescription": { "message": "浏览技术网页时即时向 AI 提问，自动整理知识库" }\n}\n```\n\n## 质量评审\n\nGuard Agent 的量化评分：\n\n| 维度 | 权重 | 得分 | 说明 |\n|------|------|------|------|\n| 功能完整性 | 30% | 85 | 核心模块集成完成，wiki-store 留给下轮 |\n| 代码质量 | 25% | 88 | 错误对象携带 `.classified` 属性，设计清晰 |\n| 测试覆盖 | 25% | 78 | 1873 测试全部通过，但缺少 error-handler 集成测试 |\n| 文档同步 | 10% | 90 | 四份文档齐全 |\n| 安全合规 | 10% | 95 | 无硬编码密钥，权限最小化 |\n\n**加权总分：88/100** — 通过（修复一个 P1 问题后从 83 提升至 88）。\n\nP1 问题：`service-worker.js` 中有一个未使用的 `import` 语句，已移除。\n\n## 经验总结\n\n### 1. 「先有后优」的错误处理策略\n\n项目初期没有统一错误处理是可以接受的——过早抽象反而增加复杂度。但当错误处理代码超过 80 处时，统一分类的收益就显现了：\n- 调试时可以直接看 `.classified.type` 而不是解析 error message\n- 用户端可以根据错误类型显示不同的提示\n- 监控时可以按错误类型聚合统计\n\n### 2.
