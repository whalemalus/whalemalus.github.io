> **摘要**：GitHub CI/CD 是什么？为什么每个现代项目都在用？本文从零开始，用 DocMind 和 PageWise 两个真实项目的经验，带你理解 CI/CD 的原理、作用、应用场景，并手把手教你从零搭建一套完整的自动化流水线。\n>\n> **关键词**：`GitHub Actions` `CI/CD` `自动化测试` `持续集成` `持续部署`\n\n---\n\n## 楔子\n\n凌晨两点，你刚改完一个 bug，信心满满地推送到 GitHub。第二天早上，同事告诉你：你的代码把整个项目的测试搞炸了。你一脸懵——"我本地跑得好好的啊？"\n\n这种场景，每个开发者都经历过。问题的根源很简单：**你的本地环境和别人的不一样**。你装了某个依赖、改了某个配置、用了一个特定版本的 Python——这些在你的电脑上没问题，但到了别人那里就炸了。\n\nGitHub CI/CD 就是来解决这个问题的。它在每次你推送代码时，自动在一个干净的环境里跑一遍所有检查：代码风格对不对？测试过不过？能不能正常打包？如果任何一步失败，它会立刻告诉你，而不是等到半夜同事打电话来。\n\n---\n\n## 引言\n\n本文将从实际项目经验出发，讲解 GitHub CI/CD 的方方面面。我们有两个真实案例：\n\n- **DocMind**（本地文件智能助手）—— Python FastAPI 后端 + Vue 3 前端，3400+ 测试用例\n- **PageWise**（Chrome 浏览器扩展）—— TypeScript + Chrome Extension MV3，5700+ 测试用例\n\n这两个项目从零搭建了 CI/CD 流水线，过程中踩了不少坑。本文会把这些经验一并分享。\n\n---\n\n## 📖 目录\n\n1.             │\n├─────────────────────────────────────────────────────────┤\n│                                                         │\n│  开发者 ──push──→ GitHub ──触发──→ GitHub Actions        │\n│                                      │                  │\n│                              ┌───────┴───────┐          │\n│                              │               │          │\n│                            CI 阶段        CD 阶段        │\n│                              │               │          │\n│                        ┌─────┴─────┐   ┌─────┴─────┐    │\n│                        │           │   │           │    │\n│                      Lint       Test  Build     Release  │\n│                    (代码风格)  (测试) (打包)    (发布)    │\n│                                                         │\n└─────────────────────────────────────────────────────────┘\n```\n\n- **CI（Continuous Integration，持续集成）**：每次推送代码，自动检查代码质量和运行测试\n- **CD（Continuous Deployment/Delivery，持续部署/交付）**：测试通过后，自动打包、发布\n\n### 为什么需要 CI/CD？\n\n| 没有 CI/CD | 有 CI/CD |\n|------------|----------|\n| 手动跑测试，经常忘记 | 每次推送自动跑 |\n| "我本地没问题" | 统一环境，结果一致 |\n| 发布靠人工，容易出错 | 自动打包发布 |\n| Bug 到生产环境才发现 | PR 阶段就拦截 |\n| 回归测试靠记忆 | 全量测试每次跑 |\n\n### GitHub Actions 的核心组件\n\n```\n┌──────────────────────────────────────────┐\n│           GitHub Actions 架构            │\n├──────────────────────────────────────────┤\n│                                          │\n│  Workflow (工作流)                        │\n│  ├── .github/workflows/ci.yml           │\n│  │                                       │\n│  ├── Trigger (触发条件)                   │\n│  │   ├── push: 推送时触发                 │\n│  │   ├── pull_request: PR 时触发          │\n│  │   └── tag: 打标签时触发                │\n│  │                                       │\n│  ├── Job (作业)                           │\n│  │   ├── lint (代码风格检查)              │\n│  │   ├── test (运行测试)                  │\n│  │   └── build (打包构建)                 │\n│  │                                       │\n│  └── Step (步骤)                          │\n│      ├── actions/checkout (拉取代码)      │\n│      ├── actions/setup-python (装环境)    │\n│      └── 运行具体命令                     │\n│                                          │\n└──────────────────────────────────────────┘\n```\n\n### 学习路径\n\n```\n理解概念 → 写第一个 workflow → 添加 lint → 添加测试 → 添加构建 → 添加自动发布\n```\n\n---\n\n## 2. 核心概念\n\n> 关键术语和原理的深度解释\n\n### 2.1 Workflow（工作流）\n\nWorkflow 是 CI/CD 的顶层单元，一个 `.yml` 文件就是一个 workflow。它定义了"什么时候触发、做什么事"。\n\n**类比**：Workflow 就像一份菜谱——写好步骤，每次有食材（代码推送）进来，就按菜谱自动做菜（检查、测试、打包）。\n\n```yaml\n# .github/workflows/ci.yml\nname: CI              # 工作流名称，显示在 GitHub Actions 页面\n\non:                   # 触发条件\n  push:               # 每次 push 都触发\n    branches: [master]\n  pull_request:       # 每次 PR 也触发\n    branches: [master]\n```\n\n### 2.2 Job（作业）\n\n一个 workflow 可以包含多个 job，默认**并行执行**。每个 job 运行在一个独立的虚拟机上。\n\n```yaml\njobs:\n  lint:               # Job 1: 代码风格检查\n    runs-on: ubuntu-latest\n    steps: [...]\n\n  test:               # Job 2: 运行测试\n    runs-on: ubuntu-latest\n    steps: [...]\n```\n\n**为什么要分 job？**\n- 并行执行，更快得到结果\n- 互不影响，一个失败不阻塞另一个\n- 独立环境，避免依赖冲突\n\n### 2.3 Step（步骤）\n\n每个 job 由多个 step 组成，按顺序执行。\n\n```yaml\nsteps:\n  - name: 拉取代码              # Step 1\n    uses: actions/checkout@v4\n\n  - name: 安装 Python           # Step 2\n    uses: actions/setup-python@v5\n    with:\n      python-version: '3.12'\n\n  - name: 安装依赖              # Step 3\n    run: pip install -r requirements.txt\n\n  - name: 运行测试              # Step 4\n    run: pytest tests/ -v\n```\n\n### 2.4 Runner（运行器）\n\nRunner 是执行 job 的虚拟机。GitHub 提供免费的云端 runner：\n\n| Runner | 系统 | 适用场景 |\n|--------|------|----------|\n| `ubuntu-latest` | Ubuntu 24.04 | 大多数项目（推荐） |\n| `windows-latest` | Windows Server | Windows 特定项目 |\n| `macos-latest` | macOS | iOS/macOS 开发 |\n\n也可以用**自托管 runner**（self-hosted），比如在自己的服务器上运行，适合需要特殊环境的场景。\n\n### 2.5 Actions（动作）\n\nActions 是可复用的组件，用 `uses` 引用。最常用的几个：\n\n```yaml\n# 拉取代码\n- uses: actions/checkout@v4\n\n# 设置 Python 环境\n- uses: actions/setup-python@v5\n  with:\n    python-version: '3.12'\n\n# 设置 Node.js 环境\n- uses: actions/setup-node@v4\n  with:\n    node-version: '20'\n```\n\n### 2.6 触发机制详解\n\n```yaml\non:\n  # 推送到 master 分支时触发\n  push:\n    branches: [master]\n\n  # PR 目标是 master 时触发\n  pull_request:\n    branches: [master]\n\n  # 推送 v 开头的标签时触发（用于发布）\n  push:\n    tags: ['v*']\n\n  # 手动触发\n  workflow_dispatch:\n\n  # 定时触发（每天 UTC 0 点）\n  schedule:\n    - cron: '0 0 * * *'\n```\n\n---\n\n## 3. 实战指南\n\n> 从零开始搭建 CI/CD 流水线\n\n### 3.1 第一步：创建 Workflow 文件\n\n在项目根目录创建 `.github/workflows/ci.yml`：\n\n```bash\nmkdir -p .github/workflows\ntouch .github/workflows/ci.yml\n```\n\n### 3.2 Python 项目的完整 CI 配置\n\n以 DocMind 项目为例（FastAPI + pytest）：\n\n```yaml\n# .github/workflows/ci.yml\nname: CI\n\non:\n  push:\n    branches: [master]\n  pull_request:\n    branches: [master]\n\njobs:\n  # ── Job 1: 代码风格检查 ──\n  lint:\n    runs-on: ubuntu-latest\n    steps:\n      - uses: actions/checkout@v4\n\n      - uses: actions/setup-python@v5\n        with:\n          python-version: '3.12'\n\n      - name: Install ruff\n        run: pip install ruff\n\n      - name: 检查代码风格\n        run: ruff check src/\n\n      - name: 检查代码格式\n        run: ruff format --check src/\n\n  # ── Job 2: 单元测试 ──\n  test-unit:\n    runs-on: ubuntu-latest\n    steps:\n      - uses: actions/checkout@v4\n\n      - uses: actions/setup-python@v5\n        with:\n          python-version: '3.12'\n\n      - name: 安装依赖\n        run: |\n          pip install -r requirements.txt\n          pip install -r requirements-test.txt\n\n      - name: 运行单元测试\n        run: pytest tests/unit/ -v --tb=short\n```\n\n**关键点解释**：\n\n1. **`ruff`** 是 Python 的代码风格检查工具，类似 ESLint 之于 JavaScript\n2. **`pytest`** 是 Python 的测试框架，`-v` 显示详细输出，`--tb=short` 简化错误信息\n3. **两个 job 并行运行**：lint 和 test 互不依赖，同时跑节省时间\n\n### 3.3 JavaScript / Chrome Extension 项目的 CI 配置\n\n以 PageWise 项目为例（TypeScript + Chrome Extension MV3）：\n\n```yaml\n# .github/workflows/ci.yml\nname: CI\n\non:\n  push:\n    branches: [main]\n  pull_request:\n    branches: [main]\n\njobs:\n  lint:\n    runs-on: ubuntu-latest\n    steps:\n      - uses: actions/checkout@v4\n\n      - uses: actions/setup-node@v4\n        with:\n          node-version: '20'\n          cache: 'npm'\n\n      - run: npm ci\n\n      - name: ESLint 代码检查\n        run: npm run lint\n\n  test:\n    runs-on: ubuntu-latest\n    steps:\n      - uses: actions/checkout@v4\n\n      - uses: actions/setup-node@v4\n        with:\n          node-version: '20'\n          cache: 'npm'\n\n      - run: npm ci\n\n      - name: 运行单元测试\n        run: npm test\n\n  package-check:\n    runs-on: ubuntu-latest\n    steps:\n      - uses: actions/checkout@v4\n\n      - uses: actions/setup-node@v4\n        with:\n          node-version: '20'\n          cache: 'npm'\n\n      - run: npm ci\n      - run: npm run build\n\n      - name: 检查包体积\n        run: |\n          SIZE=$(du -sh dist/ | cut -f1)\n          echo "包体积: $SIZE"\n          # 如果超过 5MB 则失败\n          MAX_SIZE=5000\n          ACTUAL=$(du -sk dist/ | cut -f1)\n          if [ "$ACTUAL" -gt "$MAX_SIZE" ]; then\n            echo "包体积超过限制: ${ACTUAL}KB > ${MAX_SIZE}KB"\n            exit 1\n          fi\n```\n\n### 3.4 自动发布（Release）\n\n当测试稳定后，可以添加自动发布 workflow：\n\n```yaml\n# .github/workflows/release.yml\nname: Release\n\non:\n  push:\n    tags: ['v*']  # 推送 v 开头的标签时触发\n\npermissions:\n  contents: write  # 需要写权限来创建 Release\n\njobs:\n  release:\n    runs-on: ubuntu-latest\n    steps:\n      - uses: actions/checkout@v4\n\n      - name: 打包项目\n        run: |\n          zip -r release.zip . -x ".git/*" "node_modules/*" ".github/*"\n\n      - name: 创建 GitHub Release\n        uses: softprops/action-gh-release@v2\n        with:\n          files: release.zip\n          generate_release_notes: true\n```\n\n**使用方式**：\n\n```bash\n# 本地打标签并推送\ngit tag v1.0.0\ngit push origin v1.0.0\n\n# GitHub Actions 自动触发 → 创建 Release → 上传 zip\n```\n\n### 3.5 Docker 构建验证\n\n对于有 Dockerfile 的项目，可以添加构建验证：\n\n```yaml\n# .github/workflows/docker-build.yml\nname: Docker Build\n\non:\n  pull_request:\n    branches: [master]\n\njobs:\n  docker:\n    runs-on: ubuntu-latest\n    steps:\n      - uses: actions/checkout@v4\n\n      - name: 构建 Docker 镜像\n        run: docker build -t myapp:test .\n\n      - name: 验证镜像能启动\n        run: |\n          docker run -d --name test-app -p 8000:8000 myapp:test\n          sleep 5\n          curl -f http://localhost:8000/health || exit 1\n          docker stop test-app\n```\n\n### 3.6 环境变量和密钥管理\n\nCI 中经常需要 API Key、密码等敏感信息。GitHub Secrets 解决了这个问题：\n\n**设置步骤**：\n1. 进入仓库 → Settings → Secrets and variables → Actions\n2. 点击 "New repository secret"\n3. 输入名称和值（如 `API_KEY`）\n\n**在 workflow 中使用**：\n\n```yaml\nsteps:\n  - name: 运行需要 API 的测试\n    env:\n      API_KEY: ${{ secrets.API_KEY }}\n    run: pytest tests/ -v\n```\n\n**⚠️ 重要**：永远不要把密钥硬编码在 `.yml` 文件中！\n\n### 3.7 缓存优化\n\n每次 CI 运行都要重新安装依赖，很慢。缓存可以加速：\n\n```yaml\n# Python 缓存\n- uses: actions/setup-python@v5\n  with:\n    python-version: '3.12'\n    cache: 'pip'  # 自动缓存 pip 包\n\n# Node.js 缓存\n- uses: actions/setup-node@v4\n  with:\n    node-version: '20'\n    cache: 'npm'  # 自动缓存 npm 包\n```\n\n效果：依赖安装从 2 分钟降到 10 秒。\n\n### 3.8 完整的三阶段流水线\n\n将所有内容组合成一个完整的 CI/CD 体系：\n\n```\n┌─────────────────────────────────────────────────────┐\n│                 三阶段流水线                          │\n├─────────────────────────────────────────────────────┤\n│                                                     │\n│  阶段1: CI（每次 push/PR）                           │\n│  ├── lint（代码风格）     ← 10-30 秒                │\n│  ├── test（单元测试）     ← 1-5 分钟                │\n│  └── build（构建验证）    ← 30 秒-2 分钟            │\n│                                                     │\n│  阶段2: CD-Release（打标签时）                        │\n│  ├── 打包项目                                       │\n│  ├── 创建 GitHub Release                            │\n│  └── 上传产物                                       │\n│                                                     │\n│  阶段3: CD-Deploy（可选，手动或自动）                 │\n│  ├── 构建 Docker 镜像                               │\n│  ├── 推送到镜像仓库                                  │\n│  └── 部署到服务器                                    │\n│                                                     │\n└─────────────────────────────────────────────────────┘\n```\n\n---\n\n## 4. 踩坑记录\n\n> 我们在 DocMind 和 PageWise 项目中踩过的坑\n\n### 4.1 测试本地通过但 CI 失败\n\n**现象**：本地 `pytest` 全绿，CI 里有一个测试失败。\n\n**原因**：测试依赖了模块级别的注册逻辑，但 CI 的导入顺序不同。\n\n```python\n# 某个 converter 模块在导入时注册自己\n# 本地跑时碰巧先导入了这个模块\n# CI 环境导入顺序不同，注册没发生\n\n# ❌ 错误：假设注册已发生\ndef test_converter(self):\n    converter = ConverterFactory.get_converter(".txt")  # 返回 None！\n\n# ✅ 正确：显式注册\ndef test_converter(self):\n    from docmind.converters.markitdown_converter import MarkItDownConverter\n    ConverterFactory.register(MarkItDownConverter)  # 确保注册\n    converter = ConverterFactory.get_converter(".txt")  # 正常工作\n```\n\n**教训**：不要依赖模块导入的副作用，测试中要显式设置所需状态。\n\n### 4.2 pytest -x 隐藏了多个失败\n\n**现象**：CI 显示 "1 failed"，本地跑看到 "5 failed"。\n\n**原因**：CI 命令里有 `-x` 参数，遇到第一个失败就停止。\n\n```yaml\n# ❌ 只报告第一个失败\n- run: pytest tests/ -x -v\n\n# ✅ 跑完所有测试，报告全部结果\n- run: pytest tests/ -v --tb=short\n```\n\n### 4.3 CSS 变量测试因设计风格变更失败\n\n**现象**：字体测试失败，提示 `--font-sans 未以 sans-serif 结尾`。\n\n**原因**：项目从无衬线风格改成了学术复古风格（serif），但测试还在检查 sans-serif。\n\n```python\n# ❌ 只接受 sans-serif\nassert value.endswith("sans-serif")\n\n# ✅ 接受两种风格\nassert value.endswith("sans-serif") or value.endswith("serif")\n```\n\n**教训**：测试要与设计决策保持同步。设计变了，测试也要跟着改。\n\n### 4.4 缺少依赖导致 CI 失败\n\n**现象**：本地有 `markitdown[all]`，CI 报 `ModuleNotFoundError`。\n\n**原因**：`requirements.txt` 里只写了 `markitdown`，但代码用到了 `markitdown[all]` 的功能。\n\n```yaml\n# ✅ 在 CI 中显式安装需要的依赖\n- name: 安装依赖\n  run: |\n    pip install -r requirements.txt\n    pip install "markitdown[all]"  # 显式安装，不依赖 requirements\n```\n\n### 4.5 Release 权限不足\n\n**现象**：Release workflow 报 `Resource not accessible by integration`。\n\n**原因**：默认的 `GITHUB_TOKEN` 没有写权限。\n\n```yaml\n# ✅ 添加权限声明\npermissions:\n  contents: write  # 创建 Release 需要写权限\n```\n\n### 4.6 Node.js 版本警告\n\n**现象**：CI 日志显示 `Node.js 20 actions are deprecated`。\n\n**原因**：GitHub Actions 正在从 Node.js 20 迁移到 24。\n\n**解决方案**：暂时只是警告，不影响运行。关注 `actions/checkout`、`actions/setup-python` 等官方 action 的更新。\n\n---\n\n## 5. 总结与展望\n\n### 核心收获\n\n1. **CI/CD 不是可选的，是必须的**。只要项目超过一个人、超过一天的开发周期，CI/CD 就值得投入。\n\n2. **从简单开始**。先加 lint + 测试，再慢慢加构建、发布。不要一开始就追求完美的流水线。\n\n3. **测试是核心**。CI 的价值在于自动跑测试。没有测试的 CI 就是没有灵魂的躯壳。\n\n4. **踩坑是正常的**。CI 环境和本地环境不一样，一定会遇到各种问题。关键是建立"CI 失败 → 分析原因 → 修复 → 推送"的肌肉记忆。\n\n### 最佳实践\n\n| 实践 | 说明 |\n|------|------|\n| lint 和 test 并行 | 节省时间，互不阻塞 |\n| 缓存依赖 | pip/npm 缓存可以节省 1-2 分钟 |\n| 不要用 `-x` | 让 pytest 跑完所有测试 |\n| 显式安装依赖 | 不要依赖
