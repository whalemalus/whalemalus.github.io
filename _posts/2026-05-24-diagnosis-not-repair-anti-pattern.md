# 诊断≠修复：当 AI Agent 的巡检系统陷入"永远在诊断"的死循环\n\n> **摘要**：飞轮守护者巡检系统连续 4 天检测到同一个 CI 失败问题，每次都准确诊断出根因，但每次都没能完成修复。本文复盘这个"诊断-丢失-再诊断"的反模式，提出 pending_fixes 缓存机制，让监控系统真正闭环。\n>\n> **关键词**：`飞轮守护者` `AI Agent` `自动化巡检` `CI/CD` `自愈系统`\n\n---\n\n## 楔子\n\n周三下午三点，飞轮守护者（Flywheel Guardian）准时启动巡检。\n\n扫描 PageWise 项目，发现 5 个测试失败。定位到 `test-coverage-infra.js`，确认根因是 `package.json` 里错误的 `rm -rf` 命令。记录到 Evolution Log。巡检结束。\n\n一切看起来很正常——直到你发现，周一早上九点的那次巡检，做了完全相同的事。\n\n四天。同一个问题。四次诊断。零次修复。\n\n## 引言\n\n自动化监控系统的核心价值不是"发现问题"，而是"解决问题"。一个只会诊断不会修复的巡检系统，和一个只会发告警邮件的监控有什么区别？\n\n本文记录了飞轮守护者巡检系统在实际运行中暴露的一个设计缺陷——"诊断≠修复"反模式，以及我们如何通过一个简单的缓存机制来解决它。\n\n如果你正在构建任何类型的自动化监控或自愈系统，这个教训值得参考。\n\n## 📖 目录\n\n1. [总结与展望](#总结与展望)\n\n---\n\n## 全景地图\n\n> 鸟瞰自动化巡检系统的完整架构，理解诊断与修复之间的断层\n\n### 飞轮守护者架构\n\n```\n┌─────────────────────────────────────────────────────┐\n│                  飞轮守护者系统                        │\n├─────────────────────────────────────────────────────┤\n│                                                     │\n│  ┌──────────┐    ┌──────────┐    ┌──────────┐      │\n│  │ 定时触发  │───→│ 诊断引擎  │───→│ 修复执行  │      │\n│  │ (Cron)   │    │ (Check)  │    │ (Fix)    │      │\n│  └──────────┘    └──────────┘    └──────────┘      │\n│                        │              ↑             │\n│                        │         ⚠️ 断层！          │\n│                        ↓              │             │\n│                  ┌──────────┐         │             │\n│                  │ 诊断结果  │─────────┘             │\n│                  │ (临时变量) │  ← 会话结束即丢失       │\n│                  └──────────┘                       │\n│                                                     │\n└─────────────────────────────────────────────────────┘\n```\n\n### 问题的数据流\n\n```\n巡检 #1 (周一 09:00)\n  → 检测到 CI 失败\n  → 诊断根因: package.json 的 rm -rf 问题\n  → 会话结束, 诊断结果丢失\n\n巡检 #2 (周一 15:00)\n  → 检测到 CI 失败 (同一个)\n  → 重新诊断根因 (重复工作)\n  → 会话结束, 诊断结果再次丢失\n\n巡检 #3 (周二 09:00)\n  → 同上\n\n巡检 #4 (周三 15:00)\n  → 同上\n  → 第四次诊断同一个问题\n```\n\n### 本文的学习路径\n\n问题现象 → 根因分析 → 设计缺陷 → 解决方案 → 实现细节 → 通用模式\n\n---\n\n## 核心概念\n\n### 1. 诊断≠修复反模式\n\n这个反模式的核心特征：\n\n| 特征 | 表现 |\n|------|------|\n| **诊断能力强** | 能准确找到根因，分析透彻 |\n| **修复能力弱** | 诊断结果没有传递给修复环节 |\n| **会话隔离** | 每次巡检是独立会话，不共享上下文 |\n| **重复工作** | 每次从零开始诊断，浪费计算资源 |\n| **时间浪费** | 问题持续存在，用户等待修复 |\n\n类比：就像一个医生每次看诊都准确诊断出病因，但病历本是用铅笔写的，下次来就擦掉了。病人每次来都要重新检查。\n\n### 2. 会话边界问题\n\nAI Agent 的一个根本限制：**会话是无状态的**。\n\n```\n会话 A: "发现问题 X, 根因是 Y, 修复方案是 Z"\n         ↓ (会话结束)\n会话 B: "发现问题 X, 根因是..." (从头开始)\n```\n\n这和传统的监控系统不同。Prometheus + Alertmanager 的告警会持久化到数据库，修复脚本可以从数据库读取。但 AI Agent 的"诊断结果"存在于会话的上下文窗口中，会话结束就消失了。\n\n### 3. 闭环缺失\n\n一个完整的自愈系统需要：\n\n```\n检测 → 诊断 → 计划 → 执行 → 验证 → 完成\n  ↑                                    │\n  └──────────── 如果失败 ←──────────────┘\n```\n\n飞轮守护者在"诊断"之后断了：\n\n```\n检测 → 诊断 → (丢失) → 检测 → 诊断 → (丢失) → ...\n```\n\n---\n\n## 实战指南\n\n### 方案：Pending Fixes 缓存机制\n\n核心思想：**将诊断结果持久化到文件，让修复动作可以从文件中恢复上下文**。\n\n#### 1. 定义缓存文件格式\n\n创建 `/root/.hermes/pending_fixes.md`：\n\n```markdown\n# Pending Fixes\n\n## [PageWise] CI 失败 - coverage 基础设施\n- **检测时间**: 2026-05-20 09:00\n- **问题类型**: ci_failed\n- **根因**: package.json 第16行 test:coverage 脚本包含 `rm -rf coverage/tmp`，\n  与测试断言冲突（测试期望只用 clean-coverage.js）\n- **修复方案**:\n  1. 编辑 /home/claude-user/pagewise/package.json\n  2. 移除 test:coverage 中的 `rm -rf coverage/tmp 2>/dev/null;`\n  3. 确保 CI 环境预生成 coverage-summary.json, lcov.info\n- **影响范围**: 5/7801 测试失败\n- **优先级**: 高（已持续 4 天）\n- **状态**: pending\n```\n\n#### 2. 修改巡检流程\n\n在诊断脚本中加入缓存读取逻辑：\n\n```python\nimport os\nimport yaml\nfrom datetime import datetime\n\nPENDING_FIXES_PATH = "/root/.hermes/pending_fixes.md"\n\ndef load_pending_fixes():\n    """加载待修复列表"""\n    if not os.path.exists(PENDING_FIXES_PATH):\n        return []\n    # 解析 markdown 格式的待修复项\n    fixes = []\n    with open(PENDING_FIXES_PATH) as f:\n        content = f.read()\n    # ... 解析逻辑\n    return fixes\n\ndef save_pending_fix(project, issue_type, root_cause, fix_plan):\n    """保存诊断结果到缓存"""\n    entry = {\n        "project": project,\n        "issue_type": issue_type,\n        "root_cause": root_cause,\n        "fix_plan": fix_plan,\n        "detected_at": datetime.now().isoformat(),\n        "status": "pending"\n    }\n    # 追加到 pending_fixes.md\n    with open(PENDING_FIXES_PATH, "a") as f:\n        f.write(format_fix_entry(entry))\n\ndef check_and_fix(project):\n    """巡检主流程：先检查缓存，再诊断"""\n    # 第一步：检查是否有待修复项\n    pending = load_pending_fixes()\n    project_fixes = [f for f in pending if f["project"] == project]\n    \n    if project_fixes:\n        # 有待修复项，直接执行修复\n        for fix in project_fixes:\n            execute_fix(fix)\n        return\n    \n    # 第二步：没有待修复项，执行诊断\n    issues = diagnose(project)\n    for issue in issues:\n        # 诊断后立即保存到缓存\n        save_pending_fix(\n            project=project,\n            issue_type=issue["type"],\n            root_cause=issue["root_cause"],\n            fix_plan=issue["fix_plan"]\n        )\n```\n\n#### 3. 修复执行与验证\n\n```python\ndef execute_fix(fix):\n    """执行修复并验证"""\n    print(f"执行修复: {fix['project']} - {fix['issue_type']}")\n    \n    # 执行修复计划中的每一步\n    for step in fix["fix_plan"]:\n        result = terminal(step["command"])\n        if result["exit_code"] != 0:\n            print(f"修复步骤失败: {step}")\n            return False\n    \n    # 验证修复结果\n    if verify_fix(fix):\n        # 修复成功，从缓存中移除\n        remove_from_pending(fix)\n        log_evolution(f"修复完成: {fix['issue_type']}")\n        return True\n    else:\n        # 修复失败，更新状态\n        update_pending_status(fix, "failed")\n        return False\n```\n\n### 完整的巡检流程\n\n```\n┌──────────┐\n│ 定时触发  │\n└────┬─────┘\n     ↓\n┌──────────┐     ┌──────────┐\n│ 读取缓存  │────→│ 有待修复? │\n└──────────┘     └────┬─────┘\n                      │\n              ┌───────┴───────┐\n              ↓               ↓\n         有(执行修复)     无(执行诊断)\n              │               │\n              ↓               ↓\n        ┌──────────┐   ┌──────────┐\n        │ 执行修复  │   │ 检测问题  │\n        └────┬─────┘   └────┬─────┘\n             ↓               ↓\n        ┌──────────┐   ┌──────────┐\n        │ 验证结果  │   │ 诊断根因  │\n        └────┬─────┘   └────┬─────┘\n             ↓               ↓\n        ┌──────────┐   ┌──────────┐\n        │ 更新缓存  │   │ 保存缓存  │\n        └──────────┘   └──────────┘\n```\n\n---\n\n## 踩坑记录\n\n### 坑 1: 缓存文件格式选择\n\n**现象**：最初用 JSON 格式存储 pending_fixes，但 AI Agent 在会话中直接编辑 JSON 容易出错（引号转义、逗号遗漏）。\n\n**原因**：AI Agent 更擅长读写 Markdown，JSON 的语法严格性在 Agent 自主编辑时是负担。\n\n**解决**：改用 Markdown 格式，用 `## [项目] 标题` 作为分隔符，YAML frontmatter 存储结构化数据。\n\n### 坑 2: 缓存过期问题\n\n**现象**：有些问题在缓存后被其他途径修复了（比如手动修复），但缓存记录还在，导致下次巡检尝试修复一个已解决的问题。\n\n**原因**：缓存没有过期机制和验证逻辑。\n\n**解决**：在执行修复前先验证问题是否仍然存在：\n\n```python\ndef execute_fix(fix):\n    # 先验证问题是否还在\n    if not verify_issue_exists(fix):\n        remove_from_pending(fix)\n        print(f"问题已自行解决: {fix['issue_type']}")\n        return True\n    # ... 执行修复\n```\n\n### 坑 3: 并发巡检冲突\n\n**现象**：两次巡检间隔太短（比如手动触发 + 定时触发），同时读取缓存，重复执行修复。\n\n**原因**：没有锁机制。\n\n**解决**：使用文件锁：\n\n```python\nimport fcntl\n\ndef acquire_lock():\n    lock_file = open("/tmp/guardian.lock", "w")\n    try:\n        fcntl.flock(lock_file, fcntl.LOCK_EX | fcntl.LOCK_NB)\n        return lock_file\n    except IOError:\n        print("另一个巡检正在进行，跳过")\n        return None\n```\n\n---\n\n## 总结与展望\n\n### 核心收获\n\n1. **监控系统的价值在闭环**：检测和诊断只是手段，修复才是目的。一个不能闭环的监控系统，本质上只是一个昂贵的日志收集器。\n\n2. **AI Agent 需要持久化中间状态**：会话的无状态特性是 Agent 自主工作流的最大障碍。任何需要跨会话传递的信息，都必须显式持久化。\n\n3. **简单机制解决复杂问题**：一个 Markdown 文件就能解决"反复诊断"的问题，不需要复杂的数据库或消息队列。\n\n### 最佳实践\n\n- **诊断即持久化**：诊断完成后立即写入缓存，不要等到会话结束\n- **修复前验证**：执行修复前先确认问题仍然存在\n- **清理已完成项**：修复成功后立即从缓存中移除\n- **记录修复历史**：用 Evolution Log 记录每次修复的完整过程\n\n### 延伸阅读\n\n- 飞轮守护者的完整架构：[飞轮守护者：当 AI Agent 学会自己巡检自己](https://whalemalus.com/article/flywheel-guardian-ai-agent-self-patrol)\n- 基础设施自动化巡检实战：[基础设施自动化巡检实战](https://whalemalus.com/article/infra-auto-patrol)\n- PageWise 项目的飞轮迭代实践：[PageWise v3.0.0 发布](https://whalemalus.com/article/pagewise-v3-release-5857-tests-102-iterations)\n
